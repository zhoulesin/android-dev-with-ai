# 第 11 章 · 持续更新

> **本节目标**：建立 AI Coding 的持续进化体系——从信息获取、Rules 演进、知识库建设到季度回顾，让团队的 AI 开发能力形成正循环。
>
> **前置知识**：已有至少 3 个月的 AI 辅助 Android 开发经验，团队中有成文的 Rules 文档和至少 2 个自定义 Skills。

---

AI 工具和模型的迭代速度远超 Android 开发生态的任何一个环节。前一个月还在用 GPT-4o，下一个月 Claude 4 就碾压了排行榜；上周刚写好的 Rules，这周因为 Compose 升级就过时了。持续更新不是锦上添花——它决定了你的团队在 AI 时代是持续领先，还是迅速落后。

本章不讨论具体的 Prompt 技巧或 Skill 开发方法，而是聚焦一个更根本的问题：**如何让 AI Coding 能力成为一个不断进化的系统**。

---

## 11.1 信息渠道：如何跟踪最新动态

AI Coding 领域的信息熵极高——每周都有新模型、新工具、新 Skills 涌现。如果没有一个高效的信息过滤系统，你很快就会陷入信息过载。

### 11.1.1 信息分层策略

我们建议采用三层过滤机制，确保信息既全面又不冗余：

| 层级 | 频率 | 渠道 | 目的 |
|------|------|------|------|
| **日常层** | 每日 10 分钟 | Twitter/X、Reddit | 捕捉突发新闻、重大更新 |
| **周览层** | 每周 30 分钟 | Android Weekly、Kotlin Weekly、掘金 | 系统性了解行业趋势 |
| **月度层** | 每月 1 小时 | 官方博客、LMSYS 排行榜、团队分享 | 深度评估、决策参考 |

三层机制的核心原则是：**日常层只浏览不深究，周览层做好标记，月度层才下决策**。这样既不会漏掉重要信息，也不会被碎片化内容消耗大量时间。

### 11.1.2 模型动态

AI 模型的更新直接影响你的代码质量和开发效率。一个新模型的发布可能意味着你的某条 Rules 不再必要，或某个 Skill 可以大幅简化。

**必追信息源：**

- **LMSYS Chatbot Arena 排行榜** (`chat.lmsys.org`)：目前业界最权威的模型能力评测。重点关注 Code 分项排名——它比综合排名更能反映模型在实际编程任务中的表现。当新模型在 Code 分项上超越你正在使用的模型超过 30 个 ELO 分时，就值得花时间评估迁移。
- **OpenAI、Anthropic、Google AI 官方博客**：模型更新公告、API 变更、最佳实践指南。建议 RSS 订阅或设置邮件提醒。
- **微信公众号/掘金/知乎**：关注"机器之心""量子位""AI 前线"等技术媒体，以及活跃的 AI 技术号。中文渠道的优势在于会对海外信息做二次消化和解读，节省时间。

> **实战建议**：不必把每个模型更新都当真。历史上"重磅发布"的模型有超过一半在真实编码场景下表现平平。判断标准只有一条：**在自己的代码库上实测**。

### 11.1.3 工具动态

编码工具的迭代甚至比模型更快。Cursor 在 2024 年更新了 40+ 个版本，Windsurf 从诞生到主流仅用了 3 个月。

**重点关注：**

| 工具 | 关注渠道 | 关注重点 |
|------|---------|---------|
| Cursor | GitHub Releases、`@cursor_ai` | Agent 模式能力、Rules 系统变更 |
| GitHub Copilot | Changelog、`@GitHubCopilot` | 模型升级、Workspace 功能 |
| Windsurf | Changelog、Reddit r/cursor | Flow 模式、上下文管理 |
| Claude Code | Anthropic 博客、`@anthropicai` | Agent 能力、Skills 系统 |
| Android Studio | Android Developers Blog | AI 插件集成、Studio Bot 更新 |

**Reddit 的价值**：`r/androiddev`、`r/githubcopilot` 等子论坛是发现坑和真实体验的最佳场所。官方的 Release Notes 只会告诉你加了什么功能，Reddit 会告诉你加了之后哪些功能反而变差了。

### 11.1.4 Skills/MCP 生态

Skills 和 MCP 是 AI Coding 的"应用商店"，其生态增长速度不亚于移动端的 App Store 早期阶段。

**必追资源：**

- **awesome-mcp-servers** (GitHub)：MCP Server 的汇总清单。重点关注数据库操作、文件管理、API 集成类 Server——它们对 Android 开发最实用。
- **awesome-claude-skills** (GitHub)：Skills 汇总清单。定期浏览新发布的 Skills，评估是否可引入团队。
- **Cursor Directory** (`cursor.directory`)：社区共享的 Cursor Rules，可作为编写团队 Rules 的参考范本。

> **实用策略**：不要一次性引入太多 Skills。我们的经验是每月评估 1-2 个新 Skill，试用一周后决定是否保留。当前团队最常用且持续保留的 Skills 通常在 5-8 个。

### 11.1.5 Android 开发 AI 应用

AI 在 Android 端的落地——包括端侧推理、AI API 接入、ML Kit 更新——也是一个不容忽视的方向。

**渠道推荐：**

- **Android Weekly**：几乎所有 Android 开发者都订阅的周刊，AI 相关内容占比已从 2023 年的不到 5% 上升到 2025 年的 25%+。
- **Kotlin Weekly**：专注 Kotlin 生态，经常有 KMP 和 AI 结合的文章。
- **掘金 Android 专题**：国内开发者的实战经验，比翻译文章更具落地参考价值。

## 11.2 Rules 文档的演进策略

Rules 不是写完就完事的静态文档。它是团队 AI Coding 能力的核心资产，必须随项目共同演进。

### 11.2.1 版本化管理的必要性

很多团队犯的第一个错误是把 Rules 放在项目根目录，随项目代码一起演变，却不做版本管理。当项目从 Compose 1.5 升级到 1.7，或从单模块变为多模块时，旧的 Rules 不仅无效，还会产生反向误导。

**推荐使用显式版本目录：**

```
docs/rules/
├── v1.0/    (2024-Q1: Compose + XML 混合)
│   ├── compose-rules.md
│   ├── architecture-rules.md
│   └── gradle-rules.md
├── v2.0/    (2024-Q3: 全面 Compose 迁移)
│   ├── compose-rules.md
│   ├── architecture-rules.md
│   ├── gradle-rules.md
│   └── navigation-rules.md
└── v3.0/    (2025-Q1: KMP 引入)
    ├── compose-rules.md
    ├── kmp-rules.md
    ├── architecture-rules.md
    └── gradle-rules.md
```

版本化管理的核心价值不在于历史追溯，而在于**新成员可以快速理解项目在不同阶段的技术决策及其背后的原因**。同时，当 AI 给出与历史阶段类似的建议时，你可以快速判断："这个建议对应我们 v2.0 阶段的水平，但我们现在是 v3.0，需要更现代的做法。"

### 11.2.2 变更触发条件

并非每次代码改动都需要更新 Rules。以下四类情况是明确的触发条件：

| 触发条件 | 典型场景 | 影响范围 |
|---------|---------|---------|
| **技术栈升级** | Kotlin 2.0、Compose BOM 升级、AGP 大版本 | 需更新所有涉及该技术栈的 Rules |
| **架构调整** | 引入新模块、重构分层、从单 Activity 改为多 Activity | 需更新架构相关 Rules，可能影响 2-3 个文件 |
| **踩坑积累** | 线上 Bug 总结、性能优化经验、代码审查中的共性问题 | 通常只需在现有 Rules 中追加条款 |
| **工具变更** | CI 工具迁移、Lint 规则调整、构建系统变更 | 需更新工具链相关的约束条款 |

**反例警示**：某团队在引入 KMP 后，因为懒得更新 Rules，AI 仍在持续生成纯 Android 平台的代码，导致 3 次提交被 CI 拦截。拖了两周后一次性更新的成本是及时更新的 4 倍。

### 11.2.3 变更流程

Rules 变更不是一个人的事。以下是经过验证的五步流程：

```
1. 发现 Rules 不足
   ↓ 记录具体案例：什么场景、什么问题、期望的 AI 行为是什么
2. 记录具体案例
   ↓ 在团队文档中归档，附带原始代码和 AI 输出对比
3. 更新 Rules
   ↓ 根据案例修改或新增 Rules 条款
4. 团队 Review
   ↓ 至少一人验证：新 Rules 是否解决了案例中的问题
5. 生效
   ↓ 发布通知，确保所有成员更新本地 Rules
```

**关键细节**：第 2 步的"记录具体案例"最容易被打折扣。没有案例支撑的 Rules 修改很容易变成空中楼阁——改了之后才发现新 Rules 在另一种场景下又出了问题。建议每个 Rules 变更都附带一个 `cases/xxx.md` 记录来龙去脉。

### 11.2.4 Rules 健康度检查

每季度至少进行一次 Rules 健康度检查，覆盖以下维度：

- **覆盖率**：当前 Rules 是否能覆盖团队 80% 以上的日常开发场景？
- **准确率**：抽取最近 5 次 AI 生成的代码，检查是否有违反 Rules 的情况？
- **精简度**：是否有冗余或互相矛盾的规则？全局 Rules 的总行数是否超过 150 行？
- **更新频率**：最近一个季度是否有超过 3 条 Rules 超过 2 个月未更新？如果有，可能存在僵尸规则。

## 11.3 团队知识库建设

如果 Rules 是让 AI"做对的事"，那么知识库就是让团队"知道怎么让 AI 做对的事"。前者是约束层，后者是经验层——两者缺一不可。

### 11.3.1 知识库结构

建议使用独立的 Git 仓库（而非散落在项目中的零散文件），结构如下：

```
team-ai-knowledge/
├── README.md            # 仓库说明与使用指南
├── prompts/             # 高频 Prompt 模板库
│   ├── new-feature.md
│   ├── bug-fix.md
│   ├── code-review.md
│   ├── refactoring.md
│   └── compose-ui.md
├── rules/               # 项目 Rules 配置
│   ├── global-rules.md
│   ├── compose-rules.md
│   └── gradle-rules.md
├── skills/              # 自定义 Skills
│   ├── feature-scaffold/
│   └── bug-triage/
├── cases/               # 实战案例
│   ├── compose-migration.md
│   ├── performance-optimization.md
│   └── ksp-migration.md
├── pitfalls/            # 踩坑记录
│   ├── 2025-03-compose-recomposition.md
│   └── 2025-04-hilt-kmp-conflict.md
└── quarterly-review/    # 季度复盘报告
    ├── 2025-Q1-review.md
    └── 2025-Q2-review.md
```

**各目录职责说明：**

| 目录 | 内容定位 | 更新频率 | 受众 |
|------|---------|---------|------|
| `prompts/` | 经过验证、可直接复制使用的 Prompt 模板 | 每次新场景总结后更新 | 全员 |
| `rules/` | 项目级 Rules（与代码仓库中的 Rules 保持同步） | 随项目技术栈演进 | AI 工具 |
| `skills/` | 团队自研 Skills 的源码和文档 | 每次功能更新后 | 高级用户 |
| `cases/` | 完整的 AI 辅助开发案例，含输入输出对比 | 每次典型任务完成后 | 新成员 |
| `pitfalls/` | 具体问题的记录与分析 | 随时更新 | 全员 |
| `quarterly-review/` | 季度复盘报告 | 每季度一次 | 管理者 |

### 11.3.2 Prompt 模板的价值

很多开发者低估了 Prompt 模板的重要性。事实上，一个好的 Prompt 模板不仅节省编写时间，更重要的是**消除了不同开发者之间的 Prompt 质量方差**。

**模板设计原则：**

```markdown
### 新功能开发模板

你是 Android 高级开发工程师，项目技术栈：[自动填入]

## 需求描述
[填入具体需求]

## 开发要求
1. 架构层：严格遵守 [填入项目架构名，如 MVVM + Clean Architecture]
2. UI 层：[Compose View 规范]
3. 数据层：[Repository → DataSource 规范]
4. DI：[Hilt Module 规范]
5. 测试：[测试覆盖率要求]

## 输出格式
- [文件1]：[描述]
- [文件2]：[描述]

## 负面约束
- 不引入新的第三方库，除非明确批准
- 不使用 @OptIn 实验性 API
- 不生成冗长的 KDoc，保留关键点即可
```

模板的价值在于它是一个活的文档——每完成一个新功能，回顾一次 Prompt，把这次多写的有效约束补进去，把没用的约束删掉。

### 11.3.3 知识库的运作机制

光有仓库不行，得有机制保障它被真正使用和更新：

1. **新功能开发前**：从 `prompts/` 选择最匹配的模板，基于它调整 Prompt。
2. **功能开发后**：如果 Prompt 有改进，提交 PR 更新模板。
3. **遇到新坑时**：在 `pitfalls/` 新建文件，记录现象、根因、解决方案。
4. **季度复盘时**：回顾 `cases/` 和 `pitfalls/` 的积累，形成下一阶段的改进计划。
5. **新人入职时**：前三天通读 `cases/` 目录，了解团队如何使用 AI。

> **工具推荐**：使用 Git 仓库管理整个知识库，搭配 GitHub Actions 做自动格式化检查。也可以使用 Notion、飞书文档等托管工具做补充，方便富文本编辑和搜索。但核心资产强烈建议用 Git 管理——版本可控、可 Review、可追溯。

## 11.4 季度回顾机制

如果说日常反馈是微调，季度回顾就是校准。没有定期回顾，团队很容易在错误的方向上越走越远。

### 11.4.1 为什么需要季度回顾

AI Coding 领域变化太快——每个季度都可能出现颠覆性的新工具或新范式。月度回顾太频繁，数据积累不足，结论不可靠；年度回顾太滞后，发现问题时已经错过了最佳调整时机。季度是一个合理的平衡点。

季度回顾的核心问题是：**过去三个月，团队的 AI Coding 实践到底带来了多少实际收益？下个季度应该怎么调整？**

### 11.4.2 回顾模板

以下模板已在多个团队验证，可直接使用：

```markdown
# 20XX QX AI Coding 回顾

## 一、使用数据

| 指标 | 本季度 | 上季度 | 环比变化 |
|------|--------|--------|---------|
| AI 代码采纳率 | __% | __% | ±__% |
| 月均 Token 消耗 | __ | __ | ±__% |
| 月均 AI 交互次数 | __ | __ | ±__% |
| 主要使用工具 | __ | __ | — |
| 活跃 Skills 数量 | __ | __ | — |
| 活跃 Rules 文件数 | __ | __ | — |

> **采纳率计算**：AI 生成后被保留到最终提交的代码行数 ÷ AI 生成的总代码行数。可以在 CI 中通过 git diff 自动化统计。

## 二、效率变化

| 指标 | 本季度 | 上季度 | 环比变化 |
|------|--------|--------|---------|
| 需求吞吐量（人均 Story Point/月） | __ | __ | ±__% |
| 生产 Bug 率（每千行代码） | __ | __ | ±__% |
| 新人上手时间（达到独立开发的天数） | __ | __ | ±__% |
| 代码审查平均耗时（小时/PR） | __ | __ | ±__% |

## 三、Skills 评估

| 状态 | Skill 名称 | 评估理由 |
|------|-----------|---------|
| ✅ 保留 | __ | __ |
| ✅ 保留 | __ | __ |
| 🆕 新增 | __ | __ |
| ❌ 移除 | __ | __ |

> Skills 评估标准：使用频率 > 每周 2 次、输出的结果 > 80% 无需返工、团队成员评价 > 3/5 分。

## 四、AI 质量抽查

抽取本季度 10 个 AI 辅助完成的 PR，评估以下维度：

| 维度 | 评分(1-5) | 典型问题 |
|------|----------|---------|
| 代码风格一致性 | __ | __ |
| 架构合理性 | __ | __ |
| 边界条件覆盖 | __ | __ |
| 安全性 | __ | __ |
| 性能 | __ | __ |

## 五、下季度计划

| 计划项 | 具体内容 | 预估影响 | 负责人 |
|--------|---------|---------|--------|
| 待尝试的工具/模型 | __ | __ | __ |
| 待优化的流程 | __ | __ | __ |
| Rules 调整计划 | __ | __ | __ |
| Skills 调整计划 | __ | __ | __ |
| 知识库更新计划 | __ | __ | __ |
```

### 11.4.3 回顾的执行要点

- **数据先行**：没有数据的回顾等于拍脑袋。至少提前一周收集 Token 消耗、采纳率、Bug 率等基础数据。
- **匿名反馈**：在季度回顾前，通过匿名问卷收集团队成员对 AI Coding 的真实感受。很多人在公开场合不会说出"我觉得用 AI 反而更慢了"这种话。
- **聚焦 2-3 个改进点**：回顾不是为了找出所有问题，而是找出最重要的 2-3 个并在下季度重点解决。贪多嚼不烂。
- **留存记录**：每次回顾的文档都放入 `quarterly-review/` 目录。一年后回看，你会惊讶于团队的进化速度。

## 11.5 保持学习的心态

最后一节不讨论工具或流程，而是讨论心态——因为再好的体系，如果心态歪了，也走不远。

### 11.5.1 AI 是工具不是替代品

这是最重要的一条，也是最容易被忽略的一条。当 AI 能够在 10 秒内生成你手写需要 2 小时的代码时，人性本能是"那就让它写呗"。

但这种依赖是有代价的：

- **代码理解盲区**：三个月后，你无法向别人解释你为什么选择了某个设计方案，因为"是 AI 选的"。
- **调试能力退化**：当 AI 生成的代码出 Bug 时，你对代码的理解不够深，定位问题的时间比手写代码还长。
- **技术判断力丧失**：你不会再思考"这个场景下 Coroutine 和 Flow 哪个更合适"，因为 AI 替你做了选择。

**核心原则：永远保持对代码的理解和控制。AI 生成的每一行代码，你都应该能在 Code Review 时说出它为什么在这里、是否有更好的写法。**

### 11.5.2 定期脱离 AI 编码

我们建议：**每月至少有一天纯手写代码**——关掉 Copilot、不打开 Cursor 的 Inline Completion、不向 AI 提任何问题。

这不是形式主义。一个月手写一天，你会发现：

- 某些你已经"熟练"的 API 其实记不住完整签名
- 某些你依赖 AI 解决的架构问题，你自己也能设计出来
- 某些你"知道"的最佳实践，其实只是 AI 给你的答案，你并不理解为什么它是最佳的

**手写日也是发现 Rules 问题的机会**。当你不依赖 AI 时，才能客观评估当前的 Rules 是否真正表达了团队的技术共识。

### 11.5.3 关注底层原理

AI 擅长生成应用层代码——写 Compose UI、调用 Retrofit API、操作 Room 数据库。但底层原理理解——Compose 的 Recomposer 机制、Gradle 的 Task Graph 执行顺序、Kotlin Coroutine 的调度器原理——是 AI 无法替代的。

针对 Android 开发者，以下底层知识值得持续深入：

| 领域 | 核心知识点 | 为什么 AI 无法替代 |
|------|----------|-------------------|
| Compose | Recomposer、Snapshot System、Modifier 链 | AI 能避开重组问题，但一旦遇到复杂重组 Bug，仍然需要人来分析 |
| Gradle | Task Graph、Configuration Cache、Build Cache | AI 很难理解你的项目具体构建瓶颈在哪 |
| Kotlin | Coroutine 调度器、Flow 的背压策略、inline/reified | 这些是决策型知识，AI 只能给建议，最终决策需要你的判断 |
| Android 系统 | WindowManager、SurfaceFlinger、Binder 通信 | 这些涉及系统底层，AI 的知识往往停留在 API 层面 |

### 11.5.4 向 AI 学习但不盲从

AI 是一个强大的学习工具——你可以问它"这段 Compose 代码为什么会导致重组"，或者"解释一下 Flow 的 buffer() 和 conflate() 的区别"。它在解释概念方面非常出色。

但 AI 的建议不是绝对的：

- **AI 可能给出过时的 API**：如果你的项目使用 Compose 1.7，而 AI 的训练数据停留在 1.5，它可能推荐已废弃的 API。
- **AI 可能遗漏性能陷阱**：比如 Compose 中的 `derivedStateOf` 和 `remember` 的使用边界，AI 可能说得不够精确。
- **AI 不理解你的业务上下文**：它无法判断某个按钮是否应该放在顶部还是底部，这需要你来决策。

**好习惯：当 AI 给你一个你不确定的建议时，先去官方文档验证，再决定是否采纳。** AI 是加速器，方向盘必须在自己手里。

## 踩坑记录

### 坑 1：信息渠道太多导致的"FOMO 焦虑"

**现象**：为了不错过任何更新，订阅了 15 个信息源、8 个 Newsletter、5 个 Telegram 频道。结果每天花 1.5 小时浏览信息，真正吸收的不到 20%，反而产生了严重的 FOMO 焦虑（Fear Of Missing Out）。

**原因**：把"收集信息"当成了"学习"。信息渠道是手段而非目的——订阅再多，不实践也不会带来任何提升。

**解决**：严格执行三层过滤机制。日常层严格控制在 10 分钟内，超过时间直接关闭所有信息窗口。周览层只做标记，不深入阅读。月度层才深度研究 1-2 个主题。关键是：**信息不是越多越好，而是被实践验证过的才是有用的。**

### 坑 2：盲目追新模型导致团队混乱

**现象**：每次有新模型发布，团队中就有成员尝试切换。结果一个月内团队同时在用 3 种不同的模型和 2 种不同的工具，Rules 在不同模型下的表现不一致，代码风格出现明显差异。

**原因**：缺乏统一的工具决策机制。个人层面的尝鲜行为影响到了团队协作的一致性。

**解决**：建立明确的工具选型决策流程：

1. 新模型/工具出现 → 指定一人负责评估（1-2 天）
2. 在团队的标准测试集上运行（包含 Compose UI、网络请求、数据库操作、单元测试四类场景）
3. 评估结果在团队会议上讨论，**团队统一决策**是否切换
4. 如果切换，同步更新所有 Rules 和 Prompt 模板
5. 个人不得在团队项目中未经讨论使用非标准工具

### 坑 3：Rules 更新后忘记同步导致 AI 产生前后矛盾的代码

**现象**：A 同学更新了 `compose-rules.md`，但 B 同学和 C 同学本地还是旧版本。结果三人用 AI 生成了遵循不同规范的代码，在 Code Review 阶段才发现明显冲突。

**原因**：Rules 文件没有纳入版本控制与同步机制，依赖口头通知或聊天群"吆喝"。

**解决**：

- Rules 文件必须跟随项目代码仓库（不是知识库仓库，是代码仓库），通过 Git 统一管理
- Rules 更新后，在 `CHANGELOG.md` 或 Rules 文件头部标注更新日期和变更摘要
- CI 中加入 Rules 版本检查——在 AI 提交的 PR 中加入一个标签，标明该 PR 使用的是哪个版本的 Rules。如果版本落后于最新，CI 给出 Warning
- 定期（如每周一）要求团队成员 `git pull` 并确认 Rules 文件已是最新

### 坑 4：季度回顾沦为形式主义

**现象**：季度回顾写了三次之后，数据还是"待统计"、改进计划还是"继续优化"。回顾文档逐渐变成了上一次的复制粘贴。

**原因**：季度回顾缺乏可量化的指标和具体的执行人，"死线"变得模糊。

**解决**：

- 季度回顾必须指定一个负责人（轮换制），负责人需要在季度最后一周完成回顾草稿
- 数据项至少要有 Token 消耗和采纳率两个硬指标（从工具平台拉取，不允许"待统计"）
- 改进计划中的每一项都必须有负责人和截止时间。没有负责人的计划等于没有计划
- 回顾文档在团队会议上过一遍（20-30 分钟），而不是发到群里让大家"有空看看"
