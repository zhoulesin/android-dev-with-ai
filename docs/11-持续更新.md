# AI 工具月月在变，如何不被 FOMO 拖死还能持续进步

上个月 Claude 出了新模型，这个月 Cursor 更新了 Agent 模式，下周 Trae 又发新版。打开 Twitter，满屏都是"我换了 XXX 之后效率翻倍""你还用 YYY？大家都转 ZZZ 了"。工具焦虑症，你有吗？

我有。而且我为此付过真金白银的代价。

2024 年上半年，我基本处于"月抛型工具人"状态：1 月用 Copilot，2 月切 Cursor，3 月试 Windsurf，4 月换回 Cursor，5 月又跑去玩 Claude Code。每次切换都要重新适应快捷键、重新配置 Rules、重新调整心智模型。三个月下来一算——工具切换的时间加起来超过 40 小时，而代码产出跟之前用单一工具时基本持平，甚至因为频繁切换导致 Rules 不一致，Code Review 阶段的返工率反而上升了。

这就是 FOMO（Fear Of Missing Out）的真正代价：你以为自己在"跟上时代"，实际上你在用战术上的勤奋掩盖战略上的懒惰。

---

## FOMO 陷阱：你追的不是工具，是焦虑

AI 编码领域的更新频率已经失控了。2025 年前 5 个月，主流 AI 编码工具累计发布了超过 60 个大版本，平均每 2.5 天就有一个值得关注的更新。如果你把每个更新都认真研究一遍，你就不用写代码了——你就成了全职的"工具体验师"。

更隐蔽的问题是：**大部分更新对你根本没有影响。** 根据我们团队的跟踪统计，过去 12 个月的 AI 工具更新中：

- ~20% 的更新确实带来了可感知的效率提升（比如 Cursor 的 Agent 模式、Claude 的上下文窗口扩大）
- ~50% 的更新是"有也不错，但没有也不影响"的边际改良
- ~30% 的更新甚至引入了新 Bug 或破坏了原有工作流

这就是 80/20 法则在工具追踪上的映射：**只有 20% 的更新值得你关注，剩下 80% 是噪音。** 你的任务不是追逐每一个更新，而是建立一套系统，让那 20% 自动浮现出来。

## 我的信息食谱：少即是多

很多"AI 学习路线图"会给你列 20 个信息源，让你每天刷。我不会这么干。以下是我实际在用的信息渠道——不是推荐给你照搬，而是让你看到一个真实的最小可行方案：

**我每天看的东西：**

两个 Twitter/X 账号。就两个。我选账号的标准不是"粉丝多"或"更新勤"，而是"信号密度"——每 10 条推特里至少有 3 条让我有"这个信息有用"的感觉。具体是谁不重要，因为半年后可能就换了。重要的是你亲自验证过这个人的信息质量。

**我每周看的东西：**

一份 Newsletter。我订阅过 6 份 AI 相关的周刊，最终只保留了一份。筛选标准很简单：连续四周，每周看完后回想——这周的内容里，有没有任何一条改变了我的实际行为？如果没有，退订。残酷但有效。

**我每月看的东西：**

GitHub Trending，但加了过滤条件：只看 `ai` `dev-tools` `android` 三个标签下的仓库。30 分钟扫一遍，标记 2-3 个感兴趣的，下个月试一个。仅此而已。

**我刻意不看的东西：**

- AI 工具对比评测视频（大多是体验 2 小时就出片，缺乏深度）
- 朋友圈/微信群里的"XX 又出新功能了"消息（纯噪音，99% 跟你无关）
- 任何标题带"震惊""颠覆""革命"字样的内容

关键是：**月度的 30 分钟深度扫描，远比每天的碎片化刷屏有价值。** 你不需要知道昨天发生了什么，你需要知道过去一个月的趋势是什么。

## Rules 文档的演进：不要每次都重写

CLAUDE.md（或 Cursor Rules，或 Copilot Instructions）是你跟 AI 之间的"接口协议"。协议不应该天天改——稳定的接口才是好的接口。

但很多人的做法是：每次新工具火了，就把 Rules 推倒重写一遍，试图适配新工具的"最佳实践"。这是最蠢的做法。你改的不是工具配置，你改的是你和你团队过去几个月积累下来的技术决策——那些是用 Bug 和返工换来的经验。

**我们所做的版本化：**

```
docs/rules/
├── v1.0/    (2024-Q1: Compose + XML 混合)
├── v2.0/    (2024-Q3: 全面 Compose 迁移)
├── v3.0/    (2025-Q1: KMP 引入)
└── v4.0/    (2025-Q2: 当前版本)
```

保持历史版本不是因为怀旧，而是因为**回溯能力**。当你看到 AI 生成了一段奇怪的代码，查一下 Rules 版本历史，你就能理解：哦，这个是 v2.0 时代的惯性建议，现在不适用了。没有版本历史，你连 AI 为什么"犯这个错"都不知道。

**什么时候该更新 Rules？四种情况：**

1. **技术栈升级**：Kotlin 2.0、Compose BOM 大版本、AGP 升级。这类变更必须更新，否则 AI 会持续生成过时的 API 调用。
2. **架构调整**：从单模块拆多模块、引入新的 DI 框架、重构分层。不更新的话，AI 会按旧架构生成代码，每次都要手动修正。
3. **踩坑积累**：线上 Bug 的根因分析、Code Review 中反复出现的共性问题。这类更新通常最简单——在现有 Rules 中加一条约束即可。
4. **工具链变更**：从 Jenkins 切 GitHub Actions、从 kapt 切 KSP。影响范围明确，更新成本低。

**反模式警告：** 千万别因为"Cursor 出了新功能"就把 Rules 全改一遍。Rules 描述的是**你的项目**应该怎么写代码，不是**Cursor**希望你怎么写代码。工具是服务的，不是被服务的。

## 团队知识库：让经验复利

我们团队的 AI 开发知识库是一个独立的 Git 仓库，结构很简单：

```
team-ai-knowledge/
├── prompts/             # 高频 Prompt 模板
├── pitfalls/            # 踩坑日志
├── skills/              # 自定义 Skills（带版本号）
├── cases/               # 完整案例
├── rules/               # Rules 的副本（与项目同步）
└── quarterly-review/    # 季度回顾
```

这里面最被低估的是**踩坑日志**（`pitfalls/`）。格式极其简单：一个 Markdown 文件，标题是日期+问题简述，内容三部分——现象、根因、解决方案。每篇 200 字以内，5 分钟写完。但它的价值是复利式的：第一个月可能只有 3 篇，第六个月有 20 篇，一年后超过 50 篇。当一个新人遇到"Compose 重组死循环"时，不用花 2 小时调试，搜一下日志就能命中之前同事记录过的同类问题——**团队知识不会因为人员流动而流失。**

**Prompt 模板库**是另一个被严重低估的资产。很多开发者觉得"Prompt 这种东西有什么好模板的，随便写写就行"。但团队中必然存在 Prompt 能力方差：有人一次就能让 AI 产出高质量代码，有人要来回修正 5-6 轮。模板消除的就是这个方差——把最优实践固化为可复用的起点。

模板设计的一个关键原则：**模板不是写死的，是活的。** 每完成一个功能，花 2 分钟回顾——这次 Prompt 里哪句话是多余的，哪句话是神来之笔。好的留下，差的删掉。一个月后，你的模板就是你们团队在 AI 协作这件事上的最高水平。

**Skills 版本化**同样重要。把 Skill 当成代码依赖来管理：有 changelog，有版本号，有测试用例。当底层工具升级时，你能精确知道哪些 Skill 需要适配、哪些可以平滑过渡。我们甚至给 Skills 仓库配了 GitHub Actions 做自动格式化检查——不是闲得慌，而是因为 Skill 文件中的缩进错误可能导致 AI 行为异常，这种 Bug 极难排查。

**新人入职文档**是知识库的"快充接口"。新人入职第一周，不是让他们看官方文档或 Twitter，而是：
- 第一天：通读 `cases/` 目录，了解团队用 AI 的真实方式
- 第二天：试用 `prompts/` 中的 3-5 个核心模板，感受"团队标准写法"
- 第三天：读 `pitfalls/` 中最近 10 篇，知道哪些坑已经有人踩过了
- 第四天起：开始实际开发，遇到任何问题先在知识库里搜

这样安排后，新人从"能写代码"到"能用 AI 高效写代码"的周期从 3 周缩短到了 1 周。

## 季度重置：别让系统腐烂

任何一个系统，运行超过 3 个月不检修，就会开始腐烂。Rules 里会积累僵尸规则（针对已废弃的 API 的约束），Skills 里会有不再使用的功能模块，Prompt 模板里会残留针对旧版工具的描述。

所以我们每季度做一次 2 小时的"重置会议"。不是那种大而全的战略复盘，而是四个具体动作：

**1. 审计 Skills（30 分钟）**

每个 Skill 过三个问题：
- 过去 3 个月使用频率如何？（低于每周 1 次 → 考虑移除）
- 输出质量怎样？（需要返工的比例超过 30% → 需要重写）
- 有没有新的需求场景需要新增 Skill？

我们团队的 Skill 数量一直维持在 5-8 个。不是不能更多，而是人的大脑只能熟练使用这么多。超过 8 个的 Skill 基本都会变成"僵尸 Skill"——存在但没人用。

**2. 更新 Rules（30 分钟）**

检查最近 3 个月的代码变更：
- 有没有引入新的技术栈或架构模式？
- 过去 3 个月 Code Review 中最常见的 AI 生成问题是什么？
- 有没有 Rules 之间互相矛盾？

一个血泪教训：有次我们引入了 KMP 但忘了更新 Rules，AI 持续生成纯 Android 代码长达 3 周。3 次 PR 被 CI 拦截后才意识到问题。及时更新的成本是 30 分钟，拖了 3 周后追溯修正的成本超过 4 小时。

**3. 评估新工具（30 分钟）**

过去 3 个月有没有值得关注的新工具？不要拍脑袋决定。我们有标准评估流程：
- 指定一人花 2 天试用
- 在标准测试集上跑（包含 Compose UI、网络请求、数据库操作、单元测试四类场景）
- 团队统一决策是否切换

**关键规则：个人不得在团队项目中未经讨论使用非标准工具。** 这不是控制狂，而是防止"工具碎片化"——一个月内团队同时在用 3 种不同模型和 2 种不同工具，代码风格出现明显差异，最终 Code Review 变成灾难。

**4. 检查指标（30 分钟）**

至少盯三个硬指标：
- **AI 代码采纳率**：AI 生成后被保留到最终提交的比例。低于 40% 说明 AI 产出的东西不行（或 Rules 有问题）。
- **生产 Bug 率**：如果 AI 辅助开发的代码 Bug 率显著高于手写代码，需要排查是哪个环节出了问题。
- **Token 消耗趋势**：不是追求省钱，而是通过消耗变化反推 AI 使用模式的变化。

## 一些反直觉的建议

写了这么多关于"如何追赶变化"的内容，我想以几条反直觉的忠告收尾：

**有时候最好的升级就是不升级。** 如果你的系统跑得稳、团队用得顺、产出质量高，那就别动。不要因为 Twitter 上说"XXX 工具又发新版了"就心痒。工具是为产出服务的，不是为尝鲜服务的。

**稳定性 > 新鲜度。** 团队效率的最大杀手不是工具不够新，而是工具不够稳。频繁切换工具带来的上下文切换成本，往往远超新工具带来的效率提升。用数学来说：一个提升 10% 效率的新工具，需要你至少使用它 10 天才能抵消切换它花费的 1 天。而大多数工具切换的成本远不止 1 天。

**等两周再上车。** 几乎所有 AI 工具的新版本发布后的前两周，都会暴露出一堆 Bug。让早期采用者去踩坑，两周后看 Reddit、GitHub Issues 的反馈，再决定要不要切。这两周的时间差不会让你落后，但能让你避开 80% 的坑。

**工具没有流程重要。** Cursor 也好，Copilot 也好，Claude Code 也好——它们本质上是同一个东西的不同界面。真正决定你团队效率的，是背后的 Rules 质量、知识库厚度、Review 流程和反馈机制。一个团队用 Copilot + 完善的 Rules 体系，一定比另一个用 Cursor + 零散口口相传规则的团队效率高。工具是放大器，但不会凭空创造价值。

---

**最终目标不是用遍每一个新工具。最终目标是更快更好地做出 Android 应用。**

工具服务于这个目标，不是反过来。当你盯着 GitHub Trending 焦虑"我是不是又落后了"的时候，问问自己：我上一次交付一个让用户满意的功能，是什么时候？

那个答案，才是你应该追逐的东西。
