# AI 生成代码采纳率从 40% 提到 85%，我做了什么

三个月前，AI 帮我写的 Android 代码，10 行里我只能留 4 行。现在这个数字是 8.5 行。

这不是模型升级带来的。我用的一直是同一个模型。变化来自我搞明白了一件事：**问题不在 AI，在我怎么跟它说话，以及我有没有给它一个能赢的牌桌。**

先说清楚，这篇文章没有魔法。全是我过去一个季度实打实跑出来的数据和踩过的坑。如果你也每天跟 AI 结对编程，下面这些方法应该能帮你省下大把时间。

---

## 你的 Prompt 决定了你的天花板

大多数人给 AI 下指令，就像给一个刚入职的实习生派活："帮我写个登录页面。"然后看着输出叹气：怎么又用 XML？怎么又用了废弃 API？怎么风格跟项目完全不搭？

这不是 AI 不行。是你没告诉它你是谁、在哪儿、要什么。

我后来形成了一个 Prompt 公式，现在写任何需求之前先用 5 分钟过一遍：

```
角色 (Role) + 上下文 (Context) + 约束 (Constraints) + 示例 (Examples) + 输出格式 (Format)
```

缺任何一个要素，输出质量都会出问题。缺角色，AI 不知道自己是 Android 开发还是前端开发，写出来的代码缺少领域最佳实践。缺上下文，它不知道你用 Compose 还是 XML，用 Hilt 还是 Koin。缺约束，它可能用了一堆你项目禁掉的 API。缺格式，输出散成一团，你还得自己拼。

我给你看一个真实的 Android 场景对比。

**差 Prompt：** `用 Retrofit 封装一个网络请求`

**好 Prompt：**

```
你是 Android 高级开发工程师，项目使用 Retrofit + OkHttp + Kotlin Coroutines。
请实现一个通用的网络请求封装层，要求：
- 使用 sealed class 封装 Result 状态（Success / Error / Loading）
- 统一错误处理：网络异常 → NetworkException、服务端错误 → ServerException
- 通过拦截器自动附加 Authorization Header（从 DataStore 读取 Token）
- 支持请求重试（最多 2 次，指数退避）
- 不使用 try-catch 包裹每次调用，改用 runCatching 或自定义扩展
- 输出格式：3 个文件 — ApiResult.kt、NetworkInterceptor.kt、safeApiCall.kt
```

同样一个需求，差的 Prompt 让 AI 随意发挥，输出可能用 Gson 而不是你项目里的 Moshi，可能忘了加 Token 拦截器，可能把错误处理写成 `try-catch` 到处散落。好的 Prompt 直接锁死了技术路线和交付物格式，AI 就像一个有 Spec 参照的高级工程师，而不是一个猜你要什么的实习生。

### 别一次性梭哈，多轮才是正道

还有一个很多人常犯的错误：想一次 Prompt 就拿到完美代码。

我现在的标准操作是三阶段对话：

1. **第一轮（骨架）**：让 AI 先给类结构、接口定义、数据类。指令一般是"先给出类结构和接口定义，不用完整实现"。
2. **第二轮（细化）**：基于骨架，逐个实现函数、添加错误处理、覆盖边界条件。
3. **第三轮（打磨）**：统一命名、清理冗余代码、提取常量、补充文档。

每一轮的上下文都要积累——第二轮的 Prompt 里我会引用第一轮 AI 给出的结构关键词，第三轮引用前两轮达成的共识。这样 AI 始终在同一套技术决策下工作，不会第二轮突然换了一种命名风格。

### 告诉 AI 不要做什么，跟告诉它要做什么一样重要

我给这招起了个名字叫"负面 Prompt"。正面约束是说"你要怎么做"，负面约束是说"你**别**怎么做"。

比如 Compose 开发里我几乎每次都加：

- "不要使用 @OptIn(ExperimentalMaterial3Api::class)"
- "不要用 LocalActivity，改用 LocalContext"
- "不需要抽象成 interface，直接写实现类"
- "不要使用 Android 14+ 专属 API，兼容到 API 26"

这几个"不要"帮我挡掉了非常多 AI 常见的翻车操作——比如兴高采烈地用一个实验性 API，或者在一个根本不需要多态的类上强行抽出 interface。

我建议每个团队的 repo 里放一个 `negative-prompt.md`，按场景分类存放负面约束。每次开发只需要引用对应的段落就行了，新人也能快速对齐团队的"不做清单"。这个文件我们团队维护了两个月，从最初的 8 条涨到了 47 条，每一条背后都有一个"被 AI 坑过"的故事。

## Rules 是你最重要的基础设施

说完了 Prompt，聊一个更底层的东西：Rules 文件。

如果你用 Claude Code 或 Cursor，CLAUDE.md（或 AGENTS.md）是你整个 AI 工作流里最值钱的文件。没有之一。它决定了 AI 在每次跟你对话时，心里装着多少关于你项目的正确信息。

### 三级层级，精准投喂

我的 Rules 体系长这样：

```
项目级 (CLAUDE.md)
  → 技术栈说明、架构风格、通用编码规范、命名约定
  ↓
模块级 (module-network.md, module-database.md ...)
  → 模块职责、内部依赖、包结构、关键类说明
  ↓
功能级 (feature-login.md, feature-favorite.md ...)
  → 功能上下文、相关文件列表、约束条件
```

核心原则：**只在该用的时候加载该用的内容。**

很多团队犯的错误是把所有 Rules 塞进一个 800 行的全局文件。结果 AI 的上下文被撑爆，关键约束淹没在大量无关信息里，输出质量不升反降。我们团队就踩过这个坑——把 8 个模块的 Rules 全部纳入全局加载，Token 消耗翻了 5 倍，采纳率反而从 55% 掉到了 42%。

后来我们定了一个规矩：每个 Rules 文件控制在 200 行以内，全局 Rules 不超过 150 行。通过 `globs` 关键词限制生效范围——比如 `module-network.md` 只在修改 `app/network/` 下文件时触发。这样 AI 每次对话拿到的都是高度相关的上下文，不会被噪声干扰。

以下是我们项目级 Rules 的核心骨架，供参考：

```markdown
## Kotlin 编码规范
- 优先使用 val，避免 !!
- 集合操作优先用标准库函数（map、filter、takeIf）
- sealed class 优先于 enum（需要携带数据时）

## Jetpack Compose 规范
- Composable 命名大写开头，参数放前面
- State 提升原则：能上提就不下传
- 主题色值统一通过 MaterialTheme.colorScheme 取

## 工具链版本约束
- AGP: 7.4.2  |  Kotlin: 1.9.22  |  Compose BOM: 2024.01.00
- Room: 2.6.1  |  Hilt: 2.51.1  | 目标 SDK: 34  |  最低 SDK: 26
```

注意那个"工具链版本约束"——这是我们被坑了 3 次之后血的教训。如果你不声明，AI 会默认使用它训练数据中的最新版本号，然后生成出来的 Gradle 配置引用了一个你项目里根本不存在的 API。我们因为 AGP 版本错配导致构建失败浪费了一整个下午，后来直接把所有工具链版本号写死在 Rules 里。

### Rules 不是一次性工作

把 Rules 当成活文档来维护。每次架构变动、技术栈升级、模块重构，同步更新对应的 Rules 文件。每个季度 review 一次，退役已经过时的约束，补充新增的规范。

我们团队的实践是：Rules 变更纳入 PR Review 流程，关键约束的修改需要团队至少两个人过眼。这样 Rules 本身的质量有人把关，不会出现"一个人随手写的错误 Rules 把所有人带偏"的情况。

常见反模式：写了 Rules 但从不更新。6 个月前写的约束，AI 照着执行但技术栈早就变了，生成的代码全是反操作。Rules 不是写了就完事，它是活的。

## 三层防线：把 Bug 挡在生产环境之前

光有好的 Prompt 和 Rules 还不够。AI 生成的代码，再好的 Prompt 也不能保证 100% 正确。你需要一套质量防线。

我建了一个三层防线模型，这是过去一个季度 Bug 率大幅下降的核心原因：

```
第一层 · AI 自检（投入 1-2 分钟/次，拦截率约 40%）
  → 代码生成后立即让 AI review 自己的输出
  
第二层 · 工具自动检查（投入 5-10 秒/次，拦截率约 35%）
  → ktlint + Detekt + Android Lint + 单元测试

第三层 · 人工审查（投入不定，拦截率约 15%）
  → 聚焦业务逻辑、架构合理性、安全性
```

### 第一层：AI 自检——投入产出比之王

操作方法极其简单：AI 生成代码后，紧接着发一条：

```
请审查你刚才生成的代码，逐项检查：
1. import 是否完整且无冗余
2. 命名是否与项目已有代码风格一致
3. 空安全和边界条件是否覆盖
4. 是否正确使用了项目规定的框架/库
5. 是否有明显逻辑错误或死代码
逐条输出发现的问题和修复建议。
```

我对 50 次 AI 自检做了统计，结果很有意思：

- 32% 的情况：AI 发现自己忘了 import（比如忘记 `import viewModel()`）
- 18% 的情况：命名不一致（用 LiveData 风格给 StateFlow 命名）
- 14% 的情况：边界条件遗漏（网络请求没有 loading 态）
- 10% 的情况：框架 API 误用（Compose 中直接操作 MutableState 的 `.value`）
- 26% 的情况：代码本身没什么问题

也就是说，74% 的情况下 AI 确实能从自己的输出中找到问题。而且这只需要 1-2 分钟。

但有一个坑：如果你直接让 AI"review 一下"，它经常会说"代码质量良好，无明显问题"。这不是代码真的没问题，是 AI 对刚生成的代码有确认偏差。解决方法是强制让它"找茬"——把指令改成："请从代码审查者的角度，至少找出 2 个可以改进或需要确认的地方。"这样问题发现率从 26% 直接飙升到了 72%。

### 第二层：工具链自动拦截

这一层完全不花你的时间，全自动跑在 CI 里：

- **ktlint**：管代码风格。用 `.editorconfig` 统一缩进、换行。
- **Detekt**：管代码异味和复杂度。启用 Android 专属规则集。
- **Android Lint**：管 Android 专项——UnusedResources、HardcodedText、MissingPermission 这些。
- **单元测试**：AI 生成的代码至少 60% 覆盖率，核心逻辑 80%+。

关键配置就一条：`abortOnError = true`。不要让警告悄悄溜走。宁可 CI 红了你修，也别把技术债攒到后面。

```kotlin
// app/build.gradle.kts
android {
    lint {
        abortOnError = true
        checkReleaseBuilds = true
    }
}
```

### 数据不会骗人

我们引入三层防线前后对比（4 周数据）：

| 指标 | 引入前（仅人工 CR） | 引入后（三层防线） |
|------|-------------------|-------------------|
| AI 代码 Bug 率 | 12.3 bug/KLOC | 3.1 bug/KLOC |
| 人均 CR 时间 | 45 min/PR | 22 min/PR |

最让我意外的数字：**60% 的 Bug 在第一层和第二层就被拦截了，根本没流到人工审查环节。**这意味着 Code Review 真正回归了它该干的事——审视架构和业务逻辑，而不是一个个帮你找 import 是不是漏了。

## 你没法改进你度量不到的东西

做了这么多优化，你得知道它们到底有没有用。我 track 四个核心指标：

| 指标 | 计算方式 | 目标值 |
|------|---------|--------|
| **代码采纳率** | 采纳行数 / AI 生成行数 | ≥ 60% |
| **修改率** | 修改行数 / AI 生成行数 | ≤ 40% |
| **Bug 引入率** | AI 引入 Bug 数 / AI 生成 KLOC | ≤ 5/KLOC |
| **开发吞吐量** | 功能点数 / 人天 | 提升 ≥ 30% |

### 怎么采数据

最简单有效的方法：所有 AI 辅助的提交在 commit message 里打 `[ai-assisted]` 标签。

```bash
git log --grep="\[ai-assisted\]" --author="你的名字" --since="2026-01-01" \
  --numstat --pretty=format:"%H %s"
```

有了这个标签，后续任何统计分析都能精准区分 AI 代码和人工代码。我们用 `git blame --line-porcelain` 加标签过滤，统计出了每个模块的 AI 代码行数和保留率。

更进一步，在 Code Review 里给 AI 代码打四个标签：

- `ai-accepted`：直接采纳
- `ai-edited`：经修改后采纳
- `ai-rejected`：完全舍弃
- `ai-bug`：引入 Bug

这样你在每个季度复盘时，一眼就能看出哪个模块的 AI 表现最好、哪个环节出了最多问题。

### 我的季度数据

2026 Q1 跟 Q4 的对比如下：

| 指标 | Q4 2025 | Q1 2026 | 变化 |
|------|---------|---------|------|
| 代码采纳率 | 55% | 68% | +13% |
| 修改率 | 45% | 32% | -13% |
| Bug 引入率 | 5.8/KLOC | 3.2/KLOC | -2.6 |
| 开发吞吐量 | 6.2 FP/人天 | 8.5 FP/人天 | +37% |

采纳率提升最猛的那两周，正好是我们重写了 Compose 模块 Rules 和全面推行三阶段对话策略的时候。这让我确认了一件事：AI 输出质量的提升，几乎不靠模型本身，全靠你给它的上下文和管理流程。

采纳率最低的模块是 Hilt 配置相关——这点我们下个季度会专项补充 Rules。另外我们还发现，一次性让 AI 生成超过 300 行的代码块，修改率高达 60%。现在我们的策略是：单个 Prompt 限定输出不超过 150 行，大功能拆成多个小轮次。

## 70/30 法则与 AI 的天花板

说了这么多提效，我得泼一盆冷水：AI 不是万能的。在 Android 开发里，有些事它确实搞不定。

```
       AI 生成 70%                  人工 30%
  ┌─────────────────┐    ┌──────────────┐
  │ 样板代码（CRUD） │    │ 架构设计       │
  │ 单元测试         │    │ 核心业务逻辑   │
  │ UI 骨架（Compose）│    │ 性能敏感代码   │
  │ 数据类/Entity    │    │ 安全相关代码   │
  │ CI/Gradle 配置   │    │ 边界条件确认   │
  │ 正则/脚本/文档   │    │ 代码审查决策   │
  └─────────────────┘    └──────────────┘
```

AI 擅长什么？重复劳动。Entity、DAO、样板 UI、单元测试、正则表达式、数据类转换——这些它做得又快又好，质量经常超过我自己手写（毕竟我不会因为写第 50 个 DAO 就犯困）。

AI 搞不定什么？跨模块的业务编排——当一段逻辑涉及 3 个 Repository、2 个 ViewModel 之间的状态协同，AI 经常画出看起来合理但实际上有死锁风险的调用链。多线程并发控制也是翻车高发区——`withContext(Dispatchers.IO)` 里直接更新 UI、在 Composable 里用 `.value` 而非 `collectAsState()` 收集 StateFlow——这种错误语法上完全正确，编译不过才暴露，但编译不出来的问题 AI 根本不负责检查。

还有一个更隐蔽的问题：**AI 幻觉**。代码语法正确，注释清晰，命名规范，一切都看着很专业，直到你在某个边界条件下发现逻辑有坑。Room 里主线程调用没标记 suspend 的 insert（编译期 Room 会报错但 AI 不知道）、`LaunchedEffect(Unit)` 里做整个页面级操作导致不必要的重组——这些问题都有完美的代码外观和糟糕的运行时行为。

**防御策略就一条：对 AI 生成的代码保持"怀疑默认"。**任何代码都可能在特定条件下出错。优先给 AI 生成的逻辑写单元测试。对于你不熟悉的领域，先小范围验证再批量使用。另外我们团队建了一个"AI 幻觉案例库"，每次发现一个，就记下来全员 review。两个月攒了 23 个案例，出镜率最高的三位：线程切换、StateFlow 收集、LaunchedEffect key 遗漏。

### 真正限制你的不是 AI

坦率地说，现在 AI 的能力已经不是瓶颈。瓶颈在用它的人：

- Prompt 写不好，一个需求来回改 5 次——问题在你，不在 AI。
- 不懂审查 AI 输出，直接粘贴上线——Bug 是你放进去的，不是 AI。
- 对项目理解不够，给 AI 提供错误的上下文——AI 只能基于你给的信息工作。
- 期望不切实际，觉得 AI 应该直接从零产出生产级代码——它不是 AutoPilot，它是 Copilot。
- 从不度量效果，搞了一堆优化但不知道到底有没有用——没有数据的优化都是玄学。

最危险的一种心态是：长期过度依赖 AI，逐渐丧失对自己代码的理解能力。当 AI 写的代码出问题时，你发现自己 debug 的能力在退化——因为你不知道这段代码到底是被生成出来的，还是被设计出来的。

我的底线是：**负责核心业务逻辑和架构决策的 30% 代码，永远是我自己手写。**不是让 AI 生成再改，是自己从头写。这不是效率问题，是职业素养问题。你得确保在任何时候，你对系统中最重要的那部分代码有完整的 ownership。
