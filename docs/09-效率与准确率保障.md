## 9.1 Prompt 设计方法论

### 本节目标

掌握高质量 Prompt 的编写技巧，显著提升 AI 输出的准确率和可用性。

### 9.1.1 Prompt 五要素

一个高质量的 Prompt 由以下五个要素构成。缺少任一要素，AI 的输出质量都会出现明显下降。

| 要素 | 作用 | 缺少的后果 |
|------|------|-----------|
| **角色 (Role)** | 定义 AI 的身份和专业领域 | 输出泛化，缺少 Android 领域最佳实践 |
| **上下文 (Context)** | 提供项目背景和技术约束 | 生成代码与现有项目风格不一致 |
| **约束 (Constraints)** | 明确技术要求和限制条件 | 使用不合适的 API 或架构模式 |
| **示例 (Examples)** | 提供期望输出的参考 | 格式和结构偏离预期 |
| **输出格式 (Format)** | 指定返回内容的组织形式 | 输出杂乱，需要二次整理 |

### 9.1.2 Android 场景·Prompt 对比

以下面三个真实场景为例，展示 Prompt 质量对输出的影响。

**场景 A：UI 开发**

> 差：`帮我写一个登录页面`

> 好：
> ```
> 你是 Android 高级开发工程师，项目使用 Jetpack Compose + MVVM。
> 请实现登录页面的 UI 层（仅 UI 层），要求：
> - Material3 风格，支持暗黑模式自适应
> - 包含邮箱输入框、密码输入框、登录按钮
> - 邮箱格式校验（实时错误提示）
> - 密码长度 ≥ 6 位校验
> - Loading 态：登录按钮变为 CircularProgressIndicator 且不可点击
> - 错误态：Snackbar 展示服务端错误消息
> - 参考项目现有 LoginViewModel 接口，State 类命名为 LoginUiState
> - 不要使用 @OptIn(ExperimentalMaterial3Api::class)，全部用稳定 API
> - 输出格式：完整 Compose 文件，含 import
> ```

**场景 B：网络层封装**

> 差：`用 Retrofit 封装一个网络请求`

> 好：
> ```
> 你是 Android 高级开发工程师，项目使用 Retrofit + OkHttp + Kotlin Coroutines。
> 请实现一个通用的网络请求封装层，要求：
> - 使用 sealed class 封装 Result 状态（Success / Error / Loading）
> - 统一错误处理：网络异常 → NetworkException、服务端错误 → ServerException
> - 通过拦截器自动附加 Authorization Header（从 DataStore 读取 Token）
> - 支持请求重试（最多 2 次，指数退避）
> - 不使用 try-catch 包裹每次调用，改用 runCatching 或自定义扩展
> - 输出格式：3 个文件 — ApiResult.kt、NetworkInterceptor.kt、safeApiCall.kt
> ```

**场景 C：数据库操作**

> 差：`写一个 Room 数据库`

> 好：
> ```
> 你是 Android 高级开发工程师，项目使用 Room + KSP。
> 请实现文章收藏功能的本地数据库，要求：
> - 使用 Room 3.x，实体命名 ArticleEntity
> - 包含字段：id(String, PK)、title、url、favicon、timestamp(Long)
> - DAO 需要：insert、delete、getAll(Flow)、isFavorite(url): Boolean
> - Database 类命名 ArticleDatabase，单例模式
> - 不要使用 LiveData，全部用 Flow
> - 输出格式：ArticleEntity.kt + ArticleDao.kt + ArticleDatabase.kt
> ```

### 9.1.3 多轮对话策略

不要期望一次 Prompt 就拿到完美代码。推荐采用以下三阶段策略：

```
第一轮（骨架） → BuildModule.kt 的结构、接口定义、数据类
    ↓
第二轮（细化） → 实现每个接口、添加错误处理、边界条件
    ↓
第三轮（打磨） → 添加注释、提取常量、统一命名、ktlint 格式
```

| 阶段 | 目标 | 典型指令 |
|------|------|---------|
| 骨架 | 建立代码框架 | "先给出类结构和接口定义，不用完整实现" |
| 细化 | 填充业务逻辑 | "实现 DataRepository 的三个查询方法，重点处理空数据情况" |
| 打磨 | 质量收尾 | "优化这段代码的命名，添加 KDoc，处理所有 TODO" |

关键原则：**每一轮的上下文要积累**——第二轮 Prompt 中应包含第一轮的输出摘要，第三轮应引用前两轮达成的共识。这样才能让 AI 保持一致的风格和技术决策。

### 9.1.4 负面 Prompt

正面约束告诉 AI 要做什么，负面 Prompt 告诉 AI **不要做什么**。在 Android 开发中，负面 Prompt 同样重要。

| 场景 | 负面 Prompt 示例 |
|------|-----------------|
| 避免废弃 API | "不要使用 LocalActivity，改用 LocalContext" |
| 避免过度抽象 | "不需要抽象成 interface，直接写实现类即可" |
| 避免冗余代码 | "不要写所有字段的 builder 模式，用 data class 的 copy()" |
| 锁定 API 级别 | "不要使用 Android 14+ 专属 API，兼容到 API 26" |
| 排除旧范式 | "不要用 XML + findViewById，全部用 Compose" |

**实用技巧**：建立一个团队共享的 `negative-prompt.md`，按场景分类存放负面约束，每次开发直接引用对应段落。这样新人也能快速对齐团队的“不做清单”。

## 9.2 Rules 文件最佳实践

### 本节目标

设计一套高效、可维护的 Rules 文件体系，让 AI 始终“懂”你的项目。

### 9.2.1 Rules 的三级层级

Rules 文件不应全部堆在一个文件里。推荐采用分级设计，精确控制每个场景下 AI 接收到的上下文。

```
┌─────────────────────────────────────────┐
│  项目级 (CLAUDE.md / AGENTS.md)          │
│  技术栈、架构风格、通用编码规范、命名约定 │
├─────────────────────────────────────────┤
│  模块级 (module-name.md)                 │
│  模块职责、内部依赖、包结构、关键类说明   │
├─────────────────────────────────────────┤
│  功能级 (feature-name.md)                │
│  功能上下文、相关文件列表、约束条件、      │
│  关联 PRD / Figma 链接                   │
└─────────────────────────────────────────┘
```

**触发机制设计**：

```markdown
# module-network.md（示例）

# 触发条件：修改 app/network/ 或 lib/network/ 下的文件
# 如果涉及 api/、retrofit/、interceptor/ 子目录，额外加载对应段

## 技术约束
- Retrofit 2.9 + OkHttp 4.12 + Moshi（不用 Gson）
- 所有 API 接口放在 api/ 目录，按功能分包
- 拦截器统一管理在 interceptor/ 目录
- 错误模型统一使用 ApiError sealed class
- 禁止在 ViewModel 中直接调用 API，必须通过 Repository
```

**精准触发原则**：
- 使用 `globs` / `path` 关键词限制 Rules 的生效范围
- 避免在全局 Rules 中写模块特定内容——信息噪声会降低 AI 的准确率
- 每个 Rules 文件控制在 200 行以内，超过则拆分为更细粒度

### 9.2.2 Rules 生命周期

```
首次建立 → 项目初始化时，基于技术方案产出项目级 Rules
    ↓
随需演进 → 每次架构变动后同步更新（技术栈升级、模块重构）
    ↓
定期审查 → 每季度评审一次 Rules 的有效性，退役无效条目
    ↓
团队同步 → Rules 变更纳入 PR Review，核心 Rules 需团队共识
```

**常见反模式**：
- 写了 Rules 但从不更新（6 个月前的内容 AI 可能误用）
- 项目级 Rules 写了 500 行+（信息过载，AI 抓不住重点）
- 用 Rules 写业务逻辑文档（Rules 是约束，不是 PRD）

### 9.2.3 Android 场景 Rules 模板

以下提供 5 个高频 Rules 模板的核心内容：

**Kotlin 模板**

```markdown
## Kotlin 编码规范
- 优先使用 val，仅在必要时用 var
- 避免 !! 非空断言，使用 ?.let {} 或 ?: run {}
- 集合操作优先使用 Kotlin 标准库函数（map、filter、takeIf 等）
- sealed class 优先于 enum（需要携带数据时）
- 扩展函数命名：动词 + 名词，如 String.toSlug()
```

**Compose 模板**

```markdown
## Jetpack Compose 规范
- Composable 函数命名以大写字母开头，参数放前面
- State 提升原则：能上提就不下传
- Preview 函数必须用 @Preview(name = "描述"), showBackground = true
- 不直接传 Dp 值，使用 dimensionResource
- Theme 色值统一通过 MaterialTheme.colorScheme 取
```

**Gradle 模板**

```markdown
## Gradle 构建规范
- 使用 Version Catalog（libs.versions.toml）管理依赖版本
- 不在 build.gradle.kts 中硬编码版本号数字
- 插件声明在顶层 build.gradle.kts，模块中仅 apply
- 禁止在 dependencies 中使用 + 号（如 2.4.+）
```

**Room 模板**

```markdown
## Room 数据库规范
- Entity 命名以 Entity 结尾
- 所有 DAO 函数声明为 suspend 或返回 Flow
- 数据库版本迁移必须写 Migration 类，不允许 fallbackToDestructiveMigration
- 外键必须显式声明 onDelete 和 onUpdate 行为
```

**Hilt 模板**

```markdown
## Hilt 依赖注入规范
- Repository 全部用 @Inject constructor，不用 @Module 单独提供
- 第三方库的实例用 @Module + @Provides 提供
- ViewModel 必须用 @HiltViewModel + @Inject constructor
- 测试中用 @UninstallModules 替换真实模块
```

## 9.3 AI Review 三层防线

### 本节目标

建立 AI 代码生成的质量保障体系，在效率与质量间找到最优平衡点。

### 9.3.1 三层防线模型

```
┌─────────────────────────────────────────────────────┐
│  第一层 · AI 自检                                    │
│  代码生成后立即让 AI 审查自己的输出                    │
│  投入：约 1-2 分钟 / 次                              │
│  拦截率：~40%（命名错误、遗漏 import、逻辑矛盾等）     │
├─────────────────────────────────────────────────────┤
│  第二层 · 工具自动检查                                │
│  ktlint + Detekt + Android Lint + 单元测试            │
│  投入：约 5-10 秒 / 次（CI 自动执行）                 │
│  拦截率：~35%（代码风格、潜在 Bug、性能警告等）        │
├─────────────────────────────────────────────────────┤
│  第三层 · 人工审查                                    │
│  CR 聚焦业务逻辑、架构合理性、可维护性                 │
│  投入：不定                                          │
│  拦截率：~15%（业务逻辑错误、架构偏差、安全漏洞等）    │
└─────────────────────────────────────────────────────┘
```

### 9.3.2 第一层：AI 自检

这是投入产出比最高的一层。操作方法：在 AI 生成代码之后，立即发一条审查指令。

**自检 Prompt 模板**：

```
请审查你刚才生成的代码，逐项检查：
1. import 是否完整且无冗余
2. 命名是否与项目已有代码风格一致
3. 空安全和边界条件是否覆盖
4. 是否正确使用了项目规定的框架/库
5. 是否有明显逻辑错误或死代码
逐条输出发现的问题和修复建议。
```

**实测效果**（基于 50 次 AI 自检统计）：

| 拦截项 | 出现频次 | 典型示例 |
|--------|---------|---------|
| 缺少 import | 32% | 忘记 import viewModel()、import mutableStateOf |
| 命名不一致 | 18% | MutableStateFlow 写成了 LiveData 风格命名 |
| 边界条件遗漏 | 14% | 网络请求未处理 loading 态 |
| 框架 API 误用 | 10% | Compose 中直接操作 MutableState 的 value |
| 无影响 | 26% | 代码本身质量良好 |

**技巧**：如果 AI 给出“代码没有问题”的结果，可以追问一句“请从代码审查者的角度，至少找出 2 个可以改进的地方”。

### 9.3.3 第二层：工具自动检查

将以下工具集成到 CI 管线中，每次提交自动触发：

| 工具 | 检查类型 | 配置要点 |
|------|---------|---------|
| **ktlint** | Kotlin 代码风格 | 使用 `.editorconfig` 统一缩进、换行规则 |
| **Detekt** | 代码异味、复杂度 | 启用 Android 专属规则集，阈值按团队习惯调整 |
| **Android Lint** | Android 专项检查 | 关注 UnusedResources、HardcodedText、MissingPermission 等 |
| **单元测试** | 功能正确性 | AI 生成的测试覆盖率至少 60%，核心逻辑 80%+ |

**Gradle 配置示例**：

```kotlin
// 在 app/build.gradle.kts 中启用严格的 Lint 检查
android {
    lint {
        checkDependencies = true
        abortOnError = true
        htmlReport = true
        xmlReport = true
        checkReleaseBuilds = true
        ignoreTestSources = false
    }
}

// Detekt 配置示例 (detekt.yml)
style:
  MagicNumber:
    active: true
    ignoreNumbers: ['-1', '0', '1', '2']
  WildcardImport:
    active: true
complexity:
  LongParameterList:
    active: true
    threshold: 6
```

**CI 集成**（GitHub Actions 片段）：

```yaml
- name: Run ktlint
  run: ./gradlew ktlintCheck

- name: Run Detekt
  run: ./gradlew detekt

- name: Run Lint
  run: ./gradlew lintDebug

- name: Run Unit Tests
  run: ./gradlew testDebugUnitTest
```

关键在于 **abortOnError = true**——不要让警告悄悄溜过。宁可构建失败，也不积累技术债。

## 9.4 度量与持续改进

### 本节目标

建立 AI Coding 的效果度量体系，用数据驱动 AI 使用方式的持续优化。

### 9.4.1 四项核心指标

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|---------|--------|
| **代码采纳率** (Acceptance Rate) | AI 生成代码被直接采纳的比例 | 采纳行数 / 生成行数 × 100% | ≥ 60% |
| **修改率** (Edit Rate) | AI 生成代码被修改的比例 | 修改行数 / 生成行数 × 100% | ≤ 40% |
| **Bug 引入率** | AI 生成代码导致的新 Bug 密度 | AI 引入 Bug 数 / AI 生成 KLOC | ≤ 5 / KLOC |
| **开发吞吐量** | 单位时间完成的功能点数 | 功能点数 / 人天 | 提升 ≥ 30% |

### 9.4.2 数据采集方法

**方法一：Git 关键字段统计**

```bash
# 通过 Git Commit Message 标记 AI 辅助的提交
git log --grep="\[ai-assisted\]" --author="用户名" --since="2026-01-01" \
  --numstat --pretty=format:"%H %s"
```

团队约定：所有由 AI 辅助生成的代码，在 commit message 中标注 `[ai-assisted]`。这样可以快速区分 AI 生成 vs 人工编写的代码。

**方法二：CR 标记分类**

在 Code Review 系统中为 AI 代码建立标签体系：

| 标签 | 含义 |
|------|------|
| `ai-accepted` | AI 生成代码直接采纳 |
| `ai-edited` | AI 生成代码经修改后采纳 |
| `ai-rejected` | AI 生成代码全部舍弃 |
| `ai-bug` | AI 生成代码引入 Bug |

**方法三：CI 嵌入度量脚本**

```kotlin
// 在 Gradle 任务中统计 AI 标记行
tasks.register("countAiCode") {
    doLast {
        val output = ByteArrayOutputStream()
        exec {
            commandLine(
                "git", "blame", "--line-porcelain",
                "--grep=\\[ai-assisted\\]"
            )
            standardOutput = output
        }
        val aiLines = output.toString().lines().count { it.contains("ai-assisted") }
        println("AI 辅助代码行数: $aiLines")
    }
}
```

### 9.4.3 季度评估模板

每个季度进行一次 AI Coding 效果复盘，建议模板如下：

```
## 2026 Q1 AI Coding 效果评估

### 核心指标
| 指标 | 本季度 | 上季度 | 变化 |
|------|-------|-------|------|
| 代码采纳率 | 68% | 55% | +13% |
| 修改率 | 32% | 45% | -13% |
| Bug 引入率 | 3.2/KLOC | 5.8/KLOC | -2.6 |
| 开发吞吐量 | 8.5 FP/人天 | 6.2 FP/人天 | +37% |

### 主要改进措施

## 9.5 效率提升的边界

### 本节目标

诚实面对 AI Coding 的能力边界，建立合理的效率预期，避免过度依赖和盲目信任。

### 9.5.1 AI 擅长与不擅长的领域

| | AI 擅长 ✅ | AI 不擅长 ❌ |
|------|-----------|-------------|
| **代码类** | 样板代码（CRUD、Entity、DAO）；正则表达式；数据类转换；mock 数据生成；Jetpack Compose UI 骨架 | 复杂业务逻辑的跨模块编排；多线程并发控制；内存泄漏排查；ANR 定位 |
| **工程类** | 单元测试生成；Gradle 配置模板；CI 脚本编写；AndroidManifest 声明；docs 文档生成 | 构建性能优化（AGP 升级兼容）；ProGuard/R8 规则调优；安全审计和漏洞修复 |
| **设计类** | 输出标准模式代码；设计模式应用（Repository、UseCase）；命名建议 | 架构决策（单模块 vs 多模块）；技术选型权衡；领域建模和抽象边界 |

### 9.5.2 人机协作黄金比例

基于团队实践数据的经验值：

```
         AI 生成 70%              人工 30%
    ┌─────────────────┐    ┌──────────────┐
    │ 样板代码          │    │ 核心业务逻辑   │
    │ 单元测试          │    │ 性能敏感代码   │
    │ UI 骨架          │    │ 安全相关代码   │
    │ 文档             │    │ 架构设计       │
    │ CI 配置          │    │ 代码审查决策   │
    │ 数据类/Entity     │    │ 边界条件确认   │
    │ 正则/脚本         │    │ 技术方案评审   │
    └─────────────────┘    └──────────────┘
```

注意：70/30 不是死板的代码行数比例，而是精力分配比例。一个 Activity 的样板代码 AI 可以 100% 生成，但其中的核心业务逻辑片段需要人工 100% 审视。

### 9.5.3 警惕“AI 幻觉”

AI 幻觉是指：代码语法正确、注释清晰、命名规范，**但逻辑有隐蔽缺陷**。这是最有欺骗性的一类问题。

**常见 AI 幻觉实例**：

| 场景 | 幻觉表现 | 正确做法 |
|------|---------|---------|
| 线程切换 | `withContext(Dispatchers.IO)` 中直接更新 UI | IO 结果切换到 Dispatchers.Main |
| StateFlow 使用 | 在 Composable 中直接 `.value` 访问，未收集 | 用 `collectAsState()` 收集状态更新 |
| 生命周期 | 在 `LaunchedEffect(Unit)` 中做整个页面级操作 | 应绑定具体 key，避免不必要的重组 |
| 数据库操作 | Room 中主线程调用未标记 suspend 的 insert | 编译期 Room 会报错，但 AI 不在编译期 |
| Compose Recomposition | 未用 `remember` 缓存高成本计算 | 滑动时频繁重组导致卡顿 |

**防御策略**：
1. 对 AI 生成的代码保持“怀疑默认”——任何代码都可能在特定边界条件下出错
2. 优先写单元测试覆盖 AI 生成的逻辑
3. 对于不熟悉的领域，先用小范围验证，再批量使用
4. 建立团队共享的“AI 幻觉案例库”，定期 review

### 9.5.4 效率提升的瓶颈不在 AI

最终约束效率的，不是 AI 的能力，而是使用 AI 的人的能力：

| 瓶颈点 | 典型表现 | 突破方法 |
|--------|---------|---------|
| **Prompt 能力不足** | 反复修改 4-5 次才能拿到可用的代码 | 刻意练习 Prompt 五要素，积累 Prompt 库 |
| **不懂审查 AI 输出** | 直接粘贴使用，上线后出 Bug | 建立 AI Review 三层防线，严格执行 |
| **对项目理解不够** | 给 AI 提供错误或不完整的上下文 | 项目启动时投入时间建立 Rules 体系 |
| **期望不切实际** | 期望 AI 直接产出生产级代码 | 理解 AI 的能力边界，人机协作而非替代 |
| **缺少度量反馈** | 不知道 AI 到底提升了多少效率 | 每周记录 AI 采纳率和 Bug 引入数据 |

## 踩坑记录

> 以下为团队在实际使用中遇到的典型踩坑，持续更新中。

### 坑 1：Prompt 越详细越好的误区

**现象**：在新项目中给了一个 500 字的超详细 Prompt，AI 输出了看起来很全面的代码，但实际上对项目结构和约定理解错误，返工率接近 80%。

**原因**：新项目 AI 缺少上下文，长 Prompt 中的假设与项目实际不一致，AI 基于错误假设生成了大量代码。

**解决**：新项目先用短 Prompt 探索，确认 AI 理解正确后再逐步加细节。多轮对话优于一次性长篇 Prompt。

### 坑 2：Rules 文件过多导致上下文膨胀

**现象**：贪心地把 8 个模块的 Rules 全部纳入全局加载，结果 AI 的 Token 消耗暴增 5 倍，且输出质量不升反降。

**原因**：上下文过长导致 AI 无法有效聚焦当前任务，关键约束被淹没在大量无关信息中。

**解决**：按精准触发策略重写 Rules，最多同时激活 2-3 个 Rules 文件。全局 Rules 控制在 150 行以内。

### 坑 3：忽略工具链版本兼容

**现象**：AI 生成的 Gardle 配置引用了 AGP 8.5 的 API，但项目实际使用 AGP 7.4，导致构建失败。

**原因**：Rules 文件未声明工具链版本约束，AI 默认使用训练数据中的最新版本。

**解决**：在项目级 Rules 中显式声明所有工具链版本号：

```markdown
## 工具链版本约束
- AGP: 7.4.2
- Kotlin: 1.9.22
- Compose BOM: 2024.01.00
- Room: 2.6.1
- Hilt: 2.51.1
- 目标 SDK: 34
- 最低 SDK: 26
- Gradle: 8.4
```

### 坑 4：AI Review 第一层检查的“自负”偏差

**现象**：让 AI 自检自己的代码，经常返回“代码质量良好，无明显问题”。

**原因**：AI 对刚生成的代码有确认偏差（confirmation bias），不愿承认错误。

**解决**：自检指令中加入“强制找茬”——"请从代码审查者的角度，至少找出 2 个可以改进或需要确认的地方"。实验表明这样可以将问题发现率从 26% 提升到 72%。
