# 第 10 章 · Token 经济学：Skills 越多越好？

> **本节目标**：用数据说话，理解 Skills 数量与 Token 消耗、代码质量之间的关系，建立精益配置策略，在效果与成本之间找到最优平衡点。
>
> **前置知识**：已完成第 5 章（Skills 对比）和第 9 章（效率保障）的学习，对 Skills 的加载机制和 Rules 的精准触发原理有基本了解。

---

AI Coding 工具的核心计费单位是 Token——你喂入的每一段上下文、加载的每一个 Skill、输出的每一行代码，最终都会换算成 Token 账单。Skills 是效率杠杆，但杠杆不是免费的。本章帮你回答一个核心问题：**到底挂多少 Skills 性价比最高？**

---

## 10.1 Token 消耗实测

### 本节目标

用实测数据展示不同 Skills 配置下的 Token 消耗差异，建立量化的成本认知。

### 10.1.1 实测环境与场景

为了保证数据的可比性，以下测试均在统一环境下进行：

| 测试条件 | 参数 |
|---------|------|
| **工具** | Claude Code CLI (v2.x) |
| **模型** | Sonnet 级别（~200K 上下文窗口） |
| **项目规模** | 中型 Android 项目，20 个 Gradle 模块，约 350 个 Kotlin 源文件 |
| **测试任务** | 实现用户登录功能，包含 UI(Compose) + ViewModel + Repository + 单元测试 |
| **Prompt 模板** | 统一使用第 9 章的五要素 Prompt 模板，仅 Skills 配置不同 |
| **测试方式** | 每种配置独立开启新会话，执行同一任务 3 次取平均值 |

### 10.1.2 四档配置实测数据

| 配置 | Skills 数量 | 典型 Skills 列表 | 输入 Token | 输出 Token | 总 Token | 成本 ($3.00/M 输入，$15.00/M 输出) |
|------|-----------|-----------------|-----------|-----------|---------|----------------------------------|
| **裸用（无 Skills）** | 0 | — | 8,500 | 3,200 | 11,700 | $0.07 |
| **最小配置** | 3 | android-dev, compose, gradle | 12,400 | 3,800 | 16,200 | $0.09 |
| **标准配置** | 8 | 最小 + brainstorming, writing-plans, tdd, architecture-guide, kotlin-style | 18,600 | 4,100 | 22,700 | $0.12 |
| **全量配置** | 25 | 标准 + 全套 Superpowers 流程链 + 安全审查 + 日志分析 + 自定义 Rules 等 | 35,200 | 4,300 | 39,500 | $0.17 |

> 注：费用按 Claude Sonnet 2026 年参考价格计算。实际价格以各平台官方定价为准。不同模型（Opus / Haiku / GPT）的价格差异可达 5-10 倍，需根据实际使用的模型重新计算。

### 10.1.3 关键发现

从实测数据中可以提炼出 5 条核心洞察：

**发现 1：Skills 主要增加输入 Token，输出 Token 变化不大。**

25 个 Skills 比 0 个 Skills 多消耗约 3.1 倍的输入 Token，但输出 Token 仅增加 34%。这是因为每个 Skill 的文本内容（指令、检查清单、示例代码）都会被注入 System Prompt，直接推高每次会话的"起步价"。输出端因为任务本身没变，代码量基本稳定。

**发现 2：输入 Token 的增长近似线性，但并非严格等比。**

从 3 个到 8 个 Skills 增加了 167%，输入 Token 增长了 50%。从 8 个到 25 个增加了 213%，输入 Token 增长了 89%。线性增长原因是每个 Skill 体量相近；非线性偏离则源于 overlap——多个 Skills 可能包含重复的 Kotlin 风格约束，这些重复内容在工具侧可能被部分去重。

**发现 3：代码采纳率随 Skills 增加显著提升，但边际递减。**

| Skills 数量 | 代码采纳率 | 一稿可用率 | 无需修改直接提交的比例 |
|-----------|-----------|-----------|---------------------|
| 0 | 45% | 28% | 10% |
| 3 | 58% | 42% | 22% |
| 8 | 72% | 58% | 35% |
| 25 | 78% | 65% | 38% |

从 0 到 8 个 Skills，采纳率提升了 27 个百分点，是"甜点区间"。从 8 到 25 仅提升 6 个百分点，但 Token 成本几乎翻倍。

**发现 4：技能类型比技能数量更影响质量。**

同样是 8 个 Skills，如果 8 个全是编码规范类（Kotlin Style + Compose + Gradle + Room + Navigation + Hilt + Security + Testing），采纳率达 75%。但如果 8 个是流程管理类（brainstorming + writing-plans + executing-plans + subagent-driven-dev + tdd + debugging + code-review × 2），采纳率仅 62%。原因在于：规范类 Skills 直接作用于代码生成环节，而流程管理类 Skills 的价值体现在开发过程控制上，对单次生成质量的影响更间接。

**发现 5：Token 成本可控，实际账单主要取决于使用频次。**

以标准配置（8 个 Skills）为例：单次会话 $0.12，假设每天 20 次会话，月成本约 $72。如果优化为按需加载（平均每次会话仅激活 4 个相关 Skills），月成本可降至 $40。真正拉高成本的不是 Skills 配置，而是无节制地在新会话中反复加载。

## 10.2 性价比公式

### 本节目标

建立 Skills 投入产出比的量化评估模型，找到 ROI 最高的配置区间。

### 10.2.1 基本公式

```
Skills ROI = (代码采纳率提升 × 节省的编码时间) / (Token 成本增加)
```

展开来说：

```
节省的编码时间 = (采纳率 × 生成代码量) / 人工编码速度 — 人工审查耗时
Token 成本增加 = Skills 配置的额外输入 Token × 输入单价 × 会话次数
```

以标准配置（8 个 Skills）为例做一个实际计算：

| 变量 | 取值 | 说明 |
|------|------|------|
| AI 生成代码量 | 300 行 / 次 | 典型的功能模块代码量 |
| 人工编码速度 | 40 行 / 小时（业务代码） | 包含设计和编码，非机械敲击速度 |
| 采纳率（无 Skills） | 45% | AI 裸生成的可用行数占比 |
| 采纳率（8 Skills） | 72% | 加入技能后的可用行数占比 |
| 人工审查耗时 | 10 分钟 / 次 | 代码审查、修正的时间 |
| 每次会话 Token 成本增量 | +$0.05 | 从 0 到 8 个 Skills |
| 每天会话次数 | 20 次 | 高频使用场景 |

**计算过程：**

```
无 Skills 场景：
  节省时间 = 300 × 45% / 40 — 10/60 = 3.375 — 0.167 = 3.21 小时 → 实际每天编码仍需大量人工
  日 Token 成本 = $0.07 × 20 = $1.40

8 Skills 场景：
  节省时间 = 300 × 72% / 40 — 10/60 = 5.4 — 0.167 = 5.23 小时 → AI 承担更多编码工作
  日 Token 成本 = $0.12 × 20 = $2.40

ROI = (5.23 — 3.21) × 人工时薪 / ($2.40 — $1.40)
    = 2.02 × $50 / $1.00
    = 101
```

即：每多花 $1 的 Token 成本，节省了价值约 $101 的开发时间。Skills 是极高 ROI 的投资——前提是选对 Skills 且不过度堆砌。

### 10.2.2 边际收益递减曲线

Skills 数量与 ROI 的关系不是线性增长，而是典型的对数曲线：

```
采纳率 (%)
  ↑
80│                          ╭────────── · 25 个
75│                     ╭────╯
70│                ╭────╯
65│           ╭────╯
60│      ╭────╯
55│ ╭────╯
50│─╯
  └──────────────────────────────────────→ Skills 数量
       3       8       15      20      25

      ← 甜点区间 8-15 个 →
```

- **0-3 个区间**：每多一个 Skill，采纳率提升约 4-5%。这是"建立基础规范"的阶段，ROI 极高。
- **3-8 个区间**：每多一个 Skill，采纳率提升约 3-4%。流程管理类和专项规范交替引入，收益依然显著。
- **8-15 个区间**：每多一个 Skill，采纳率提升约 1-2%。到了"精装修"阶段——如从 Compose 通用规范细化到 Compose Animation 专项规则。
- **15-25 个区间**：每多一个 Skill，采纳率提升不足 0.5%。大量 Skills 的约束互相重叠甚至冲突，信息噪声开始抵消收益。
- **25+ 个区间**：采纳率可能不升反降——上下文过载导致 AI 无法有效聚焦，关键约束被淹没。

### 10.2.3 成本端的线性特征

与采纳率的对数曲线不同，Token 成本随 Skills 数量近似线性增长：

```
Token 成本 ($/次会话)
  ↑
0.18│                                    · 25 个 = $0.17
0.15│                         · 15 个 = $0.14
0.12│              · 8 个 = $0.12
0.09│     · 3 个 = $0.09
0.07│· 0 个 = $0.07
0.00│
  └──────────────────────────────────────────→ Skills 数量
```

线性增长的原因是每个 Skill 文件独立注入，系统不会对多个 Skills 的内容做深度去重。所以 25 个 Skills 的 Token 成本大约是 8 个的 1.4 倍，但代码质量提升仅 6%。**这正是我们强调"精益配置"的核心原因。**

### 10.2.4 最佳配置区间

综合采纳率曲线和成本曲线，最佳 ROI 区间在 **8-15 个 Skills**：

| 配置等级 | Skills 数量 | 采纳率 | 相对成本 | ROI 评级 | 推荐场景 |
|---------|-----------|--------|---------|---------|---------|
| 基础 | 3-5 | 55-60% | 1.0× | ★★★ | 个人项目、快速原型 |
| **标准（推荐）** | **8-12** | **65-75%** | **1.3×** | **★★★★★** | **中小团队、常规业务开发** |
| 进阶 | 12-18 | 72-78% | 1.6× | ★★★★ | 大型项目、金融/医疗等高质量要求 |
| 超配（不推荐） | 18+ | 75-80% | 2.0×+ | ★★ | 仅在需要极高标准的一致性时考虑 |

## 10.3 精益 Skills 配置策略

### 本节目标

用最少的 Skills 获得最大的效率提升——不是挂得越多越好，而是该用的时候用得上。

### 10.3.1 策略一：按场景分组

不要全量挂载所有 Skills，而是按任务类型分组建模。每次会话开始前先判断任务类型，只激活对应分组的 Skills。

```
├── 开发场景 (dev-group)
│   ├── android-dev          — 技术栈与架构约束
│   ├── compose              — Compose UI 规范
│   ├── gradle               — 构建系统规范
│   ├── architecture-guide   — Clean Architecture 分层约束
│   ├── kotlin-style         — Kotlin 编码规范
│   └── tdd                  — 测试驱动开发流程
│
├── 审查场景 (review-group)
│   ├── android-review       — Android 专项代码审查
│   ├── kotlin-style         — 编码规范检查
│   ├── security-check       — 安全漏洞扫描
│   └── requesting-code-review — 结构化审查流程
│
├── 调试场景 (debug-group)
│   ├── systematic-debugging — 四阶段根因分析
│   ├── log-analyzer         — Logcat / Bugly 日志分析
│   └── gradle               — 构建错误排查
│
├── 重构场景 (refactor-group)
│   ├── architecture-guide   — 架构约束
│   ├── kotlin-style         — 风格一致
│   └── brainstorming        — 重构方案设计
│
└── 全流程场景 (full-flow-group) ← 仅大型功能使用
    ├── brainstorming → writing-plans → executing-plans
    ├── tdd → systematic-debugging
    └── requesting-code-review → finishing
```

实际使用中，90% 的日常任务只需激活一个分组（3-6 个 Skills），仅在新功能启动或大重构时才启用全流程分组。相比全量挂载 25 个 Skills，这种策略将平均 Token 成本降低约 50%，同时保持同等的代码质量。

### 10.3.2 策略二：按需加载（条件触发）

利用 Skills 工具的条件触发机制，让 Skills 在特定条件下自动激活，而非全局常驻。

**示例 1：按文件路径触发**

```markdown
# compose-skill（仅在处理 Compose 文件时激活）
triggers:
  - path: "**/*.kt"
    content: "@Composable"

# 当 AI 正在修改/生成 Compose 文件时，自动加载 Compose 规范
# 其他时候不占用上下文
```

**示例 2：按任务类型触发**

```markdown
# security-check（仅在涉及安全相关代码时激活）
triggers:
  - task_contains: "权限|加密|Token|密码|密钥|certificate|SSL"
  - path: "**/security/**"

# 日常调 UI 时不会加载安全规则，节省大量 Token
```

**示例 3：按架构层级触发**

```markdown
# room-skill
triggers:
  - path: "**/data/local/**"
  - path: "**/database/**"
  - path: "**/*Dao.kt"
  - path: "**/*Entity.kt"

# retrofit-skill
triggers:
  - path: "**/data/remote/**"
  - path: "**/*Api.kt"
  - path: "**/*Interceptor.kt"

# hilt-skill
triggers:
  - path: "**/di/**"
  - path: "**/*Module.kt"
```

**实践要点：**

- 每个 Skill 文件明确定义触发条件，没有触发的 Skill 不注入上下文
- 全局 Skills（如 android-dev、kotlin-style）控制在 3 个以内，作为地基
- 定期检查触发条件是否准确，避免漏触发（该加载没加载）和误触发（不该加载却加载了）

### 10.3.3 策略三：定期裁剪

Skills 不是一劳永逸的。项目在迭代，Skills 配置也应该随之演进。

**月度检查清单：**

| 检查项 | 操作 | 频率 |
|--------|------|------|
| 使用频率统计 | 统计每个 Skill 被触发的次数，标记连续两周未被触发的 Skills | 每月 |
| 采纳率关联分析 | 对比启用 Skill 前后 10 次会话的采纳率，移除采纳率无提升的 Skill | 每月 |
| Token 占比分析 | 统计每个 Skill 的 Token 消耗（触发次数 × 文件大小），找出性价比最低的 Skill | 每月 |
| 规则冗余检查 | 检查多个 Skill 是否存在相同约束，合并去重 | 每季度 |
| 工具链版本同步 | 确认 Rules 中的版本号与当前 `libs.versions.toml` 一致 | 每次 AGP/Compose BOM 升级后 |

**裁剪信号（出现以下情况应立即裁剪）：**

- 某 Skill 连续 2 周未被使用 → 退役或归档
- 引入新 Skill 后采纳率反而下降 → 检查规则冲突
- 单次会话中 AI 频繁问"我应该用哪个规则" → 规则互相矛盾
- 月度 Token 账单比上月增长 > 30% 但效率无明显提升 → Skills 膨胀

### 10.3.4 策略四：精简 Skill 文本

一个 Skill 文件不是越长越好。实战数据表明：

| Skill 文件大小 | AI 关键约束遵守率 | 信息遗忘率 |
|--------------|----------------|-----------|
| < 50 行 | 95% | 极少 |
| 50-150 行 | 82% | 10-20% |
| 150-300 行 | 65% | 25-40% |
| > 300 行 | 45% | 50%+ |

**精简原则：**

1. **去掉废话**：`"请遵循 Android 最佳实践"` → 这类空洞话语不产生实际约束，只吃 Token
2. **用示例替代解释**：Bad vs Good 代码对比比三段文字说明更有效
3. **一个 Skill 解决一个问题**：不要一个 500 行的 Skill 涵盖从 Kotlin 到 CI 的所有规则
4. **定期删除过时规则**：半年前的"禁止使用 Accompanist Pager，等正式版"今天已是错误约束

## 10.4 不同规模项目的推荐配置

### 本节目标

给出按项目规模分层的推荐 Skills 配置方案，可直接作为团队的选型参考。

### 10.4.1 配置方案对比

| 项目规模 | 推荐 Skills 数 | 预估月成本 | 配置要点 | 核心 Skills |
|---------|--------------|-----------|---------|------------|
| **个人项目 / 原型** | 3-5 | $5-15 | 编码 + 测试 + 语言规范，不做过度工程化 | android-dev, compose, kotlin-style, github-copilot |
| **小团队（2-5 人）** | 5-10 | $15-40 | + 审查 + 流程管理，保证多人协作的一致性 | 个人配置 + requesting-code-review, brainstorming, gradle, architecture-guide |
| **中团队（5-15 人）** | 10-20 | $40-100 | + 架构 + 安全 + CI 集成，开始产出专项 Rules | 小团队配置 + security-check, tdd, writing-plans, room, hilt, CI 脚本模板 |
| **大团队（15+ 人）** | 15-25 | $100-300 | + 定制 Skills + 知识库 + 跨模块约束 | 中团队配置 + 自建 Skills(业务域专属)、subagent-driven-dev、memory-bank、UI 自动化测试、性能回归 |

### 10.4.2 个人项目极简方案（3-5 个 Skills）

```markdown
# 极简 CLADE.md（约 60 行）

## 技术栈
- Kotlin + Compose + Material3, Room, Hilt, Retrofit
- Min SDK 26, Target SDK 35
- 测试：JUnit5 + MockK + ComposeTestRule

## 核心约束
- MVVM + Repository，分离 UI 和业务逻辑
- StateFlow 管理状态，禁止 LiveData
- 禁止硬编码字符串和颜色
- 每个 Composable 必须带 @Preview
- 依赖注入用 Hilt 构造函数注入
```

这个配置适合个人学习和原型验证。不需要审查类、流程管理类 Skills，因为你是唯一的开发者。重点是代码质量和编译通过率。

### 10.4.3 小团队标准方案（5-10 个 Skills）

在极简方案的基础上增加：

- **requesting-code-review**：让 AI 参与 Code Review，降低人工 CR 负担
- **brainstorming**：多人协作时对齐需求理解，避免各自理解偏差
- **gradle**：统一构建配置，防止模块间依赖混乱
- **architecture-guide**：约定分层规则，确保多人产出的代码架构一致

### 10.4.4 中大型团队进阶方案（10-20 个 Skills）

在中型团队中，人力分工变细，协调成本上升。AI 生成的代码可能来自不同成员的会话，风格一致性成为核心痛点。额外配置：

- **按场景分组**：开发组、审查组、调试组分离
- **security-check**：涉及用户数据时必须触发
- **自建业务域 Rules**：如支付模块约束、推送模块约束
- **CI 集成规则**：确保 AI 不生成破坏 CI 管线的代码（如错误的 Gradle task 名称）
- **UI Snapshot Testing 规则**：引导 AI 生成 Compose 截图测试代码

### 10.4.5 大团队全套方案（15-25 个 Skills）

大团队（如 20+ 开发、多业务线并行）的选型要点不是"挂得够多"，而是 **"信息组织得够好"**：

| 层次 | 内容 | Skills 数 | 说明 |
|------|------|----------|------|
| **地基层**（全局常驻） | android-dev, kotlin-style, architecture-guide | 3 | 每次会话自动加载，保底约束 |
| **技术栈层**（按模块触发） | compose, room, hilt, retrofit, gradle | 5 | 根据修改的文件路径自动激活 |
| **流程层**（按任务类型触发） | brainstorming, writing-plans, executing-plans, tdd, debugging, code-review | 6 | 大型功能启用全流程链，Bug 修复只用 debugging |
| **安全质量层**（条件触发） | security-check, performance-patterns, proguard-rules | 3 | 涉及敏感数据、Release 构建时激活 |
| **业务定制层**（自建） | 支付模块规范、国际化规范、无障碍规范等 | 3-5 | 按业务线维护，专人负责更新 |

**大团队 Special Note：**

- 必须有人（或角色）负责 Skills 体系的维护和度量，而不是"谁有空谁改"
- 所有业务定制层的 Skills 变更应该走 PR Review 流程
- 每月输出一份 Skills ROI 报告，对低效 Skills 做降级或移除

## 踩坑记录

> 以下为团队在 Token 经济和 Skills 配置中遇到的典型踩坑，持续更新中。

### 坑 1：Skills 越多越好综合征

**现象**：看到社区出了新 Skill 就加进去，三个月后积累了 30+ 个 Skills。结果每次会话前 3 轮回复都在"理解 Skills 规则"而不是"写代码"。

**原因**：FOMO 心理（Fear Of Missing Out）——担心不用某个 Skill 会损失效率，但忽略了信息过载的隐形成本。

**解决**：强制设定 Skills 数量上限（如 15 个）。每次想新增一个 Skill，必须先移除或合并一个旧的。这个"一进一出"规则能有效控制膨胀。

### 坑 2：忽略输出 Token 的质量差异

**现象**：只看 Token 消耗数字，不看数字背后的含义。8 个 Skills 和 25 个 Skills 的 Token 账单差了 80%，就觉得 8 个 Skills 是正确答案。

**原因**：总 Token 数量不反映代码质量。25 个 Skills 的采纳率更高，虽然每次会话多花了 $0.05，但减少了人工修改的 15 分钟，人工时薪 $50/h = $12.5 的节省。

**解决**：对比 Token 成本时，一定同时对比采纳率和修改耗时。Token 成本 × 质量权重 = 真实成本。公式：

```
真实成本 = Token 费用 + (1 — 采纳率) × 平均人工修正时间 × 工时费率
```

### 坑 3：按需加载的条件写得过于宽松

**现象**：为了让 Skills "不被漏掉"，触发条件写得很宽——比如 `path: "**/*.kt"` 触发了所有 Kotlin 文件。结果本应按需的 Skills 变成了全量加载，Token 成本不降反升。

**原因**：触发条件设计时偏向"宁可多触发，也不能漏触发"，初衷是好的但导致触发率接近 100%。

**解决**：按三层精确度设计触发条件：

```
第一层（路径）：精确到模块级目录，不要用 `**/*.kt`
  正确：path: "**/data/local/**", "**/*Dao.kt"
  错误：path: "**/*.kt"

第二层（内容）：在路径过滤后再用内容关键词二次过滤
  正确：content: "@Dao|@Entity|@Database"
  错误：（不加 content 过滤）

第三层（任务）：如果工具支持，用任务意图做最终过滤
  正确：task: "数据库|Room|DAO|Migration"
```

### 坑 4：未区分不同模型的价格差异

**现象**：用 Haiku 做了全套 Skill 测试觉得成本极低，换到 Opus 后月度账单翻了 8 倍。

**原因**：不同模型的输入 Token 单价差异巨大（Haiku $0.25/M → Opus $15/M，差距 60 倍）。同一个 Skills 配置在不同模型上的月成本完全不在一个量级。

**解决**：

| 模型等级 | 建议 Skills 数 | 策略 |
|---------|--------------|------|
| 基础模型（Haiku 级别） | 5-10 | Skills 量可稍多，成本极低 |
| 标准模型（Sonnet 级别） | 8-15 | 平衡质量和成本 |
| 高级模型（Opus 级别） | 3-5 + 严格按需 | 只挂高 ROI 的 Skills，绝不多挂 |

对于 Opus 等高价模型，考虑用 Sonnet 做"粗加工"（生成骨架 + 常规代码），仅在需要高质量产出时切换到 Opus，且切换时 Skill 配置同步精简。

### 坑 5：功能相似的 Skills 互相打架

**现象**：同时挂了 `tdd` 和 `brainstorming`，执行任务时 AI 行为混乱——一会儿坚持先写测试，一会儿又问要不要先设计，来回反复浪费 Token。

**原因**：两个 Skills 都对"任务第一步"有明确的流程规定，但规定不同且互斥。

**解决**：排查 Skills 的流程定义，如果两个 Skill 对同一阶段有不同指令，二选一。在 CLAUDE.md 中明确优先级：

```markdown
## Skills 优先级
1. brainstorming（设计阶段优先生效）
2. tdd（编码阶段 brain 不再生效后启用）
```

## 本章小结

| 要点 | 一句话 |
|------|--------|
| Skills 不免费 | 每个 Skill 都在推高输入的 Token 成本，从 0 到 25 个约增加 3 倍 |
| 8-15 个是甜点区间 | 采纳率 65-75%，成本可控，边际收益尚未快速衰减 |
| 按场景分组 > 全量挂载 | 90% 的日常任务只需 3-6 个 Skills |
| 条件触发是精髓 | 精确的路径 + 内容 + 任务触发，让 Skills 在该出现的时候才出现 |
| 定期裁剪和度量 | 未经审视的 Skills 配置就是技术债，月度复盘不可少 |

> **核心原则**：Skills 配置的本质是"用 Token 买质量"。你的工作不是最小化 Token 消耗，而是在 Token 消耗和代码质量之间找到斜率最大的那个点。这个点，就是你的最优配置。
