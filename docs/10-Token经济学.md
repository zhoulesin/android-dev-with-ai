# Skills 越多越好？我测了 5 种配置，Token 消耗差了 5 倍

上个月收到 API 账单，$317。我瞪大了眼睛又看了一遍，确实是三百一十七美元，不是三十一。而坐在我旁边的同事，同做一个 Android 项目、同样用 Claude Sonnet，账单 $52。产出质量呢？Code review 通过率几乎一样。

差别的源头极其反直觉：**Skills 数量**。我挂了 25 个，他挂了 6 个。

这不是一个"省钱"的故事——省 $265 不值得花一篇文章讲。真正让我决定做这件事的原因，是当我把自己砍到 8 个 Skills 之后，代码质量反而**变好了**。我花更多的钱，得到更差的输出。这是任何一个工程师都不该接受的局面。

于是有了这场测试。同一项目、同一任务、同一模型，只改一个变量：Skills 数量。

---

## 实验设计：5 种配置，公平对决

测试环境统一：

- **工具**：Claude Code CLI v2.x
- **模型**：Claude Sonnet（~200K 上下文窗口）
- **项目**：中型 Android 项目，~350 个 Kotlin 源文件，20 个 Gradle 模块
- **任务**：实现完整的用户登录功能——UI（Compose）+ ViewModel + Repository + 单元测试
- **Prompt**：每个配置用完全相同的 prompt 模板，只改变 Skills 加载量
- **重复次数**：每种配置独立开 3 轮新会话，数据取均值

我设了 5 档配置：

| 配置 | Skills 数 | 说明 |
|------|----------|------|
| 裸用 | 0 | 不挂任何 Skill，AI 纯凭基座模型知识 |
| 最小配置 | 3 | android-dev + compose + kotlin-style |
| 标准配置 | 8 | 最小配置 + brainstorming + writing-plans + tdd + architecture-guide + gradle |
| 发烧友配置 | 20 | 标准 + 全套 Superpowers 流程链 + 安全审查 + 日志分析 + 自定义 Rules × 6 |
| 极客配置 | 40 | 发烧友 + 所有能找到的社区 Skills，来者不拒 |

## 核心数据：Token 和成本的真实差距

| 配置 | Skills | 输入 Token | 输出 Token | 总 Token | 单次成本 | 相对增长 |
|------|--------|-----------|-----------|---------|---------|---------|
| 裸用 | 0 | 8,500 | 3,200 | 11,700 | $0.07 | 1.0× |
| 最小配置 | 3 | 12,400 | 3,800 | 16,200 | $0.09 | 1.3× |
| 标准配置 | 8 | 18,600 | 4,100 | 22,700 | $0.12 | 1.7× |
| 发烧友配置 | 20 | 31,500 | 4,250 | 35,750 | $0.16 | 2.3× |
| 极客配置 | 40 | 49,200 | 4,300 | 53,500 | $0.22 | 3.1× |

> 成本按 Sonnet 级别参考价计算：输入 $3.00/M，输出 $15.00/M。实际价格以平台为准。

一眼就能看出三个反常点：

**第一，输入 Token 暴涨，输出 Token 纹丝不动。** 40 个 Skills 的输入是裸用的 5.8 倍，但输出只多了 34%。AI 生成的代码量几乎一样——每个 Skill 的文本内容都被注入 System Prompt 抬高"起步价"，但任务没变，最终产物没变。你在为同一顿饭支付越来越贵的"餐位费"。

**第二，代码质量并非一路走高。** 这才是最让人不安的发现。来看采纳率：

| 配置 | 采纳率 | 一稿可用率 | 无需修改直接提交 |
|------|--------|-----------|----------------|
| 裸用 | 45% | 28% | 10% |
| 最小配置 | 62% | 45% | 24% |
| 标准配置 | 72% | 58% | 35% |
| 发烧友配置 | 75% | 60% | 36% |
| 极客配置 | 68% | 47% | 18% |

极客配置的采纳率，**还不如标准配置**。40 个 Skills 比 8 个产生了更差的代码。为什么？因为 AI 被太多互相矛盾或重叠的指令淹没，开始"困惑"。当一个 Skill 说"优先写测试"、另一个说"先设计方案"、第三个说"先画架构图"时，AI 不会选择，而是来回摇摆——体现在代码里就是风格不统一、逻辑跳跃、过度设计。

**第三，技能类型比技能数量更重要。** 同样是 8 个 Skills，如果 8 个全是编码规范类（Kotlin Style + Compose + Gradle + Room + Navigation + Hilt + Security + Testing），采纳率达 75%；如果 8 个全是流程管理类（brainstorming + writing-plans + executing-plans + subagent-driven-dev + tdd + debugging + code-review × 2），采纳率仅 62%。规范类 Skills 直接影响代码生成；流程类 Skills 控制的是"怎么干活"，对单次产出的影响力更间接。

## ROI 曲线：钱花在哪才有用？

把采纳率和成本放到一张图上，曲线长这样：

```
采纳率 (%)
  ↑
75│               ╭─── · 标准 (8个) ROI最高
70│          ╭────╯
65│     ╭────╯
60│╭───╯
55│
50│╯
  └──────────────────────────────────────→ Token 成本 ($)
   0.07   0.09   0.12   0.16   0.22

   裸用   3个    8个    20个   40个
```

- **0→3 个**：每增加一个 Skill，采纳率提升 ~5%，成本增加几乎可忽略。这是"地基区"。
- **3→8 个**：每增加一个 Skill，采纳率提升 ~2-3%。流程管理和专项规范交替引入，收益仍然可观。
- **8→15 个**：每增加一个 Skill，采纳率提升 ~1%。进入"精装修区"——比如从 Compose 通用规范细分到 Compose Animation 专项规则。
- **15→25 个**：每增加一个 Skill，采纳率提升不足 0.5%。大量约束重叠甚至冲突，信息噪声开始抵消收益。
- **25→40 个**：采纳率**下跌**。上下文过载，AI 无法有效聚焦，关键约束被淹没在噪声中。

成本端却是另一条线——几乎完全线性。每个 Skill 文件独立注入，系统不做深度去重，40 个 Skills 的 Token 成本刚好是 8 个的 1.8 倍，但产出质量还低了 4 个百分点。

用钱的角度算笔账。以标准配置为例，单次会话 $0.12，假设每天 20 次会话：

```
8 个 Skills 月成本：$0.12 × 20 × 30 = $72
采纳率：72%

40 个 Skills 月成本：$0.22 × 20 × 30 = $132
采纳率：68%

多花 $60/月，产出更差。
```

更精明地算——按需加载的话，90% 的日常任务只需 3-6 个相关 Skills，月成本可控制在 $40 以内，代码质量保持不变。

## 我现在怎么配置？

经过这次测试，我砍掉了 17 个 Skills，留下 9 个。砍的不一定是"坏的"，而是"值得占 2000 个 Token 上下文吗"。清单如下：

**保留的 9 个：**

| Skill | 类型 | 理由 |
|-------|------|------|
| android-dev | 全局地基 | 技术栈和架构约束，每次会话必用 |
| compose | 规范类 | Compose UI 规范，直接决定 UI 代码质量 |
| kotlin-style | 规范类 | 编码风格一致性，最直接的采纳率提升 |
| gradle | 规范类 | 构建系统约束，防止依赖混乱 |
| architecture-guide | 规范类 | Clean Architecture 分层约束 |
| tdd | 流程类 | 关键功能开发时启用 |
| brainstorming | 流程类 | 需求不明确或方案设计时启用 |
| requesting-code-review | 流程类 | 代码审查阶段启用 |
| systematic-debugging | 流程类 | 仅排障时启用 |

**砍掉的典型例子：**

- **security-check**：逻辑本身有价值，但 700 行的 Skill 文件吃掉了 3500 Token——每次会话的 20% 预算被它独占。改为在 `android-dev` 里加两行关键安全约束：禁止硬编码密钥、禁止日志打印敏感信息。代价从每次 $0.01 降到几乎为零。
- **subagent-driven-dev**：流程定义精美，但只有大型功能才需要。日常使用中它和 brainstorming + tdd 的流程冲突，导致 AI 频繁问"该用哪个流程"。独立存档，需要时再激活。
- **自定义 Rules × 6**：写 Skill 会上瘾。给 Room 写一个、给 Hilt 写一个、给 Navigation 写一个……最后发现这些约束在 `android-dev` 和 `architecture-guide` 里已经覆盖了 80%。合并、缩短、删掉。

**场景分组：该用的时候才用**

```
日常开发（4 个）：android-dev + compose + kotlin-style + gradle
设计方案（5 个）：日常开发 + brainstorming
编码实现（6 个）：设计方案 + tdd
代码审查（4 个）：android-dev + kotlin-style + requesting-code-review
Bug 修复（3 个）：android-dev + systematic-debugging + 相关技术栈 Skill
```

不是我发明的——这是实测出来的：80% 的会话场景只需 4-6 个 Skills。剩下的 20% 场景才值得挂 8-10 个。全量常驻 25 个，本质上是在为一个不太可能出现的场景持续付费。

## 省 Token 的 4 个实操技巧

除了控制 Skills 数量，还有几个立竿见影的手段：

**1. 用便宜模型干粗活**

不是每次调用都需要 Sonnet。我的实际用法：

| 任务类型 | 模型选择 | 说明 |
|---------|---------|------|
| 生成测试数据、写文档字符串 | Haiku | 输出量大有格式要求，质量敏感度低 |
| 日常编码、Code Review | Sonnet | 主要工作流 |
| 架构设计、复杂重构 | Opus | 仅在需要最高质量推理时启用 |

一套组合拳下来，月成本比全用 Sonnet 低约 35%，关键产出质量不受影响。

**2. 定期裁剪上下文**

长会话是 Token 杀手。当 AI 和你的对话超过 50 轮，上下文中的历史信息量已经远超 Skills 的消耗量。一个原则：

> 如果你开始手动翻页找之前的代码，而 AI 还在引用第 3 轮的对话——结束当前会话，开新的。

实测数据：会话超过 60 轮后，后续每次请求的输入 Token 约 40% 花在历史对话上。新开会话可以节省这部分成本，而且 AI 的回复质量更高（不被陈旧上下文干扰）。

**3. 别复制粘贴**

多个 Skill 文件里有相同的 Kotlin 风格约束？删除重复。工具侧的去重能力远比你想象中差——实测两个 Skill 各有一段 200 Token 的相同文本，最终被计费的是 400 Token，不是 200。

一个自查方法：把你所有的 Skill 文件合并成一个文本，用 `grep` 搜高频短语。重复 3 次以上的，删到只剩一份，放到最高优先级的 Skill 里。

**4. 1 分钟法则**

这是我给自己定的硬规则：

> 如果你花在管理和调优 Skills 上的时间，超过了这些 Skills 帮你节省的时间，就砍。

听起来简单，但极其难遵守。FOMO 心理会让你觉得"万一需要呢"——相信我，你不会需要那个"Compose 无障碍规范（iOS 移植版）"的 Skill。如果它连续两周没被触发，删掉。需要时再写回来，写一个 Skill 往往比维护一个从不用的 Skill 三个月更省时间。

## 总结：不是最小化 Token，是最大化 ROI

做这次测试之前，我以为结论会是"少挂 Skills 省 Token"。但真正的结论是：

**Skills 配置本质是用 Token 买质量。问题不是 Token 消耗了多少，而是每多花一个 Token，换回了多少代码质量的提升。**

- 0→8 个 Skills：ROI **极高**。采纳率从 45% 跳到 72%，单次成本从 $0.07 涨到 $0.12。每多花 $0.05，减少约 15 分钟的人工修改时间——按 $50/h 时薪算，净赚 $12.45。
- 8→20 个 Skills：ROI 勉强打平。采纳率只多 3 个百分点，成本翻近一倍。
- 20→40 个 Skills：ROI 为**负**。多花钱买更差的代码，这是一个信号，不是例外。

我的建议：从 8 个核心 Skills 开始。每两周测一次采纳率，如果新加一个 Skill 后采纳率提升超过 2%，保留；低于 2%，砍掉。这是工程判断，不是信仰选择。
