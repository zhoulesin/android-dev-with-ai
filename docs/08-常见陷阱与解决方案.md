# AI 写 Android 代码的 10 个坑，我都替你踩过了

用 AI 写了一年 Android 代码，省了不少时间，但也踩了不少坑。这些坑，每个都让我 debug 了半天以上。

最冤的是，问题往往不是 AI 不行，而是**你没说清楚**。AI 就像一个才华横溢但完全不了解你项目的新同事——你不告诉它代码规范、模块依赖、历史决策，它当然只能瞎猜。

下面这 10 个坑，每一个都有血有泪的故事，以及我最终找到的解法。

---

## 坑一：迭代开发时，AI 把上一个版本的功能给"优化"掉了

那是一个平平无奇的周三下午。产品说 V1.2 要给用户中心加个"注销账号"功能，我想这还不简单，让 AI 改一下 `UserRepository` 就行。

代码生成出来，看着挺漂亮。编译通过，运行——崩溃。

排查了半小时，我才发现 AI 重写了整个 `UserRepository`，把 V1.1 我花了三天写的本地缓存逻辑删得一干二净。它还"好心"地引入了一个 V1.0 就废弃掉的旧 API，并重新发明了一个已经在 `core:common` 模块里躺了两周的 `StringValidator`。

问题在哪？AI 没有"项目记忆"。它把每次对话当成从零开始的新项目。你给它一个文件让它改，它不知道这个文件之前经历过什么。

**解法一：Git diff 注入。** 在开始新版本开发前，先把当前分支相对于基准分支的变更喂给 AI：

```bash
git diff main..HEAD -- ':!*.lock' ':!build/' ':!.gradle/' > /tmp/v1.2-changes.diff
```

筛选出当前任务相关的模块 diff（控制在 500 行以内），贴进对话里。AI 看完之后就知道哪些代码是"历史遗产"，不敢乱动。

**解法二：模块锁定法。** 直接在 prompt 里划红线：

```
本次任务约束：
❌ 不得修改 feature:payment 模块中的任何文件
❌ 不得修改 UserRepository.kt 第 45-120 行的缓存逻辑
✅ 只允许在 feature:profile 目录下新增代码
```

**解法三：维护 CHANGES.md。** 按版本记录关键变更决策，每次对话自动注入。比如：

```markdown
## V1.1 关键变更
- UserRepository: 新增本地缓存层（Room），不可删除
- 废弃 fetchUserLegacy()，改用 fetchUserV2()
```

说白了，你得给 AI 建立一个"版本记忆"。你作为人类开发者不会删掉你不理解的代码（至少不应该），AI 也需要同样的上下文。

## 坑二：Figma 生成 Compose，间距永远差 4dp

你有没有这种感觉——Figma 设计稿上明明标的 `16.dp`，AI 生成的代码永远写 `12.dp`？颜色明明是 `#FF6B35`，它非要写 `Color(0xFFFF6B35)` 而不是项目里已经定义好的 `AppTheme.colors.primary`？字体用得随心所欲，`PingFang SC Medium 15sp` 被它自动替换成了 `MaterialTheme.typography.bodyLarge`。

我第一次让 AI 从 Figma 生成一个设置页的时候，回来一看代码，感觉像设计师画了 Porsche，AI 给我交了一辆夏利——大框架是对的，但每个细节都差了那么一点。

问题在于，AI 是"像素级"解析 Figma 的，但它不知道你的设计系统。它不知道你项目里已经定义了 `PrimaryOrange` 这个 token，也不知道卡片圆角全局是 `12.dp`。

**解法：分层生成法。** 这是我试下来效果最好的方案。把 UI 生成拆成三步：

第一步——**结构层**，只生成骨架和布局，颜色全用 `Color.Gray` 占位。

第二步——**样式层**，注入 Design Token 后着色。在 prompt 里附上一份完整的 token 定义：

```kotlin
object DesignTokens {
    object Colors {
        val Primary = Color(0xFFFF6B35)
        val Background = Color(0xFFF8F9FA)
        val TextPrimary = Color(0xFF1A1A2E)
    }
    object Spacing {
        val xs = 4.dp; val sm = 8.dp; val md = 12.dp
        val lg = 16.dp; val xl = 20.dp; val xxl = 24.dp
    }
    object Radius {
        val Card = 12.dp; val Button = 8.dp
    }
}
```

第三步——**细节层**，微调阴影、分割线、滚动行为、空状态。

三步走完之后，还原度能从 60% 拉到 90% 以上。剩下的 10% 比如深色模式适配、横竖屏切换、RTL 布局这些，还是得人工检查——AI 在这些维度上目前确实做不到位。

## 坑三：上下文溢出，AI 开始"说胡话"

我们项目有 50 多个模块，200+ Kotlin 文件。有一次我正在修一个登录模块的 Token 刷新逻辑，聊到第 20 轮对话的时候，AI 突然生成了一段完全无关的代码，里面引用了一个不存在的类 `UserTokenManager`。我一看，这名字有点像我们项目里 `TokenRepository` 的功能，但它是 AI 自己"发明"的。

上下文窗口是有上限的。主流模型在 128K 到 200K Token 之间，而一个中型 Android 项目的全量代码轻松超过 500K Token。当你无筛选地把所有文件塞进去，对话早期的上下文就被"遗忘"了——跟 LRU 缓存淘汰一个道理。

**解法一：模块化 Rules。** 不要在项目根目录搞一个巨大的 Rules 文件。按模块拆分：

```
project-root/
├── .cursorrules          # 通用规则：代码风格、命名规范
├── feature/
│   ├── login/
│   │   └── .cursorrules  # 登录模块规则
│   └── profile/
│       └── .cursorrules  # 个人中心规则
├── core/
│   ├── network/
│   │   └── .cursorrules  # 网络层规则
│   └── database/
│       └── .cursorrules  # 数据库规则
```

每个模块的 Rules 控制在 50-100 行，只写本模块的核心约束和依赖关系。

**解法二：按需加载。** 每次对话只注入当前任务相关的文件，不要让 AI 了解整个项目。比如修 Login 模块的问题，只给 `LoginViewModel.kt` 和 `TokenInterceptor.kt`，其他的等 AI 问你要再说。

**解法三：对话 checkpoint。** 当对话超过 15 轮，让 AI 自己压缩历史：

```
请将之前的对话压缩为一份结构化摘要，保留：
1. 已确认的技术决策
2. 已修改的文件及改动内容
3. 当前待解决的问题
```

然后用这份摘要开新对话。

核心思路就一句话：给 AI 刚好够用的上下文，而不是全量塞进去。就像你不会把公司的全部代码打印出来给一个新同事看一样。

## 坑四：Gradle 依赖地狱 × AI = 灾难加倍

Gradle 依赖版本冲突本来就是 Android 开发的老大难。加上 AI 之后，问题翻倍。

有一次 AI 给我生成了一个依赖声明：

```kotlin
implementation("com.squareup.retrofit2:retrofit:2.11.0")
```

看起来很合理对吧？但这个版本号是 AI 编的，Retrofit 根本没有 `2.11.0` 这个版本。Gradle sync 直接红了。还有一次 AI 在 `feature:login` 模块里想引入 `core:network` 的类，结果它写的依赖是 `implementation(project(":core:network:retrofit"))`——这个子模块根本不存在。

更致命的是循环依赖。AI 不了解你的模块依赖图是个 DAG，偶尔会搞出 `core:common → feature:login → core:common` 这种死循环。

**解法：版本目录注入 + 依赖图描述。** 在 Rules 中维护一份依赖图：

```yaml
dependency_graph:
  core:common:
    depends_on: []
  core:network:
    depends_on: [core:common]
  feature:login:
    depends_on: [core:network, core:database, core:common]
```

同时在 prompt 里明确告诉 AI：所有第三方库版本号请使用 `libs.versions.toml` 中的声明，不要自己编版本号。

还有一个好习惯：让 AI 生成代码后自己验证 import 路径是否真实存在、是否存在循环依赖。AI 的自我检查能力比你想象的好用。

## 坑五：AI 生成的代码风格像三个不同的人写的

同一个项目里，AI 上午写了个 `fun GetUserProfile()`（大写开头），下午写了个 `val UserName = ""`（应该是 `userName`），晚上又搞了个混搭架构——`StateFlow` 和 `LiveData` 齐飞，`Channel` 和 `SharedFlow` 共舞。

ktlint 能检查出缩进和空行的问题，但它管不了命名、架构选择、注释规范这些东西。

**解法：用"反面教材"教 AI。** 我发现 AI 学 Bad Case 比学正面规范快得多。在 Rules 里直接给出对比：

```kotlin
// ===== BAD CASE - 不要这样写 =====
class LoginService {           // ❌ 不用 Service 后缀
    fun Login() {}             // ❌ 函数名不用大写开头
    val data = LiveData<User>() // ❌ 不用 LiveData
}

// ===== GOOD CASE - 应该这样写 =====
class LoginRepository @Inject constructor(
    private val api: LoginApi
) {
    fun login(credentials: Credentials): Flow<Result<User>> =
        flow { emit(api.login(credentials)) }
}
```

再加上 pre-commit hook 做最后一道防线：

```bash
#!/bin/bash
./gradlew ktlintCheck detekt
if [ $? -ne 0 ]; then
    echo "❌ 代码风格检查未通过"
    exit 1
fi
```

两条腿走路：Rules 管"怎么教"，hook 管"怎么拦"。

## 坑六："幻觉"式重构——AI 自信地删了一段关键逻辑

我的支付模块里有一段代码，注释写着 `// Workaround for Alipay SDK crash on Android 12`，里面是一段看起来非常诡异的 try-catch 加反射调用。AI 看了一眼，觉得这是"死代码"，自信地把它删了，还贴心地附上注释：`// Removed unused workaround`。

结果第二天 QA 在 Android 12 设备上复现了支付崩溃。

这是 AI 最危险的行为模式：它在不完全理解代码意图的情况下，用"看起来更优雅"的方式重写。它不知道那段"丑陋"的代码是某次线上事故后用血泪换来的。

**解法：关键代码加"不要删"注释。** 听起来很蠢，但真的有用：

```kotlin
// CRITICAL: DO NOT REMOVE - 支付宝 SDK 在 Android 12 上反射崩溃的 workaround
// 详见 Jira BUG-2847，移除会导致支付功能在 Android 12 设备 100% 崩溃
try {
    // ...
}
```

AI 对这类强语气注释还是比较敏感的。

**另一条防线：commit 级别的 review。** 不要信任任何 AI 生成的代码，逐 commit 审查。尤其是发现 AI 删了代码的时候，一定要问自己：它是真的理解这段逻辑才删的，还是单纯觉得"不够优雅"？

## 坑七：测试用例只覆盖 happy path

AI 写测试有个特点：又快又好——如果你只看 happy path 的话。

它会给你的 `login()` 方法生成一个测试，传入正确的用户名密码，断言返回 `Success`。测试全绿，你很满意。

然后用户输入了一个空密码——崩溃。再试一个超长的邮箱地址——崩溃。网络超时——没处理。API 返回了一个你没见过的错误码——当然也没处理。

**解法：在 Rules 里显式列出边界条件要求：**

```
测试覆盖要求（生成任何单元测试时必须遵守）：
✅ 正常输入 - 至少 1 个 case
✅ null 输入 - 每个可空参数
✅ 空集合输入 - 列表/字符串
✅ 网络异常 - IOException / HttpException
✅ 超时 - TimeoutException
✅ 服务端错误 - HTTP 500 / 业务错误码
✅ 边界值 - 最大/最小长度字符串、极限数值
```

一开始我觉得这样写太啰嗦，后来发现不写的话，AI 真的就只给你 3 个 happy path case 然后觉得自己大功告成了。

## 坑八：中文注释和英文代码混在一起

AI 有时候会在一个函数里上一行写 `// 获取用户信息`，下一行写 `// fetch user profile`。变量名也不统一，一会儿 `userName`，一会儿 `yongHuMing`。

这对于我这种习惯全英文代码的人来说，阅读体验极其割裂。

**解法：写死语言规则。** 在 Rules 里加一条：

```
语言约定：
- 所有标识符（类名、函数名、变量名）使用英文
- 所有代码注释使用中文
- 所有 TODO 使用中文，格式：// TODO: 描述
- 所有的字符串常量放在 strings.xml，使用 stringResource() 引用
```

这条规则看似简单，但不写的话，AI 会在中英文之间反复横跳，尤其是当你用中文提问的时候，它很容易把中文概念直接带进代码。

## 坑九：Compose Preview 总是不更新

AI 生成的 Composable 函数都挺漂亮，但有一个问题：它几乎从来不加 `@Preview` 注解，或者加了也不加 `@PreviewLightDark`、`@PreviewFontScale` 这些变体。

这意味着你在 Android Studio 的 Preview 面板里什么都看不到，每次都得跑模拟器才能看到效果。来回折腾几次，半天就没了。

**解法：Compose Rules 强制要求 Preview：**

```yaml
compose_rules:
  preview:
    - 每个 @Composable 函数必须附带 @Preview 注解
    - 必须包含 @PreviewLightDark（深色/浅色双模式）
    - 必须包含 @PreviewFontScale（大字体模式）
    - Preview 中使用真实数据样本，不允许全用 "placeholder" 占位
    - 多个 Preview 用 @Preview 注解叠加，不要写多个 Preview composable
```

配上示例代码让 AI 照葫芦画瓢：

```kotlin
@Preview(showBackground = true)
@Preview(uiMode = UI_MODE_NIGHT_YES, showBackground = true)
@Preview(fontScale = 1.5f)
@Composable
private fun UserCardPreview() {
    AppTheme {
        UserCard(
            user = User.sample(),
            onClick = {}
        )
    }
}
```

## 坑十：AI 生成硬编码密钥

这是我踩过最危险的一个坑。有一次我让 AI 帮我写一个调七牛云上传的 demo，它非常"贴心"地把 AccessKey 和 SecretKey 直接写在了代码里：

```kotlin
val accessKey = "your-access-key-here"  // 还好是占位符
```

但下一次，我自己的密钥居然出现在了生成的代码里——因为之前的对话里有。虽然那只是一个测试环境的密钥，但这足够让我后背发凉。

更隐蔽的问题是，AI 喜欢把一些"看起来不像密钥但其实是的"东西直接写死在代码里，比如内网 API 地址、第三方 SDK 的 AppID。

**解法：安全 Rules + pre-commit 扫描。** 在 Rules 中明确约定：

```yaml
security_rules:
  - 禁止在任何代码文件中硬编码密钥、Token、密码
  - 所有敏感信息通过 BuildConfig / local.properties / 环境变量 注入
  - API 地址等环境相关配置放进 build.gradle 的 buildTypes
  - 发现疑似密钥的字符串，立即用占位符替换并加 `// TODO: 替换为真实密钥`
```

再加上 pre-commit 的 secret scanning：

```bash
# 使用 git-secrets 或 detect-secrets 扫描
pip install detect-secrets
detect-secrets scan --all-files
```

AI 只管生成，你得管安全。这条底线不能交给任何人——包括 AI。

## 速查表

| 序号 | 问题 | 典型症状 | 核心解法 |
|------|------|---------|---------|
| 1 | 迭代破坏已有功能 | AI 重写模块时删除已有逻辑 | Git diff 注入 + 模块锁定 + CHANGES.md |
| 2 | Figma 还原度低 | 间距差 4dp、颜色用裸值、字体乱选 | 分层生成 + Design Token 注入 |
| 3 | 上下文溢出 | AI 中途"失忆"、发明不存在的类 | 模块化 Rules + 按需加载 + 对话压缩 |
| 4 | Gradle 依赖错误 | 版本号不存在、import 路径错误、循环依赖 | 版本目录注入 + 依赖图描述 + 自我验证 |
| 5 | 代码风格混乱 | 命名、架构、注释风格不统一 | Bad Case 示例 + ktlint/detekt pre-commit |
| 6 | 幻觉式重构 | AI 删除关键 workaround 或业务逻辑 | "DO NOT REMOVE" 注释 + commit 级 review |
| 7 | 测试只覆盖 happy path | 空值、异常、边界条件全未处理 | Rules 列出边界条件清单 |
| 8 | 中英文混用 | 注释中英混杂、变量名拼音 | 语言规则明文约定 |
| 9 | Compose Preview 不更新 | Composable 缺少 @Preview 注解 | Compose Rules 强制 Preview 规范 |
| 10 | 硬编码密钥 | API Key、Token 直接写在代码里 | 安全 Rules + pre-commit secret scanning |

---

写到这里你会发现一个规律：**80% 的坑不是因为 AI 能力不够，而是因为你没把规则说清楚。** 

每次踩坑之后，我的第一反应不是怪 AI，而是打开 Rules 文件，把这次的教训加进去。慢慢地，AI 越来越像"你的人"，而不是一个随机代码生成器。

说到底，用 AI 写代码不是魔法，而是一种新的工程实践——你需要学会管理它的上下文、约束它的输出、审查它的结果。上面这 10 个坑我都替你踩过了，希望你能少 debug 几个下午。
