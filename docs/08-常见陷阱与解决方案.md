# 第 8 章 · 常见陷阱与解决方案

> **本节目标**：识别 AI 辅助 Android 开发中 6 大类高频陷阱，掌握其根因分析与可落地解决方案。
>
> **前置知识**：熟悉 Kotlin/Jetpack Compose/Gradle 多模块开发，并已阅读第 7 章的全流程集成方法。

---

AI 工具能显著提升开发效率，但也容易在生产代码中埋下难以察觉的问题。本章逐一剖析陷阱根因并提供可复现的解决方案，每节附动手实践便于你直接验证效果。

---

## 8.1 迭代开发：AI 不理解历史版本差异

### 问题表现

最典型的场景：V1.0 已上线，你正在开发 V1.1，AI 生成了一个"看起来正确"的 `UserRepository`，但它：

- 覆盖了 V1.0 新增的缓存策略逻辑
- 引入了一个已在 V1.0 被废弃的 API 调用
- 重新实现了一个已经存在于 `core:common` 模块中的工具方法

本质上是 AI 把每一次迭代当作**从零开始的项目**，而不是在既有代码之上演进。

### 根因分析

AI 模型没有"项目记忆"。它每次对话只能基于你当前提供的上下文来推理。当你只说"帮我改 UserRepository"，而没有告诉它 V1.0 中这个文件经历了什么变更，它自然无法理解历史差异。

### 解决方案

#### 方案一：Git diff 注入法（推荐，最直接）

在发起新版本开发任务前，先让 AI 看完当前分支相对于基准分支的 diff：

```bash
# 生成 V1.0 到当前开发的差异
git diff main..HEAD -- ':!*.lock' ':!build/' ':!.gradle/' > /tmp/v1.1-changes.diff
```

然后在对话中：

```
以下是我们 V1.1 迭代相对 V1.0 的全部代码变更（diff 文件），请先阅读变更摘要，
然后针对「用户模块重构」这个任务生成代码，确保不破坏已有逻辑：

[粘贴 /tmp/v1.1-changes.diff 的核心内容]
```

**要点**：diff 文件可能很大，需要先筛选关键文件的变更。建议只注入当前任务相关的模块 diff，控制在 500 行以内。

#### 方案二：Context 文件法

在项目根目录维护一个 `CHANGES.md`，按版本记录关键变更决策：

```markdown
## V1.1 关键变更

### UserRepository（变更）
- V1.0 新增了本地缓存层（Room），不可删除
- V1.0 废弃了 fetchUserLegacy()，改用 fetchUserV2()
- V1.1 新增了 fetchUserProfile() 方法，与 fetchUserV2() 并行

### 不可修改模块
- core:network - HTTP 客户端升级 OkHttp 5.x，不稳定，禁止改动
- feature:payment - 支付模块接入了第三方 SDK，禁止动签名
```

每次调用 AI 时，用 `project_rules` 或 `.cursorrules` 自动注入此文件：

```yaml
# .cursorrules 或 AGENTS.md
context_files:
  - CHANGES.md
```

#### 方案三：模块锁定法

直接在 prompt 中划定"禁区"：

```
本次任务约束：
❌ 不得修改 feature:payment 模块中的任何文件
❌ 不得修改 UserRepository.kt 中的缓存相关代码（第 45-120 行）
✅ 只允许在 feature:profile 目录下新增代码
```

### 动手实践

模拟场景：你有一个 V1.0 的 `Calculator.kt`：

```kotlin
// V1.0 版本 - Calculator.kt
object Calculator {
    // V1.0 新增的缓存，不可删除
    private val cache = mutableMapOf<String, Int>()

    fun add(a: Int, b: Int): Int {
        val key = "$a+$b"
        return cache.getOrPut(key) { a + b }
    }

    fun subtract(a: Int, b: Int): Int {
        val key = "$a-$b"
        return cache.getOrPut(key) { a - b }
    }
}
```

V1.1 需求：新增 `multiply()` 和 `divide()` 方法。

**实践步骤**：

1. 先用"无上下文"方式让 AI 生成代码，观察它是否重写了整个 `Calculator` 对象而丢了缓存逻辑
2. 再用 Git diff 注入法，告诉 AI："Calculator.kt 第 3-8 行的缓存机制不能改动，只新增 multiply/divide"
3. 对比两次输出，你会发现方案 2 的代码只新增方法，不动缓存

## 8.2 Figma 生成页面还原度低

### 问题表现

AI 从 Figma 链接或截图生成 Compose UI 代码时，常见的还原度问题：

| 维度 | 典型偏差 |
|------|---------|
| 间距 | Figma 标注 16.dp，AI 输出 12.dp 或 `Modifier.padding(horizontal = 16.dp)` 写成 `start = 16.dp` |
| 颜色 | `#FF6B35` → `Color(0xFFFF6B35)` 而不是项目已有的 `AppTheme.colors.primary` |
| 字体 | 用了 `MaterialTheme.typography.bodyLarge` 而不是设计稿指定的 `PingFang SC Medium 15sp` |
| 复杂布局 | Figma 的 Auto Layout + Constraints 被 AI 错误映射为 `Box` + 硬编码定位 |

### 根因分析

AI 对 Figma 设计稿的解析是"像素级"的，但缺乏对设计系统的结构化理解。它不知道项目中已经定义了 `PrimaryOrange` 这个 token，也不知道 `Card` 组件的圆角是全局 12.dp。

### 解决方案

#### 方案一：分层生成法（推荐）

将 UI 生成拆成三步，每步只关注一个维度：

**Step 1 - 结构层**：只生成骨架，用占位色

```
基于这个 Figma 设计稿，先生成 Compose 的布局结构骨架。
使用文字占位符，颜色全用 Color.Gray。
专注：组件层级、间距关系、对齐方式。
```

**Step 2 - 样式层**：注入 Design Token 后着色

```
现在基于项目 Design Token 着色：
- 主色：AppTheme.colors.primary (#FF6B35)
- 背景：AppTheme.colors.background (#F5F5F5)
- 字体：BodyMedium(15sp) / CaptionSmall(12sp)
- 圆角：12.dp (card) / 8.dp (button)
- 间距标准：4-8-12-16-20-24 dp 尺度

将上一步的占位色替换为实际 token。
```

**Step 3 - 细节层**：微调边缘情况

```
最后修正：
- 按钮阴影参数
- 分割线样式
- 滚动行为
- 空状态/加载态
```

#### 方案二：Design Token 注入

在 Rules 文件中维护一份完整的 Design Token 定义：

```kotlin
// DesignTokens.kt - 注入 Rules 的设计常量
object DesignTokens {
    object Colors {
        val Primary = Color(0xFFFF6B35)
        val PrimaryVariant = Color(0xFFE55A2B)
        val Background = Color(0xFFF8F9FA)
        val Surface = Color(0xFFFFFFFF)
        val TextPrimary = Color(0xFF1A1A2E)
        val TextSecondary = Color(0xFF6C757D)
    }

    object Spacing {
        val xs = 4.dp
        val sm = 8.dp
        val md = 12.dp
        val lg = 16.dp
        val xl = 20.dp
        val xxl = 24.dp
    }

    object Typography {
        val Heading1 = TextStyle(fontSize = 24.sp, fontWeight = FontWeight.Bold)
        val Body = TextStyle(fontSize = 15.sp, fontWeight = FontWeight.Normal)
        val Caption = TextStyle(fontSize = 12.sp, fontWeight = FontWeight.Light)
    }

    object Radius {
        val Card = 12.dp
        val Button = 8.dp
        val Chip = 16.dp
    }
}
```

每次生成 UI 前，在 prompt 中附上这段 token 定义。这样 AI 输出的是 `DesignTokens.Colors.Primary` 而不是裸 `Color(0xFFFF6B35)`。

#### 方案三：人工微调清单

以下维度是 AI 做不好的，建议人工检查：

- [ ] 深色模式适配（AI 通常只生成 Light 主题）
- [ ] 横竖屏布局切换
- [ ] RTL（从右到左）语言布局
- [ ] 动画与过渡效果
- [ ] 可访问性（contentDescription，semantics）
- [ ] 大字体模式下的 UI 缩放

### 动手实践

取一个中等复杂度的页面（如设置页），用两种方式生成：

1. **一次性生成**：直接把 Figma 截图给 AI，生成完整 Compose 代码
2. **分层生成**：按上面三步法逐步生成

对比两组代码在以下维度的得分（1-5 分制）：

| 维度 | 一次性生成 | 分层生成 |
|------|-----------|---------|
| 间距准确度 | | |
| 颜色匹配度 | | |
| 组件可复用性 | | |
| 代码可维护性 | | |

## 8.3 大项目上下文溢出

### 问题表现

当项目超过 50 个模块、200+ 文件时，常见症状：

- AI 在对话中途"忘记"之前的约束条件
- 生成的代码只引用了部分 import，缺少关键依赖
- 幻觉加重：生成了项目中并不存在的类名或方法
- 输出突然截断（Token 限制）

### 根因分析

主流 AI 工具的上下文窗口在 128K-200K Token 之间。一个中型 Android 项目的全量代码轻松超过 500K Token。当你无筛选地把所有文件塞给 AI，它必然溢出——对话早期的内容被"遗忘"（类似 LRU 淘汰机制）。

### 解决方案

#### 方案一：模块化 Rules

不要用一个巨大的 `.cursorrules` 文件描述整个项目。按模块分层：

```
project-root/
├── .cursorrules              # 通用规则：代码风格、命名规范
├── app/
│   └── .cursorrules          # 主模块特有规则
├── feature/
│   ├── login/
│   │   └── .cursorrules      # 登录模块规则
│   └── profile/
│       └── .cursorrules      # 个人中心规则
├── core/
│   ├── network/
│   │   └── .cursorrules      # 网络层规则
│   └── database/
│       └── .cursorrules      # 数据库规则
```

每个 `.cursorrules` 只写本模块的核心约束，控制在 50-100 行：

```yaml
# feature/login/.cursorrules 示例
module: feature:login
dependencies:
  - core:network (HTTP 请求)
  - core:common (工具类)
patterns:
  - MVVM + Repository
  - StateFlow 状态管理
constraints:
  - 密码不得明文存储，必须使用 EncryptedSharedPreferences
  - 验证码倒计时逻辑已实现在 LoginViewModel 第 89-120 行，不可修改
key_files:
  - LoginViewModel.kt
  - LoginRepository.kt
  - LoginScreen.kt
```

#### 方案二：按需加载

不要在每次对话中一股脑把所有文件都打开。使用动态注入策略：

```
当前任务：修复 LoginViewModel 中的 Token 刷新逻辑

核心文件（必须注入）：
  - feature/login/src/.../LoginViewModel.kt
  - core/network/src/.../TokenInterceptor.kt

关联文件（AI 按需请求）：
  - 当 AI 需要了解 Retrofit 配置时，再注入 NetworkModule.kt
  - 当 AI 需要了解存储实现时，再注入 SecureStorage.kt
```

一个好的技巧是用脚本自动提取依赖：

```bash
#!/bin/bash
# prepare-context.sh - 根据任务自动收集相关文件
TASK=$1
# 从 git log 中找到最近修改的相关文件
git log --oneline -20 --name-only | grep -E "\.kt$" | sort -u | head -20
```

#### 方案三：摘要压缩

当对话已经很长时，让 AI 自己压缩：

```
请将我们之前的对话历史压缩为一份结构化摘要，保留：
1. 已确认的技术决策（3 条）
2. 已修改的文件及其改动内容（5 条）
3. 当前待解决的问题（2 条）
压缩后开始新对话，把摘要作为第一条消息。
```

### 动手实践

为一个假设的大型项目（50+ 模块）设计分层 Rules 结构。项目结构如下：

```
├── app/ (主模块)
├── core/
│   ├── network/
│   ├── database/
│   ├── common/
│   └── ui/
├── feature/
│   ├── login/
│   ├── home/
│   ├── search/
│   ├── cart/
│   ├── order/
│   └── profile/
└── lib/
    ├── payment-sdk/
    └── analytics/
```

为 `core:network`、`feature:login`、`feature:order` 各写一个 `.cursorrules` 模板，确保：
- 描述本模块依赖关系
- 列出 3-5 个核心约束
- 控制在 20 行以内

## 8.4 多模块依赖理解错误

### 问题表现

在 Gradle 多模块项目中，AI 经常出现：

```kotlin
// AI 生成的错误 import
import com.example.feature.login.data.LoginRepository  // ❌ 实际路径是 com.example.core.auth.data.AuthRepository

// AI 生成的错误依赖声明
implementation(project(":core:network:retrofit"))  // ❌ 不存在此模块，实际是 :core:network
```

更严重的是，AI 可能引入循环依赖：

```
:core:common → :feature:login → :core:common  // 💀 编译不过
```

### 根因分析

Gradle 多模块的依赖关系是一个有向无环图（DAG），但 AI 没有这张图的全貌。它只能通过文件名和包名猜测，而 Android 项目中 `module/` 和 `package name` 往往不一致。

### 解决方案

#### 方案一：依赖图注入

在 `.cursorrules` 或 `AGENTS.md` 中用文本描述依赖图：

```yaml
# 模块依赖关系图
dependency_graph:
  app:
    depends_on: [feature:*]
  core:network:
    depends_on: [core:common]
    provides:
      - RetrofitClient.kt
      - ApiService.kt
      - TokenInterceptor.kt
  core:database:
    depends_on: [core:common]
    provides:
      - AppDatabase.kt
      - dao/*
  core:common:
    depends_on: []  # 基础模块，无依赖
    provides:
      - util/*
      - extensions/*
  feature:login:
    depends_on: [core:network, core:database, core:common]
    provides:
      - LoginViewModel.kt
      - LoginScreen.kt
      - LoginRepository.kt
```

或者直接生成 ASCII 图：

```
app
 ├── feature:login ───┬── core:network ─── core:common
 ├── feature:home ─────┤
 ├── feature:order ────┼── core:database ── core:common
 └── feature:profile ──┘
```

#### 方案二：接口契约先行

在跨模块调用时，先让 AI 定义接口，再分别实现：

```
Step 1：在 core:common 中定义 AuthProvider 接口
Step 2：在 core:network 中实现 AuthProviderImpl
Step 3：在 feature:login 中依赖 core:common 的 AuthProvider 接口（不依赖实现）
```

这样 AI 对依赖的认知模式变为：

```
feature:login → core:common (interface) ← core:network (implementation)
```

而非错误的：

```
feature:login → core:network (直接依赖，可能循环)
```

#### 方案三：类型安全检验

生成代码后，让 AI 自己验证 import 路径：

```
请逐条检查你生成的 import 语句：
1. 该模块是否声明了对 import 路径所在模块的依赖（检查 build.gradle.kts）？
2. import 的类是否真实存在于目标模块中？
3. 是否存在循环依赖（A → B → A）？

列出所有不满足以上条件的 import，并修正。
```

### 动手实践

以你的实际项目为例，画出模块依赖图（参考上面的 YAML 格式），包含至少 5 个模块。然后用这份依赖图作为上下文，让 AI 实现一个跨模块功能（如从 `feature:order` 调用 `feature:login` 的获取用户信息接口），观察 AI 是否正确引用了模块路径。

## 8.5 代码风格不统一

### 问题表现

即使配置了 ktlint 和 detekt，AI 生成的代码仍然可能出现：

```kotlin
// 风格问题 1：命名混乱
fun GetUserProfile() {}          // ❌ 团队规范是动词开头小驼峰
val UserName = ""                // ❌ 应该是 userName

// 风格问题 2：架构不一致
class UserScreen {
    // ❌ 部分代码用 MVI，部分用 MVVM，混在一起
    val state = MutableStateFlow(UserState())
    val events = Channel<UserEvent>()  // ❌ Channel 已被 SharedFlow 替代
}

// 风格问题 3：注释风格各异
// 这是注释
/** 这也是注释 */
// TODO: fix this  ← 没有 Jira ID，团队规范要求 TODO(jira-1234)
```

### 根因分析

AI 的代码风格来自训练数据的统计分布，而非你的团队规范。`ktlint` 能检查出格式问题（如缩进、空行），但管不了命名、架构选择、注释规范。这些"表层规则"需要在 Rules 中明确约定。

### 解决方案

#### 方案一：风格 Rules 模板

在 `.cursorrules` 中注入具体的 Kotlin 编码规范，**不要只说"遵循 Android 规范"**，要用 Bad Case 示例：

```yaml
kotlin_style:
  naming:
    functions: "lowerCamelCase，动词开头（fetchUser, observeState）"
    classes: "UpperCamelCase，名词（LoginViewModel, UserCard）"
    constants: "UPPER_SNAKE_CASE，在 companion object 中定义"
  architecture:
    pattern: "MVVM + Repository + UseCase（全局统一）"
    state_management: "StateFlow，不使用 LiveData 和 Channel"
    di: "Hilt，构造函数注入"
  comments:
    todo_format: "// TODO(jira-ticket): 具体描述"
```

#### 方案二：Pre-commit Hook 检查

配置 ktlint + detekt 的 pre-commit hook，让 AI 生成的代码在提交前被自动检查：

```kotlin
// detekt 自定义规则示例 - 检测禁止的模式
// detekt-config.yml
style:
  ForbiddenMethodCall:
    active: true
    methods:
      - 'kotlinx.coroutines.channels.Channel'  # 禁止 Channel

naming:
  FunctionNaming:
    active: true
    ignoreAnnotated: ['Composable']
    functionPattern: '^[a-z][a-zA-Z0-9]*$'  # 强制小驼峰
```

```bash
# .git/hooks/pre-commit
#!/bin/bash
./gradlew ktlintCheck detekt

if [ $? -ne 0 ]; then
    echo "❌ 代码风格检查未通过，请在提交前修正"
    exit 1
fi
```

#### 方案三：Bad Case 示例（效果最好）

AI 对"反面教材"的学习比"正面规范"更有效。在 Rules 中加入 Bad Case：

```kotlin
// ===== BAD CASE =====
// ❌ 不要这样写
class LoginService {  // 命名：不用 Service 后缀，用 Repository
    fun Login() {}    // 命名：函数名不用大写开头
    val data = LiveData<User>()  // 架构：不用 LiveData
}

// ===== GOOD CASE =====
// ✅ 应该这样写
class LoginRepository @Inject constructor(
    private val api: LoginApi
) {
    fun login(credentials: Credentials): Flow<Result<User>> =
        flow { emit(api.login(credentials)) }
}
```

### 动手实践

为你的团队编写一份完整的 Kotlin 编码规范 Rules，要求包含：

1. 至少 5 条命名规则（含 Bad Case 示例）
2. 架构模式约定
3. 至少 2 个 Bad Case → Good Case 对照
4. 注释与 TODO 规范

将这份 Rules 以 `.cursorrules` 或 `AGENTS.md` 的格式写出。

## 8.6 其他常见陷阱速查表

| 问题 | 现象 | 快速解决 |
|------|------|---------|
| **ProGuard/R8 混淆错误** | 反射调用失效、Gson 反序列化崩溃 | 在 Rules 中注入 keep 规则：`-keep class com.example.model.** { *; }` |
| **Compose Preview 不更新** | AI 生成 Composable 但忽略 `@Preview` 注解 | Rules 中强制要求：每个 Composable 必须附带 `@Preview` |
| **测试用例质量低** | 仅覆盖 happy path，空值/异常未处理 | Rules 注入测试边界条件要求：必须包含 null 输入、网络异常、空列表 |
| **AI 重复生成已有代码** | 不确定项目是否已有类似实现就动手写 | 先让 AI grep 搜索：`请先搜索项目中是否已存在 UserAvatar 相关实现` |
| **Room 迁移脚本错误** | AI 直接改 Entity 字段却不生成 Migration | Rules 约定：Schema 变更必须同时提供 Migration + 测试 |
| **Hilt 模块配置遗漏** | 生成新类但没注册 DI Module | Rules 约定：每次新增 `@Inject constructor` 类后，提醒检查是否需要 `@Module` 注册 |
| **Coroutine 作用域用错** | `GlobalScope.launch` 满天飞 | Rules 注入：禁止使用 GlobalScope，必须用 viewModelScope 或 lifecycleScope |
| **资源硬编码** | AI 直接在代码中写 `"加载中..."` 而非引用 `R.string.loading` | Rules 注入：所有用户可见文字必须经过 `stringResource()` |

### 速查表使用建议

- 将高频问题加入 `.cursorrules` 的 `constraints` 字段，作为全局规则
- 每遇新坑更新速查表，团队共享，避免重复踩坑
- 将问题分为"编译期可检测"（如混淆、Hilt 注册）和"运行时才暴露"（如风格不统一），前者优先用自动化检查覆盖

## 本章小结

| 陷阱 | 核心策略 | 一句话 |
|------|---------|--------|
| 迭代开发丢历史 | Git diff 注入 + CHANGES.md | 让 AI 知道代码"从哪里来" |
| Figma 还原度低 | 分层生成 + Token 注入 | 结构、样式、细节分步攻关 |
| 上下文溢出 | 模块化 Rules + 按需加载 | 用多少给多少，而非全量塞入 |
| 多模块依赖错误 | 依赖图注入 + 接口契约先行 | 画一张图胜过千行代码 |
| 代码风格不统一 | Bad Case 示例 + Pre-commit Hook | 反面教材比正面规范更有效 |
| 其他高频陷阱 | 速查表 + Rules 约束 | 每次踩坑都是 Rules 的升级素材 |

> **核心原则**：AI 的缺陷不在模型能力，而在信息不对称。用 Rules、diff、依赖图、Token 定义这些"元信息"为 AI 填补认知空白，80% 的陷阱都可以预防。
