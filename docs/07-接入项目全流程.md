# 从 PRD 到上线：AI 参与 Android 开发全流程的真实复盘

> 时效性标注
> - 最后更新：2026-05
> - 统计周期：单次项目复盘（2026-04 ~ 2026-05）
> - 适用说明：本章为案例复盘，流程可复用；具体耗时与指标需以你的项目实测为准

上个月我们接了一个新需求——用户个人资料编辑页。放在以前，这就是一个普普通通的中等复杂度功能：表单校验、图片上传、状态同步，常规套路。但这次我决定把它当成一次实验：**让 AI 全流程参与，从产品需求解读到最终测试交付，每个环节都过一遍 AI**。我想看看，在真实项目中，AI 到底能帮多少忙，会犯什么错，以及最终的效率提升到底是不是空话。

以下是完整复盘。

---

## 零、前置配置：先让 AI 认识你的项目

在正式开始之前，我做了一件被很多人忽略的事——**给 AI 写了一份项目上下文文件**（可直接从 [`templates/rules/CLAUDE.md`](../../templates/rules/CLAUDE.md) 复制后按项目改）。我们用的工具支持 `CLAUDE.md` / `GEMINI.md` / `AGENTS.md` 自动加载。这份文件大概 100 行，包含了：

- 技术栈清单（Kotlin 1.9+，Compose + Material 3，Hilt，Retrofit，Room，DataStore，Coil 3）
- 架构规范（MVVM + Repository，sealed class 封装 UI State，StateFlow 暴露状态）
- 代码风格约束（kltint 规则，禁止硬编码字符串，Composable 命名规范）
- 模块依赖规则（feature 模块不得互相依赖，core 模块不得依赖 feature 模块）
- 常见代码模式（UI State sealed class 的标准写法示例）

这一步大概花了 20 分钟，但后来证明**这是整个流程中 ROI 最高的 20 分钟**。后面的每一次对话，AI 产出的代码都严格遵守了我们的架构规范，没有出现"AI 用 MVP 写了一个功能"这种需要推倒重来的惨案。

我的建议：不要吝啬配 Rules 文件的时间。每次踩坑后把解决方案沉淀进去，版本号精确到 minor，再加一个"禁止事项"清单——比如"禁止使用 SharedPreferences，统一用 DataStore"。随着项目推进，这份文件会越来越值钱。

## 一、产品阶段：AI 让 PM 的 PRD 变成我的技术方案

### 1.1 从 PRD 到技术要点，3 分钟

需求文档大概是这样的：

> 用户可以编辑头像（从相册选择或拍照）、昵称（2-20 字符）、性别（男/女/保密）、个人简介（最多 200 字符）。提交前需做表单校验，保存成功后返回上一页并刷新数据。

我把整份 PRD 贴给 AI，要求它以"资深 Android 架构师"的身份，输出：功能点列表（按用户故事拆分）、技术要点、风险点、以及需要向产品确认的模糊点。

AI 的输出让我有些意外——它不仅完成了标准输出，还**标注了 PM 漏掉的一个边界条件**：如果用户选择了新头像但还没上传就退出页面，是否需要弹窗提醒保存。我把这个问题截图发给 PM，PM 说："确实没考虑，加上吧。"

这就是 AI 在产品阶段的第一个价值：**它不是替代产品思考，而是补全产品思考**。人在读 PRD 时太容易代入"工程师视角"——只关心怎么实现，忽略了"用户视角"下的交互细节。AI 因为没有预设视角，反而能在两个视角之间切换。

最终 AI 产出了一份技术要点清单，大概长这样：

| 编号 | 功能 | 优先级 | 复杂度 | 涉及技术 |
|------|------|--------|--------|----------|
| F1 | 头像选择与裁剪 | P0 | 中 | CameraX / PhotoPicker + uCrop |
| F2 | 表单输入与校验 | P0 | 低 | Compose TextField + 自定义 Validator |
| F3 | 资料保存提交 | P0 | 中 | Retrofit Multipart + StateFlow |
| F4 | 保存后数据同步 | P1 | 低 | SavedStateHandle + 事件总线 |

时间：3 分钟。传统做法：读 PRD、划重点、写技术笔记——至少 30 分钟。

### 1.2 技术方案骨架，5 分钟

接着我让 AI 基于技术要点生成一份完整的技术方案：架构选型、模块划分、接口设计、数据流设计。

模块划分的输出很标准：

```
feature:user/
├── EditProfileScreen.kt
├── EditProfileViewModel.kt
├── ProfileFormValidator.kt
└── components/
    ├── AvatarPicker.kt
    └── GenderSelector.kt
data/
├── ProfileRepository.kt
├── ProfileRepositoryImpl.kt
└── UserApiService.kt
```

我唯一的人工调整是把"用户模块"归入现有的 `feature:user` 而非新建一个独立模块——这个决策涉及对团队规范的判断，AI 猜不到。

还有一个惊喜：AI 在 API 接口设计阶段，自动生成了 Multipart 格式的上传方法签名，连 `@Part` 注解都写对了。Retrofit Multipart 配置在团队里是出了名的"每次都要查文档才能写对"的东西，AI 直接省了这一步。

### 1.3 工时评估 + 配置沉淀，7 分钟

最后我让 AI 估算工时并标注置信度。它给出的数字：

| 功能 | UI | 逻辑 | 数据层 | 测试 | 合计 | 置信度 |
|------|-----|------|--------|------|------|--------|
| 头像选择裁剪 | 1d | 0.5d | 0.5d | 0.5d | 2.5d | 中 |
| 表单输入校验 | 1d | 0.5d | — | 0.5d | 2d | 高 |
| 资料保存提交 | 0.5d | 0.5d | 0.5d | 0.5d | 2d | 高 |
| 数据同步 | 0.5d | 0.5d | 0.5d | 0.5d | 2d | 低 |

表单校验的预估值偏低（我手动上调了 0.5d），但整体偏差在可接受范围内。AI 搞不定的主要是"和已有业务的耦合度"这种隐性复杂度——它不知道我们的用户模块后台接口有多么不稳定。

产品阶段总计 15 分钟，产出：技术方案文档、工时估算表、更新的 Rules 文件。传统方式这个阶段至少要 2 小时。

## 二、UI 阶段：3 轮迭代，从 60% 到 94% 还原度

### 2.1 第一轮翻车：间距差了 4dp

我把 Figma 设计稿截图喂给 AI，让它生成 Compose 代码。AI 快速产出了布局骨架——结构是对的，但颜色和间距惨不忍睹。最明显的问题是**全局间距统一差了 4dp**：设计稿里卡片上下间距是 16dp，AI 写成了 12dp；按钮圆角是 8dp，AI 写了 ?dp。

这暴露了 Compose 代码生成的一个根本问题：**AI 对"视觉细节"缺乏精确感知**。截图识别不是 Figma Dev Mode 的像素级读取，AI 只能猜大概。

于是我调整了策略，把 Design Token 明确定义好再喂进去：

```
主色：#FF6B35
标题字体：20sp，Medium
正文字体：14sp，Regular
圆角：12dp（卡片），8dp（按钮）
间距系统：4-8-16-24
```

第二轮效果明显提升，但新的问题出现了——AI 在某些地方仍然用错了间距等级。我不再逐一用文字描述"这里改成 16dp，那里改成 8dp"，而是**在 Figma Dev Mode 里截图标注数字，发给 AI**。它立刻改对了。

这让我总结出了一个关键的沟通技巧：**文字描述 UI 问题是歧义最大的方式，"这里间距不对"远不如一张标了红框和数字的截图**。后面几个项目我开始直接用 Figma 的 comment 功能截图标注问题区域，AI 修复效率提高了至少 3 倍。

### 2.2 我总结的 3 轮迭代法

经过这个项目的试错，我沉淀了一套可靠的 Figma → Compose 转换流程：

**第 1 轮：结构骨架（5 分钟）**
只关注布局层次和组件类型。颜色用占位值，忽略字体细节。要求 AI 列出所有子组件及其嵌套关系。这一轮的目标是"组件数量和层次关系正确"。

**第 2 轮：像素级对齐（15 分钟）**
基于第 1 轮代码，逐个对齐背景色、文字色（精确到 hex）、字体大小和粗细、padding/margin（精确到 dp）、分割线、阴影渐变。配合 Figma Dev Mode 和截图标注反馈。

**第 3 轮：交互状态（10 分钟）**
补充按钮的 pressed/disabled 状态、列表的 Loading/Empty/Error 三态、表单校验错误的视觉反馈、暗黑模式适配（`LightColorScheme` + `DarkColorScheme` 双份）。

| 迭代轮次 | 还原度 | 耗时 | 人工修改量 |
|----------|--------|------|-----------|
| 第 1 轮 | ~60% | 5 min | 30+ 处 |
| 第 2 轮 | ~85% | 15 min | 10 处 |
| 第 3 轮 | ~94% | 10 min | 3 处 |
| 纯手写 | ~100% | 120 min | — |

94% 还原度对于这个功能来说足够了——剩下的 6% 是动画细节和微交互，人工写更划算。

### 2.3 什么行，什么不行

**Design Token 注入非常有效**。提前定义好颜色、字体、间距常量，AI 产出的代码风格一致性极高，不需要后续"找颜色不一致替换"的苦力活。

**复杂动画基本不行**。头像上传的进度条动画、表单校验抖动的弹性动效——AI 要么写不出来，要么写出性能灾难（无限重组）。这部分我是手写的。

**嵌套滚动也有坑**。页面里有一个在 `LazyColumn` 里嵌套 `HorizontalPager` 的结构，AI 的方案导致了滑动冲突。这个问题不完全是 AI 的锅——嵌套滚动在 Compose 本身就是个坑——但 AI 没有主动提醒我"这个布局有潜在冲突"，而是直接生成了有问题的代码。后来我在 Rules 文件里加了一条："涉及嵌套滚动时，先说明潜在问题再生成代码"。

## 三、开发阶段：TDD 模式，AI 写测试，我审实现

### 3.1 ViewModel 一次过

先说好消息：`EditProfileViewModel` 是 AI 一次性生成通过的。我给了它接口定义和状态模型，几秒钟后它就产出了完整代码：

```kotlin
@HiltViewModel
class EditProfileViewModel @Inject constructor(
    private val profileRepository: ProfileRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow<EditProfileUiState>(EditProfileUiState.Initial)
    val uiState: StateFlow<EditProfileUiState> = _uiState.asStateFlow()

    fun updateNickname(nickname: String) { /* ... */ }
    fun updateGender(gender: Gender) { /* ... */ }
    fun updateBio(bio: String) { /* ... */ }
    fun selectAvatar(uri: Uri) { /* ... */ }
    fun submit() { /* ... */ }
}
```

代码风格、错误处理、协程作用域——全部符合项目规范。为什么？因为 **ViewModel 是 Android 开发中最模式化的组件**。MVVM + StateFlow 的写法高度固定，AI 的训练数据极其充分。对于这类"标准套路"，AI 的成功率接近 100%，基本不需要人工介入。

### 3.2 Repository 踩坑：错误处理不完整

ViewModel 没问题，但 `ProfileRepositoryImpl` 花了 2 轮迭代才通过所有测试。

第一轮 AI 产出的 Repository 实现，正常流程是对的，但异常处理有两处遗漏：
1. **文件上传超时没有正确处理**：AI 只 catch 了 `HttpException`，但 Multipart 上传可能抛出 `SocketTimeoutException`，这种情况应该转成可恢复的 `Result.failure` 而不是让异常飞上去
2. **并发上传头像和资料时没有做去重保护**：用户在弱网环境下快速点击两次提交，会触发两次上传请求，浪费带宽且可能产生重复数据

第二个问题是我做 Code Review 时发现的。AI 的 Review 能力（我让它先自查了一遍）没发现这个并发问题——**AI 对运行时副作用（网络请求的幂等性、线程安全性）的判断力较弱**，这部分必须人工把关。

### 3.3 一个吓我一跳的 Bug：进程死亡

测试时我发现了一个诡异的问题：在开发者选项中开启"不保留活动"，进入资料编辑页、切到后台、再切回来——昵称和简介输入框的内容都丢了。

排查后发现，AI 生成的 ViewModel 使用的是普通 `MutableStateFlow` 作为状态容器，没有通过 `SavedStateHandle` 持久化表单草稿。进程被杀死重建后，ViewModel 重新初始化，表单数据全部归零。

这是我**在 Code Review 阶段没有看出来的问题**——因为代码逻辑本身是对的，问题在于缺少一个系统级的状态保持机制。AI 不了解 Android 框架的"进程死亡"这个隐式需求，除非你在 Prompt 里明确提到。

修复很简单，把关键状态通过 `SavedStateHandle` 保存：

```kotlin
// 修复后
private val _nickname = savedStateHandle.getStateFlow("nickname", "")
```

但这件事值得反思：AI 擅长处理"你明确告诉它"的事，但对"你没告诉它的系统约束"几乎盲视。当你用 AI 写 Android 代码时，必须把自己当成最后一道防线——你知道哪些是框架层面的硬需求，而 AI 不知道。

## 四、测试阶段：AI 写了 90% 的测试

### 4.1 单元测试：AI 的舒适区

我采用 TDD 流程：先让 AI 写测试 → 我审核 → 运行确认失败 → 让 AI 写实现 → 运行确认通过。

AI 生成单元测试的质量非常高。以 `ProfileRepositoryTest` 为例，它覆盖了：
- 获取资料成功 / 失败 / 超时
- 更新资料成功 / 校验失败 / 网络异常
- 上传头像成功 / 文件过大 / 格式不支持

唯一需要我手动补充的是两个边界场景：文件上传时模拟进度回调的中断情况，以及在网络恢复后的自动重试逻辑。这些场景比较冷门，AI 的测试用例生成有明显的"覆盖主干、忽略分支"倾向。

### 4.2 Compose UI 测试：AI 的短板

Compose UI 测试方面，AI 的表现就没那么好了。让它写"验证 Loading 状态下显示进度条"这种静态断言没什么问题，但一旦涉及手势交互——比如长按拖拽排序、双指缩放裁剪头像区域——AI 生成的测试要么不可靠（有时通过有时失败），要么直接用 `Thread.sleep()` 代替 `awaitIdle()`。

**复杂手势交互的 UI 测试我全部手写了**。这是我的选择：与其花时间教 AI 写一段不可靠的测试再修，不如自己直接写。AI 在 UI 测试领域的训练语料远少于单元测试，这个差距短时间内不会消失。

### 4.3 覆盖率：45% → 82%

最终成果：
- **代码覆盖率**：82%（AI 生成前约 45%）
- **分支覆盖率**：76%
- 单元测试全部 AI 生成（我补充了约 10% 的边界场景）
- Compose UI 测试 70% AI 生成，30% 手写
- 集成测试 80% AI 生成（MockWebServer + 内存 Room）

## 五、算账：数字不会骗人

| 阶段 | 传统方式 | AI 辅助 | 提效比 |
|------|---------|---------|--------|
| 产品（PRD 解读 + 技术方案 + 工时） | ~2h | 15min | **8x** |
| UI（设计稿转代码 + 3 轮校准） | ~2h | 40min | **3x** |
| 开发（ViewModel + Repository + 联调） | ~4h | 1h | **4x** |
| 测试（单元 + UI + 集成 + 覆盖率） | ~2h | 30min | **4x** |
| **合计** | **~10h** | **~2.5h** | **4x** |

分解一些有意思的数据：

- **AI 写了约 70% 的代码**（按行数统计），主要集中在数据层、ViewModel 和测试。UI 代码的 AI 产出占比反而不到 50%，因为大量的样式微调是我手动完成的。
- **AI 引入的 Bug 数量：3 个**。进程死亡状态丢失（前面说了）、表单校验的边界条件（空字符串 = 0 字符 vs 纯空格 = 0 字符）、头像缓存未清理导致提交了旧文件。三个 Bug 全部在 Code Review 阶段被我发现，上线前零逃逸。
- **我的角色从"码农"变成了"决策者"**。架构选型、模块边界划分、线程模型设计、最终审核——这些高层次判断仍然是我在做。AI 不能替你思考，但可以替你执行。

## 六、如果重来一次，我会怎么做

### 问题 1：上下文配置没有一步到位

最初我只在 Rules 里写了技术栈和架构规范。后面发现每轮 AI 对话都要额外解释"这个颜色值用 DesignToken 引用不要硬编码"、"图片加载用 Coil 不要用 Glide"——这些我应该一开始就写进 Rules。**项目中后期补上下文配置的成本远高于初期一次性投入**。现在我维护的 Rules 文件有将近 200 行，涵盖了编码规范、常见模式、禁止事项、甚至几个关键第三方库的版本号。

### 问题 2：Prompt 粒度太粗

第一次做 UI 生成时，我直接把整个页面的设计稿截图扔给 AI，要求它"生成完整页面代码"。结果产出了一份 500 行的 Composable 函数，层次混乱，调试难度极高。

后来拆成了 4 个子区域单独生成——顶部栏、头像区域、表单区域、底部按钮——每个区域独立验证后再组装。**小粒度的简洁 Prompt，效果远胜一次性的冗长 Prompt**。AI 在有限上下文内更容易保持一致性。

### 问题 3：UI 反馈方式是瓶颈

"这个按钮往上移一点"、"这里的间距太大了"——这些模糊描述导致的来回沟通是效率的最大浪费。后来的项目我改用 Figma Dev Mode 的数值标注 + 红框截图，AI 一次就能改对。**当你觉得 AI 理解不了你的意图时，先反思一下自己的表达方式是否足够精确**。

## 七、上线与回流：别让流程停在"测试通过"

前面几个阶段跑完，功能在测试环境里已经看起来很稳定了。但真实线上环境和测试环境最大的区别是：**用户行为不可控，网络状态不可控，设备分布不可控**。如果流程只做到"测试全绿"，其实还差最后一段关键闭环。

我现在固定会做 **发包前检查 → 灰度 → 监控 → 回滚演练 → 复盘** 五步。其中 **7.0 发包前** 对 AI 辅助项目尤其关键：模型提速后，Gradle/Manifest 类问题更容易被「带上线」。

### 7.0 发包前：Android 构建与 Manifest 检查

AI 改过多模块或依赖后，打 Release 包前必须过一遍检查项（可与 CI 对齐）。

**可直接复制进 PR 的完整清单**：[`templates/release-checklist.md`](../../templates/release-checklist.md)（含 Markdown 勾选块 + 灰度/回滚占位）。

摘要：

| 类别 | 要点 |
|------|------|
| **Gradle** | catalog 单一版本源；`api`/`implementation`；`assembleRelease` 或模块级 compile；无 Duplicate class |
| **Manifest** | 查看 Merged Manifest；Debug 不泄漏；不删 SDK 节点（[坑十一](./08-常见陷阱与解决方案.md)） |
| **AI 加查** | toml/gradle 人工 Diff；`sealed`/`when` 穷尽；无密钥文件提交 |

勾选与阈值须人工确认；AI 只可生成检查表初稿。

### 7.1 灰度策略先写，再发包

过去我经常是先把包发出去，再看监控。后来发现这很容易被动。现在做法是上线前先写一份灰度卡片，至少包含：

- 灰度范围：例如 5% → 20% → 50% → 100%
- 关键指标：崩溃率、ANR、接口错误率、关键转化率（如资料保存成功率）
- 观察窗口：每一档至少观察 2-4 小时
- 回滚条件：例如崩溃率超过基线 20%，或保存成功率下降超过 5%

这一步我会让 AI 先产出初稿，再按业务优先级人工补齐阈值。AI 擅长把模板写完整，但**阈值该设多严，必须由你拍板**。

### 7.2 监控不是"有没有报错"，而是"业务有没有变差"

上线后我会让 AI 帮我汇总监控维度，但我只看三类信号：

1. **稳定性**：Crash / ANR 是否异常抬头
2. **可用性**：接口超时、失败码、重试次数是否上升
3. **业务性**：编辑资料提交成功率、页面停留时长、退出率是否劣化

以前我只盯 Crash，后来吃过亏：没有 Crash，但提交成功率掉了 8%，因为服务端字段兼容有问题。**不看业务指标的"稳定"，是假的稳定。**

### 7.3 回滚预案要提前演练一次

很多团队把回滚写在文档里，但从没真正演练过。我现在会在预发布环境做一次"模拟失败"：

- 人工制造一个可控异常（例如接口超时阈值调低）
- 按预案触发告警
- 执行回滚动作（版本回退或开关降级）
- 记录恢复时间（MTTR）和失败点

这一步特别适合让 AI 帮你生成"回滚检查清单"和演练记录模板，但执行动作本身一定要人工确认，避免自动化误操作。

### 7.4 复盘要回灌到 Rules，不然同样的坑还会再踩

每次上线后，我会做一个 20 分钟小复盘，只回答 3 个问题：

- 这次 AI 产出的哪一类内容最可靠？
- 哪个 Bug 本可以在更早阶段拦住？
- 哪条规则如果提前写进 Rules，就能避免这次问题？

然后把结论回灌到 `CLAUDE.md` / `AGENTS.md` / 模块 Rules。这样流程才算闭环：**线上问题 → 复盘结论 → 规则更新 → 下次自动避免**。

下面是我最近一次上线后的简化记录：

| 维度 | 结果 | 处理动作 |
|------|------|---------|
| 灰度 5%（2h） | 崩溃率正常，提交成功率 -1.2% | 继续放量 |
| 灰度 20%（4h） | 网络超时上升 | 调整重试策略，观察 2h |
| 灰度 50%（4h） | 指标恢复基线 | 放量到 100% |
| 全量 24h | 稳定 | 复盘并更新 Rules 2 条 |

这套流程增加的工作量不大（通常 30-60 分钟），但它把"上线靠感觉"变成了"上线有护栏"。对 AI 辅助开发来说尤其重要：AI 能把开发速度拉快，你更需要一套机制把风险收回来。

## 写在最后

如果要用一句话总结"AI 全流程参与 Android 开发"的体验：

> AI 负责写初稿，你负责做决策。

产品阶段，AI 帮你从 PRD 里提取技术要点、生成方案骨架——但哪些技术选型符合团队规范、哪些风险需要加缓冲，只有你知道。

UI 阶段，AI 帮你写布局代码、对齐样式——但复杂动画、嵌套滚动、交互细节的最终打磨，需要你的经验和审美。

开发阶段，AI 帮你写 ViewModel 和 Repository 的初稿——但线程安全、并发保护、进程死亡等框架层面的隐式约束，是你的"常识"，不是 AI 的。

测试阶段，AI 帮你生成 90% 的测试用例——但手势交互、边界条件、异常注入等复杂测试场景，仍然是你补充的那 10% 决定了整体质量。

4 倍效率提升是真的。Bug 会变少是真的。但 AI 不会替你承担"代码出了问题谁负责"这个问题——答案是，你。

**贯穿全流程的铁律只有一条：不要让 AI 替你思考架构，但可以让它替你执行重复劳动。**
