## 7.1 产品阶段：AI 辅助需求到技术方案

### 本节目标

学会用 AI 加速"PRD → 技术方案"的全过程，让 AI 帮你做需求解读、架构拆解、工时预估和项目上下文配置。

### 前置知识

- 熟悉 Android 常见架构模式（MVVM、MVI）
- 了解项目常用技术栈（Retrofit、Room、Hilt、Compose）
- 有 PRD 或需求文档的基本阅读经验

---

### 7.1.1 用 AI 解读 PRD，提取技术要点

一份 PRD 往往包含大量业务描述、交互细节、边界条件。传统做法是工程师逐字阅读、划重点、列技术清单。用 AI 可以大幅压缩这个环节的耗时。

**操作流程**：

```
PRD 原文 → AI 结构化提取 → 技术要点清单 → 人工确认
```

**推荐 Prompt 模板**：

````markdown
你是一位资深 Android 架构师。请分析以下 PRD，输出：

1. **功能点列表**：按用户故事拆分，用表格列出（编号、功能、优先级、复杂度初步评估）
2. **技术要点**：每个功能涉及的技术栈（网络请求、本地存储、UI 组件等）
3. **风险点**：可能的技术难点、依赖冲突、性能瓶颈
4. **澄清问题**：需要向产品确认的模糊点

PRD 内容如下：
---
[粘贴 PRD 原文]
---
````

**输出示例**：

| 编号 | 功能 | 优先级 | 复杂度 | 涉及技术 |
|------|------|--------|--------|----------|
| F1 | 用户登录/注册 | P0 | 中 | Retrofit + DataStore + Biometric |
| F2 | 个人资料编辑 | P1 | 低 | Compose Form + Retrofit + 图片裁剪 |
| F3 | 消息推送 | P1 | 高 | FCM + NotificationChannel + WorkManager |
| F4 | 离线缓存 | P2 | 中 | Room + NetworkBoundResource |

**关键技巧**：

- 如果 PRD 很长，分章节喂给 AI，每次聚焦一个模块
- 让 AI 用 Mermaid 或文本流程图描述业务流程，直观发现遗漏
- 要求 AI 以"开发视角"而非"产品视角"输出，避免冗余的业务复述

### 7.1.2 技术方案拆解：架构选型、模块划分、接口设计

拿到技术要点后，下一步是制定技术方案。AI 在这一步的强项是快速生成 **骨架方案**，供你评审和细化。

**Prompt 模板**：

````markdown
基于以下技术要点，请输出一份 Android 技术方案，包含：

1. **架构选型**：推荐架构模式及理由（MVVM/MVI/Clean Architecture）
2. **模块划分**：建议的 Gradle Module 结构，说明每个模块的职责
3. **关键接口设计**：以 Kotlin 代码形式给出核心 Repository/Service 接口
4. **数据流设计**：用 ASCII 流程图描述关键场景的数据流转
5. **依赖清单**：第三方库选型建议（版本、用途、替代方案）

技术要点：
- [粘贴 7.1.1 的技术要点清单]
- 项目约束：[Jetpack Compose / ViewBinding]、[Hilt / Koin]、[单模块 / 多模块]
````

**AI 产出的模块划分示例**：

```
app/                    # Application 壳，DI 装配
├── feature/
│   ├── auth/           # 登录注册模块
│   ├── profile/        # 个人资料模块
│   └── notification/   # 推送模块
├── core/
│   ├── network/        # 网络层封装
│   ├── database/       # 数据库封装
│   ├── ui/             # 通用 UI 组件
│   └── common/         # 工具类、扩展函数
└── data/               # 数据层（DTO、Repository 实现）
```

**人工介入点**：AI 给出的模块划分是通用解，你需要根据团队规范调整。例如，你们的项目是否统一使用 `domain/` 层做 UseCase、是否强制 DataStore 而非 SharedPreferences。

### 7.1.3 工时预估：AI 辅助评估复杂度

传统工时评估依赖经验，容易出现"拍脑袋"式的偏估。AI 可以基于功能拆解结果给出参考估算，但最终仍需人工校准。

**Prompt 模板**：

````markdown
以下是某 Android 项目的功能拆分表。请为每个功能估算开发工时（单位：人天），
并标注预估置信度（高/中/低）及估算依据。

估算维度：UI 开发 + 业务逻辑 + 网络/数据库 + 单元测试 + 联调

[粘贴功能拆分表]
````

**输出示例**：

| 功能 | UI | 逻辑 | 数据层 | 测试 | 联调 | 合计 | 置信度 |
|------|-----|------|--------|------|------|------|--------|
| F1 登录注册 | 1d | 1d | 0.5d | 0.5d | 0.5d | 3.5d | 高 |
| F2 资料编辑 | 1.5d | 1d | 0.5d | 0.5d | 0.5d | 4d | 中 |
| F3 消息推送 | 0.5d | 1.5d | 1d | 1d | 1d | 5d | 低 |

**注意**：AI 的估算是基于通用经验的参考值。实际工时受团队能力、代码质量要求、联调复杂度影响。建议在 AI 估算基础上预留 20%-30% 缓冲。

### 7.1.4 Rules 文件配置：让 AI 理解项目上下文

这是整个流程的**关键第一步**。如果你不给 AI 项目上下文，它会用"通用 Android 知识"作答，导致技术方案不切实际。

**CLAUDE.md / GEMINI.md / AGENTS.md 示例**：

````markdown
# 项目上下文

## 技术栈
- 语言：Kotlin 1.9+
- UI：Jetpack Compose + Material 3
- DI：Hilt
- 网络：Retrofit + OkHttp + Kotlinx Serialization
- 本地存储：Room + DataStore
- 图片加载：Coil 3
- 测试：JUnit 5 + MockK + Turbine

## 架构规范
- MVVM + Repository 模式
- 统一使用 sealed class 封装 UI State
- ViewModel 通过 StateFlow 暴露状态
- 禁止在 Activity/Fragment 中直接调用 Repository

## 代码风格
- 使用 ktlint 官方规则
- 函数体超过 30 行必须拆分
- Composable 函数命名以大写字母开头
- 禁止硬编码字符串，统一使用 strings.xml

## 模块依赖规则
- feature 模块不得互相依赖
- core 模块不得依赖 feature 模块
- 跨模块通信通过接口抽象 + DI 注入

## 常见模式
```kotlin
// UI State 示例
sealed class ProfileUiState {
    object Loading : ProfileUiState()
    data class Success(val profile: Profile) : ProfileUiState()
    data class Error(val message: String) : ProfileUiState()
}
```
````

将这个文件放在项目根目录（或 `.claude/`、`.gemini/` 目录），AI 工具会自动加载，确保每次对话都在正确的上下文中。

**配置增量更新建议**：

- 每次踩坑后，将解决方案沉淀到 Rules 文件
- 版本号精确到 minor，避免 AI 混用不兼容的 API
- 添加"禁止事项"清单，防止 AI 重复犯已知错误

### 7.2.1 Figma 链接 → AI 解析设计稿

AI 工具（如 Cursor、Copilot、Claude Code）可以直接读取 Figma 链接或截图，解析出设计稿的结构和样式。

**操作流程**：

```
Figma 设计稿 → AI 截图/链接识别 → 结构解析 → 组件树描述 → 代码生成
```

**推荐做法**：

1. **在 Figma 中打开 Dev Mode**，获取设计稿的精确数值（间距、字体、颜色）
2. **截图 + 文字描述**双重输入：截图让 AI 视觉理解，文字描述补充交互状态
3. **分层喂入**：复杂页面按区域分段（顶部栏 → 主体内容 → 底部操作区），避免上下文溢出

**Prompt 模板**：

````markdown
请分析以下 Figma 设计稿[截图]，将该页面转换为 Jetpack Compose 代码。

要求：
1. 识别所有 UI 区域并分层描述
2. 提取颜色、字体、圆角、间距等 Design Token
3. 标注动态内容区域（列表、图片、文本）
4. 识别组件复用机会

[附上设计稿截图或 Figma 链接]
````

### 7.2.2 Compose 代码生成

**完整 Prompt 模板**：

````markdown
请基于以下设计描述生成 Kotlin + Jetpack Compose 代码：

【页面名称】商品详情页
【框架要求】Compose + Material 3
【架构】ViewModel + StateFlow
【设计 Token】
- 主色：#FF6B35，辅色：#004E89
- 标题字体：20sp，Medium
- 正文字体：14sp，Regular
- 圆角：12dp（卡片），8dp（按钮）
- 间距系统：4-8-16-24

【布局描述】
1. 顶部：图片轮播区（高度 375dp，使用 HorizontalPager）
2. 中间：商品信息区（标题、价格、规格、库存）
3. 底部：操作栏（收藏 + 加入购物车 + 立即购买）

请输出：
- DesignToken.kt（颜色、字体、间距常量）
- 页面 Composable 代码
- ViewModel 骨架
- Preview 函数
````

**生成代码示例**（AI 输出）：

```kotlin
// DesignToken.kt
object DesignToken {
    object Colors {
        val Primary = Color(0xFFFF6B35)
        val Secondary = Color(0xFF004E89)
        val Background = Color(0xFFF5F5F5)
        val OnPrimary = Color.White
    }

    object Typography {
        val Title = TextStyle(fontSize = 20.sp, fontWeight = FontWeight.Medium)
        val Body = TextStyle(fontSize = 14.sp, fontWeight = FontWeight.Normal)
        val Price = TextStyle(fontSize = 24.sp, fontWeight = FontWeight.Bold, color = Colors.Primary)
    }

    object Spacing {
        val xs = 4.dp
        val sm = 8.dp
        val md = 16.dp
        val lg = 24.dp
    }

    object Radius {
        val card = 12.dp
        val button = 8.dp
    }
}

// ProductDetailScreen.kt
@Composable
fun ProductDetailScreen(
    viewModel: ProductDetailViewModel = hiltViewModel(),
    onBack: () -> Unit,
    onAddToCart: (Product) -> Unit,
    onBuyNow: (Product) -> Unit,
) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()

    Scaffold(
        topBar = { ProductDetailTopBar(onBack = onBack) },
        bottomBar = { ProductBottomBar(/* ... */) }
    ) { paddingValues ->
        when (val state = uiState) {
            is ProductUiState.Loading -> LoadingIndicator()
            is ProductUiState.Success -> ProductContent(
                product = state.product,
                modifier = Modifier.padding(paddingValues)
            )
            is ProductUiState.Error -> ErrorView(state.message) { viewModel.retry() }
        }
    }
}
```

### 7.2.4 设计稿还原度调优流程

单次生成通常只能达到 60%-70% 还原度。推荐**3 轮迭代法**：

```
第 1 轮：结构生成      →  大框架正确（布局层次、组件类型）
第 2 轮：样式对齐      →  颜色、字体、间距精确匹配
第 3 轮：细节微调      →  交互状态、动画、Edge Case
```

**第 1 轮 Prompt 策略**：聚焦"结构正确"
```
只关注布局层次和组件类型，颜色用占位值，忽略字体细节。列出所有子组件及其嵌套关系。
```

**第 2 轮 Prompt 策略**：聚焦"视觉还原"
```
基于第 1 轮代码，逐个对齐以下样式：(1) 背景色和文字色精确到 hex 值 (2) 字体大小、粗细、行高
(3) padding 和 margin 精确到 dp (4) 分割线颜色和高度 (5) 阴影、渐变等特效
```

**第 3 轮 Prompt 策略**：聚焦"交互与动画"
```
为以下交互状态补充代码：(1) 按钮的 pressed/disabled 状态 (2) 列表的加载中/空数据/加载失败状态
(3) 图片轮播自动播放和指示器联动 (4) 规格选择的高亮切换动画 (5) RTL 布局适配
```

### 7.2.6 踩坑记录

| 问题 | 现象 | 解决方案 |
|------|------|----------|
| 颜色值偏差 | AI 输出的 hex 与设计稿不一致，偏差 1-2 个色阶 | 明确在 Prompt 中列出 Design Token 的精确 hex 值，要求 AI 使用 `Color(0xFFXXXXXX)` 长格式避免混淆 |
| 字体映射错误 | AI 使用了系统默认字体而非设计稿指定的自定义字体 | 在 Rules 文件中配置 `FontFamily` 定义，并要求 AI 通过 `MaterialTheme.typography` 引用 |
| 复杂布局识别失败 | 嵌套层级多的页面，AI 会忽略某些子组件或将同级元素错放为父子 | 分层喂入设计稿：先主体布局，再逐个区域细化 |
| 图片尺寸缺失 | AI 使用了猜测的宽高值，导致拉伸变形 | 在 Prompt 中提供 Figma 的宽高数据，使用 `fillMaxWidth().aspectRatio()` 固定比例 |
| 暗黑模式遗忘 | 生成的代码只考虑了 Light 主题 | 在 Rules 中强制要求生成 `LightColorScheme` + `DarkColorScheme` 双份 |

### 7.3.1 功能实现：TDD 模式

**TDD 的标准循环**：先写测试 → 跑失败 → 写实现 → 跑通过 → 重构。引入 AI 后，流程变为：

```
AI 生成测试 → 人工审核测试 → 运行确认失败 → AI 生成实现 → 运行确认通过 → AI 重构优化
```

**实操 Prompt（第 1 步：生成测试）**：

````markdown
以下是一个 ProfileRepository 的接口定义。请用 JUnit 5 + MockK 为其编写完整的单元测试，
覆盖正常流程、异常流程和边界条件。

```kotlin
interface ProfileRepository {
    suspend fun getProfile(userId: String): Result<Profile>
    suspend fun updateProfile(profile: Profile): Result<Unit>
    suspend fun uploadAvatar(userId: String, file: File): Result<String>
}
```

要求：
- 每个测试方法独立运行，不依赖执行顺序
- 使用 MockK 的 coEvery/coVerify
- 测试方法名使用清晰的英文描述
````

**实操 Prompt（第 2 步：生成实现）**：

````markdown
基于以下测试代码，编写 ProfileRepositoryImpl 使其全部通过。

[粘贴测试代码]

要求：
- 实现类名为 ProfileRepositoryImpl
- 使用 Retrofit 进行网络请求，Room 进行本地缓存
- 网络失败时返回 Result.failure，不抛异常
````

**关键认知**：不是"AI 替你写代码"，而是"AI 替你写初稿，你来审核和决策"。测试代码要由你来确认边界条件是否覆盖完整，实现代码要由你来检查线程安全和错误处理。

---

### 7.3.2 代码重构：AI 辅助架构迁移

**场景**：将一个 MVC 风格的 Activity 重构为 MVVM + Compose。

**Prompt 模板**：

````markdown
以下是一个遗留的 Activity 代码，包含大量业务逻辑。请将其重构为 MVVM + Compose 架构。

要求：
1. 提取业务逻辑到 ViewModel，使用 StateFlow 暴露状态
2. UI 部分用 Compose 重写
3. 保留原有的 Retrofit 调用，迁移到 Repository 层
4. 保持对外接口不变（Intent 接收、onActivityResult 等）
````

### 7.3.4 多模块协作：让 AI 理解模块间依赖

### 7.4.1 单元测试自动生成

针对已有的实现代码反向生成测试，是 AI 最高效的使用场景之一。

**Prompt 模板**：

````markdown
为以下 ViewModel 生成完整的单元测试，使用 JUnit 5 + MockK + Turbine。

覆盖场景：
- 初始状态验证
- 成功路径（完整数据流）
- 异常路径（网络错误、空数据）
- 边界条件（空输入、超长输入）

```kotlin
@HiltViewModel
class SearchViewModel @Inject constructor(
    private val searchRepository: SearchRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow<SearchUiState>(SearchUiState.Idle)
    val uiState: StateFlow<SearchUiState> = _uiState.asStateFlow()

    fun search(query: String) {
        viewModelScope.launch {
            _uiState.value = SearchUiState.Loading
            searchRepository.search(query)
                .onSuccess { _uiState.value = SearchUiState.Success(it) }
                .onFailure { _uiState.value = SearchUiState.Error(it.message ?: "Unknown") }
        }
    }
}
```
````

```kotlin
    // given: Repository 返回成功
    // when: 调用 search("Kotlin")
    // then: uiState 依次为 Loading → Success
}

@Test

@Test
```
