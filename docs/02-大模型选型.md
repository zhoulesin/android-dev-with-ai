# 7 个大模型写 Android 代码，谁才是真正懂 Kotlin 的那个

我最近干了件挺无聊但很有用的事：让 7 个模型做了同一道题。

题目是写一个 Compose 聊天列表——LazyColumn、头像加载、左右气泡、时间戳格式化、自动滚到底部。听起来不复杂对吧？一个合格的 Android 开发照着需求文档，半小时怎么也能撸出来。但就是这 50 行代码，7 个模型的表现差距大到让我重新思考"到底该为哪个模型买单"。

先看结果：**Claude Opus 4 和 Sonnet 4 断档领先，Gemini 2.5 Pro 在一个你绝对猜不到的场景里拿了满分——Gradle 多模块冲突。GPT-4o 全面但不拔尖，DeepSeek V3 的性价比炸裂到让人心疼，而 GLM-4……这么说吧，如果你只用 GLM-4 写 Android 代码，你的开发体验大约相当于用 2018 年的 Android Studio。**

这篇文章是一次真实的横向评测。不是抄官网 Benchmark，不靠 LMArena 排行榜，而是用五项你在工位上会真实遇到的任务，跑出了七个模型的实际表现。如果你正在为团队选模型，或者纠结要不要给 Claude 付那 20 美金月费，这篇文章应该能帮你做出决定。

## 七位参赛选手

先简单介绍下这七个模型的画风，方便你对号入座：

**GPT-4o** 是班里那个啥都会、但啥都不是第一的学霸。快，快得离谱；生态最完善，各种插件接了一圈；但 Kotlin 的深度确实没到"精通"那层。我之前用它写一个 sealed class 的 when 表达式，它漏了分支，这在 Claude 身上几乎不会发生。

**Claude Opus 4** 是那个交作业能当参考答案的学生。200K 上下文窗口意味着它能吃下你半个项目的代码再给你建议；推理能力是七个里最强的，经常能指出你自己都没意识到的架构问题。缺点只有一个：贵。贵到每调用一次都能听见余额叮当作响。

**Claude Sonnet 4** 是 Opus 4 的亲弟弟，考试少答一道附加题但其他答案几乎一样。如果你大部分工作是写 ViewModel、Composable 和单元测试，它能给你 95% 的 Opus 体验，价格却只有五分之一。这是我目前的主力模型。如果 Opus 是沃尔沃 XC90，Sonnet 就是 XC60——少点马力，但日常完全够用。

**Gemini 2.5 Pro** 是七个里唯一能让我喊"赢麻了"的模型。不是因为它在综合评分上多高（它排第三），而是它在 Gradle 这件事上的表现——有多好我们后面细说。100 万 token 的上下文窗口是独一档的存在，你可以把整个项目的 `build.gradle.kts` 全家桶丢进去，它不会晕。

**DeepSeek V3** 是那个"我不贵但我能干活"的实干派。1M token 的输出成本只要 $1.1，做一次代码生成的边际成本接近零。中文理解是七个里最好的之一，中文注释质量甚至超过 GPT-4o。但 Compose 能力和边界场景处理确实离 Claude 有差距。

**GLM-4** 是国产化项目的合规首选。中文理解最深，插件生态在国内大模型里算丰富的。但实话实说，它的 Android 代码能力是目前七个里最弱的。如果你的技术栈是 Java + XML 的老项目，可能还行；Kotlin + Compose 的话，你得做好经常手动改 AI 输出的心理准备。

**Qwen3-235B** 是阿里通义系的旗舰，开源最强中文代码模型。它的中文注释和文档生成质量高到离谱——我之前让它给一个复杂 ViewModel 加中文注释，输出质量比我手写的还好。同样开源、同样便宜，但私有化部署友好度比 DeepSeek 更高，对数据不出内网有硬性要求的团队这是核心考量。

## 五道题的考卷

评测这种事，最怕自欺欺人。我见过太多评测拿 LeetCode 题或者"请解释 Kotlin 协程"这种知识问答来凑数——那种题测的是模型有没有背过文档，不是它能不能干活。

我的五道题是：

1. **写一个 Compose 聊天列表** —— 从零写 Composable，含 `LazyColumn`、Coil 头像加载、左右气泡、时间戳格式化、自动滚动。这是日常 UI 工作的基本单位。
2. **解决 Gradle 多模块依赖冲突** —— 三个模块，两个间接依赖了不同版本的 Room，导致 Duplicate class 错误。这是每个多模块项目都大概率会遇到的问题。
3. **为 ViewModel 写单元测试** —— JUnit 5 + MockK + Turbine，含 debounce 并发测试。这是团队对代码质量有要求的底线。
4. **Java → Kotlin + Coroutines 迁移** —— 老项目常见的 AsyncTask 到现代协程的重构任务，考的是模型对 Android 生命周期和协程作用域的理解。
5. **ANR 日志分析** —— 给一段 ANR trace，分析原因并给出修复方案。这是线上问题排查的基本功。

每个任务各模型只有**一次生成机会**。不追加 Prompt，不修正。代码由三位资深 Android 开发按编译通过率、最佳实践匹配度、异常场景覆盖度三个维度分别打分，满分 30 分（ANR 任务不涉及编译，满分 20）。额外记录了一个"首次可用率"——代码不经修改就能通过编译的比例。

环境统一：Prompt 中给了相同的技术栈约束（Kotlin 2.x、Compose BOM 2025.x、Hilt），确保起跑线一致。

## Compose 聊天列表：AI 真的懂重组吗？

这是五个任务里最让我意外的。

七个模型里有六个都写出了"看起来能用"的代码。但真正写得漂亮的只有 Claude 两兄弟。它们的差距体现在几个细节上：

**第一个坑：`LazyColumn` 自动滚动。** 需求里有一句"列表自动滚动到底部"，六个模型的处理方式都是"在 `LaunchedEffect` 里调 `scrollToItem`"——看似对，但 Claude Opus 4 多做了一个判断：它只在有新消息加入时才滚动，不会在用户手动往上翻历史消息时强制拉回底部。这个交互细节，GPT-4o 和 Gemini 都没处理。

**第二个坑：`key` 参数。** `LazyColumn` 的 `items` 如果不传 `key`，列表重组时 Compose 无法正确追踪 item 的 identity，性能会大幅下降。DeepSeek V3 和 GLM-4 都没加 `key`，Qwen3 加了但用了 `message.id` 这种语义正确但并不总是唯一的字段。只有 Claude 系列和 Gemini 用了 `message.id` + 明确标注了 `key` 的重要性。

**第三个坑：时间戳格式化。** "今天 HH:mm，昨天 HH:mm，其他 MM/dd"这个需求，GPT-4o、DeepSeek V3、Qwen3 都写出来了，但年份跨年时边界处理有 bug。比如 2025 年 12 月 31 日和 2026 年 1 月 1 日，按它们的逻辑都会显示"MM/dd"，不会有年份提示。Claude Opus 4 单独写了一个工具函数处理了这个边界。

| 模型 | 编译通过 | 最佳实践 | 异常覆盖 | 总分 | 首次可用率 |
|------|:---:|:---:|:---:|:---:|:---:|
| Claude Opus 4 | 10 | 10 | 8 | **28** | 95% |
| Claude Sonnet 4 | 10 | 9 | 7 | **26** | 95% |
| Gemini 2.5 Pro | 9 | 8 | 7 | **24** | 80% |
| GPT-4o | 9 | 8 | 6 | **23** | 85% |
| DeepSeek V3 | 8 | 8 | 5 | **21** | 70% |
| Qwen3-235B | 8 | 7 | 5 | **20** | 70% |
| GLM-4 | 7 | 6 | 5 | **18** | 60% |

如果让我用一句话总结 Compose 能力：**Claude 系列写的代码像是读过 Compose 的源码，GPT-4o 像是读过官方文档，DeepSeek 像是看过几篇 Medium 教程，GLM-4 像是只看了 API reference。** 状态提升（state hoisting）和重组（recomposition）的理解深度，是目前区分模型 Android 能力的核心指标。

Claude Opus 4 的输出代码值得一看——它把消息气泡拆成了独立的 Composable，左右布局通过 `Arrangement.End` 和 `Arrangement.Start` 控制，头像用 `if (!isMine)` 和 `if (isMine)` 决定出现位置：

```kotlin
@Composable
fun ChatMessageItem(message: ChatMessage, isMine: Boolean, modifier: Modifier = Modifier) {
    val arrangement = if (isMine) Arrangement.End else Arrangement.Start
    Row(
        modifier = modifier.fillMaxWidth(),
        horizontalArrangement = arrangement,
        verticalAlignment = Alignment.Bottom,
    ) {
        if (!isMine) ChatAvatar(imageUrl = message.avatarUrl)
        ChatBubble(message = message, isMine = isMine)
        if (isMine) ChatAvatar(imageUrl = message.avatarUrl)
    }
}
```

而 DeepSeek V3 的输出就简陋得多——一个 `Row` 里平铺了 `AsyncImage` 和 `Text`，没有 `isMine` 判断，没有 `key`，也没有关心过性能。

## Gradle 多模块冲突：为什么 Gemini 是隐形王者？

这是五个任务里**唯一一个榜首不是 Claude 的任务**。Gemini 2.5 Pro 拿了满分 28 分，力压 Claude Opus 4 的 26 分。

需求的场景很真实：三个模块，`:feature:home` 和 `:feature:profile` 都依赖 `:core:database`，后者用 Room 2.6.1。但 `:feature:home` 引入了一个第三方库，间接依赖了 Room 2.4.0——Duplicate class 错误。

GPT-4o、DeepSeek V3、GLM-4、Qwen3 都给出了 `exclude group` 的方案，这个思路是对的，但都不完整。只有 Claude Opus 4 和 Gemini 2.5 Pro 给出了三层防御策略：

1. **`exclude` 排除传递依赖**（治标）
2. **`resolutionStrategy.force` 强制统一版本**（治本）
3. **Version Catalog 统一声明版本号**（防复发）

而 Gemini 2.5 Pro 比 Claude 多做了一件事：它解释了 Room 2.4.0 和 2.6.1 之间 ABI 不兼容的根本原因——Room 的编译期注解处理器生成的 `_Impl` 类在跨版本时内部结构会变化，即使 API 兼容也不能混用。这个解释不是一个代码助手能给的，更像是一个熟悉工具链的人。

| 模型 | 编译通过 | 最佳实践 | 异常覆盖 | 总分 |
|------|:---:|:---:|:---:|:---:|
| Gemini 2.5 Pro | 10 | 10 | 8 | **28** |
| Claude Opus 4 | 9 | 9 | 8 | **26** |
| Claude Sonnet 4 | 9 | 9 | 7 | **25** |
| GPT-4o | 8 | 8 | 6 | **22** |
| DeepSeek V3 | 8 | 7 | 5 | **20** |
| Qwen3-235B | 8 | 7 | 5 | **20** |
| GLM-4 | 7 | 6 | 5 | **18** |

我的解读：**Google 内部有巨量的 Android 构建系统训练数据，这部分壁垒其他模型短时间内追不上。** 如果你做的是多模块、Gradle 定制化程度高的项目，Gemini 目前是你绕不开的选项。它的 100 万 token 上下文意味着你可以把全量构建脚本一次丢进去，这在 Claude 200K 窗口也不一定能做到的事。

Gemini 输出的 `resolutionStrategy` 代码也是最完整和安全的：

```kotlin
allprojects {
    configurations.all {
        resolutionStrategy {
            force("androidx.room:room-runtime:2.6.1")
            force("androidx.room:room-ktx:2.6.1")
            force("androidx.room:room-compiler:2.6.1")
        }
    }
}
```

如果你经常和 Gradle 较劲，这句代码应该能省你不少时间。

## 单元测试和 Java→Kotlin 迁移

这两道题我放一起说，因为结果高度一致：**Claude Opus 4 在两个任务上都是满分 29 分**，且是唯一一个在所有五个任务上单项评分都有 9 分以上的模型。

测试任务中最难的考点是 debounce——用户在快速连续输入时要取消前一次搜索只保留最后一次。六个模型都写出了基本测试，但只有 Claude Opus 4 正确使用了 `turbineScope` 验证中间状态的取消。GPT-4o 的 Turbine 语法是正确的，但 debounce 场景没使用 `advanceTimeBy` 来模拟时间推进，所以那个测试实际上没有真正验证 debounce 行为。DeepSeek V3 的 MockK `coEvery` 使用是准确的，但它完全没覆盖并发场景——而这恰恰是 debounce 测试的核心价值。

快速连续搜索的正确测试写法是这样的：

```kotlin
@Test
fun `rapid search triggers debounce and only keeps latest`() = runTest {
    val repository = mockk<SearchRepository>()
    coEvery { repository.search("A") } returns Result.success(emptyList())
    coEvery { repository.search("AB") } returns Result.success(listOf(item))

    val viewModel = SearchViewModel(repository)
    viewModel.uiState.test {
        viewModel.search("A")
        viewModel.search("AB")

        expectMostRecentItem() shouldBeInstanceOf SearchUiState.Loading::class
        val success = awaitItem()
        success shouldBeInstanceOf SearchUiState.Success::class
        (success as SearchUiState.Success).results shouldHaveSize 1
    }
}
```

Java→Kotlin 迁移任务里，GLM-4 犯了一个让我血压升高的错误：它把 AsyncTask 的 `doInBackground` 直接转换成了 `GlobalScope.launch`——在 2025 年的 Kotlin 项目里，`GlobalScope` 基本等于代码审查时被驳回的通行证。每个 Android 开发者都知道应该用 `viewModelScope` 或 `lifecycleScope`，GLM-4 偏偏没做到。

Qwen3 虽然用了正确的作用域，但写的是过时的 `lifecycle.coroutineScope` 而不是 `lifecycleScope`。这说明模型的训练数据里可能是 2019-2020 年的 Kotlin 代码，没有及时更新到当前的惯用法。

ANR 分析任务上，Claude Opus 4 依然是第一（满分 20 分拿了 20）。它不只是说"你用 `Dispatchers.IO` 就好了"（那是 DeepSeek 级别的回答），而是分析了线程 47 持锁的原因——长查询未释放数据库连接——并给出了三种递增的修复方案：Room Flow 响应式查询、手动 `withContext` 切线程、整个数据访问层加 ThreadPool。这种分层方案的质量，已经接近一个高级工程师的工单回复了。

| 排名 | 模型 | 平均得分 | 优势场景 | 短板场景 |
|:---:|------|:---:|------|------|
| 1 | Claude Opus 4 | 26.4 | Compose UI、单元测试、Java→Kotlin | 成本高 |
| 2 | Claude Sonnet 4 | 25.2 | Compose UI、测试、迁移 | 无明显短板 |
| 3 | Gemini 2.5 Pro | 24.2 | Gradle、ANR 分析 | Compose 细节 |
| 4 | GPT-4o | 22.6 | Java→Kotlin、综合均衡 | Gradle、测试并发 |
| 5 | Qwen3-235B | 20.0 | 中文场景、注释质量 | Compose 细节 |
| 6 | DeepSeek V3 | 20.0 | 基础任务、速度 | Compose UI、分析深度 |
| 7 | GLM-4 | 17.4 | 中文理解、合规 | Android 专项能力 |

## 钱的账：一次代码生成本质上是多少钱

技术能力说完，聊点实际的——成本。很多人对 API 成本没概念，觉得"几刀能贵到哪"。但当你把这个数拉大到一年 250 个工作日，算上全队的调用次数，差距就很吓人了。

以下是七个模型 2026 年 5 月的官方 API 定价（每 100 万 token）：

| 模型 | 输入价格 ($) | 输出价格 ($) | 说人话 |
|------|:---:|:---:|------|
| Claude Opus 4 | $15 | $75 | 一个字一个字烧钱 |
| Claude Sonnet 4 | $3 | $15 | 一杯拿铁换一天 AI 编程 |
| GPT-4o | $2.5 | $10 | 性价比不错的全能选手 |
| Gemini 2.5 Pro | $1.25 | $5 | Google 在赔本赚吆喝 |
| GLM-4 | $2.3 | $9.2 | 国产里不算便宜 |
| Qwen3-235B | $0.40 | $1.6 | 已经很便宜了 |
| DeepSeek V3 | $0.27 | $1.1 | 约等于不要钱 |

同一个任务让不同模型做，成本差距超过 100 倍。我算过一个完整 Android 页面（含 ViewModel + Composable + 测试 + DI 配置）的单次生成成本：

- Claude Opus 4：约 $0.15
- Claude Sonnet 4：约 $0.03
- DeepSeek V3：约 $0.001

如果你一天跑 30 次 AI 代码生成（经常用 AI 的人很容易达到），一个月下来：
- Claude Opus 4：$135
- Claude Sonnet 4：$27
- DeepSeek V3：$0.9

这就是为什么"只用最强的"在商业上不成立。不是 OpenAI 或 Anthropic 在割韭菜，是这票价差得实在太大了。

## 我的选型方案：三个模型各司其职

经过半年多的实际使用和这次评测，我现在的日常配置是这样一个三层结构：

**第一层（日常写码，80% 工作量）：Claude Sonnet 4**

写 Composable、写 ViewModel、写 Repository、改 Bug、做 Code Review——日常最有质量要求的工作，全部交给他。它的 Compose 代码质量几乎是 Opus 的水平，但成本只有五分之一。这个性价比在 2026 年的 Android 开发场景下找不到对手。

但我不会用 Sonnet 做架构设计。这是两层区别的关键：当你需要模型在多个技术方案之间做权衡、分析利弊、输出长篇方案时，Sonnet 的推理深度明显不如 Opus。

**第二层（复杂任务，15% 工作量）：Claude Opus 4 + Gemini 2.5 Pro**

架构设计、复杂重构、大型模块迁移，上 Opus。它多的那 4-5 分代码质量，在日常 CRUD 里不明显，但在架构级决策时是"方案完整度 80% vs 95%"的区别。

Gradle 和构建系统相关的一切——Gemini。多模块冲突、依赖排查、Gradle 插件开发、ProGuard 规则，没有模型比它更懂。这个场景下我宁愿用 Gemini 也不用 Opus，因为 Gemini 给的是一个"这个方案能用于线上"的答案，其他模型给的是一个"理论正确"的答案。

**第三层（批量廉价，5% 工作量）：DeepSeek V3**

自动生成测试、写代码注释、格式化数据类、生成 `strings.xml` 多语言翻译——这些重复性高、对创造性要求低、但量大的工作，DeepSeek 是最优解。它在我日常调用中占比可能不超过 5%，但它省下的钱和 Claude 省下的质量，是一个硬币的两面。

一个典型 Android 功能的分工用这个方案跑是这样的：

| 阶段 | 模型 | 为什么 |
|------|-----------|------|
| 技术方案设计 | Claude Opus 4 | 需要深度推理和架构权衡 |
| 核心代码编写 | Claude Sonnet 4 | Compose/Kotlin 代码质量最高 |
| 单元测试生成 | DeepSeek V3 | 量大、重复性高、成本敏感 |
| Code Review | Claude Sonnet 4 | 能发现 subtler 的问题 |
| Gradle 配置 | Gemini 2.5 Pro | 理解多模块构建的复杂关系 |

这个组合算下来，一个 5 人团队的月花费大概在 $40-60，每天约 100 次 AI 调用。如果把所有任务都交给 Opus，同样的调用量费用会奔着 $150-200 去。

## 说点大实话

写这篇文章不是想做一个排行榜然后告诉大家"买第一名的就行"。

我见过太多团队在选型这件事上踩坑，问题都差不多：

**跑分不等于好用。** LMArena 排行榜第一的模型，可能在你写 Compose UI 时连 `remember` 和 `derivedStateOf` 的区别都说不清。所有的公开 Benchmark 都偏向通用能力，对特定领域的"手感"没办法量化。

**贵的不一定对。** Claude Opus 4 是综合最强，但如果你 80% 的时间是在写 Kotlin 数据类和简单 ViewModel，你多付的钱基本是沉没成本。Sonnet 和 Opus 的差距只在边界场景——而大部分日常工作用不到边界场景。

**免费的也不定比得过不免费但便宜的。** 当前国产开源的进度确实快，但在 Android 这种技术栈深度要求高的领域，DeepSeek 和 Qwen 距离 Claude 还是有肉眼可见的差距。这个差距在降本的大背景下可以接受，但不应该假装不存在。

**用多模型是"全都要"的唯一正确答案。** 你不可能既享受 Claude Opus 的推理质量，又享受 DeepSeek 的接近零成本，同时还不让账单爆炸。但你可以把不同任务路由到不同的模型，从而同时拿到质量和成本的双赢。

最后，这件事值得一提：**模型进化的速度太快了。** 这篇文章写于 2026 年 5 月，三个月后的排名几乎一定会变。建议你每季度拿一次自己项目的真实任务跑一次个人评测——数据比排行榜可靠得多。

**没有最好的模型，只有最适合你工作流的模型。** 如果看完这篇文章你只想拿走一句话，我希望是这句。
