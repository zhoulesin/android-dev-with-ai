### 2.1.5 动手实践

**任务**：用同一句 Prompt 测试 3 个模型，感受输出差异。

**测试 Prompt**：

```
Kotlin 中，以下哪种写法更好？为什么？

// 写法 A
val list = mutableListOf<String>()
viewModelScope.launch {
    val data = repository.getData()
    list.addAll(data)
}

// 写法 B
val list = viewModelScope.async {
    repository.getData()
}.await()
```

**操作步骤**：

1. 在 ChatGPT / Claude / Gemini 中分别发送同一段 Prompt
2. 观察三个模型的回答：是否正确指出写法 A 的问题（`mutableListOf` 非线程安全、直接修改 UI 状态不能这样做）？是否正确指出写法 B 的问题（`async` 在 ViewModel 中几乎无用，应该用 `launch` + `StateFlow`）？
3. 比较回答的深度和实用性，记录哪个模型的建议最符合 Android 开发最佳实践

**自检清单**：

- [ ] 模型是否识别了 ViewModel 中不应直接修改可变列表？
- [ ] 模型是否推荐了更优的方案（`StateFlow` + `MutableStateFlow`）？
- [ ] 回答是否带 Kotlin 代码示例？示例是否可编译？

---

### 2.1.6 踩坑记录

| 问题 | 现象 | 解决方案 |
|------|------|----------|
| 模型版本过时 | AI 使用了已废弃的 Kotlin API（如 `kotlinx.android.synthetic`） | 在 Prompt 中明确指定 Kotlin 和 Compose BOM 版本号；在 Rules 文件中声明技术栈版本 |
| 代码混用不同 Compose 版本 | 同一份代码中出现 `material` 和 `material3` 的混用 | 在 Rules 中禁止 `import androidx.compose.material.*`，只允许 `material3` |
| Gradle 语法错误 | 模型在 `build.gradle.kts` 中使用了 Groovy 语法（如单引号字符串、`implementation 'xxx'`） | 在 Prompt 中强调是 `build.gradle.kts` (Kotlin DSL)；Claude/GPT-4o 在这方面表现较好 |
| 中文 Prompt 生成英文代码 | 用中文提问但模型输出英文注释和变量名 | 明确要求"使用中文注释"和"业务变量使用中文拼音或完整英文"；Qwen3 和 GLM-4 中文遵循度更高 |
| 上下文窗口被截断 | 长对话后期，模型"忘记"了项目背景和约束 | 选择 200K+ 上下文模型（Claude Sonnet 4 / Gemini 2.5 Pro）；对于复杂项目开启新会话 |

## 2.2 各模型 Android 开发实测

### 本节目标

### 2.2.1 测试方法说明

**测试规则**：

- 每个任务给各模型**一次生成机会**（不追加 Prompt，不回退修正）
- 代码由资深 Android 开发者按照编译通过率、最佳实践匹配度、异常场景覆盖度三个维度评分（每个维度 10 分，总分 30 分）
- 额外评估维度：首次生成可用率（代码不经修改即能通过编译的比例）

**环境统一**：Prompt 中给定相同的技术栈约束（Kotlin 2.x、Compose BOM 2025.x、Hilt、MockK、Turbine）

### 2.2.2 任务一：从零写一个 Compose 聊天列表

**需求描述**：

```
写一个聊天列表 Composable：
- 使用 LazyColumn 展示消息
- 每条消息含头像（Coil 加载圆形图片）、发送者名称、消息内容、时间戳（格式：今天 HH:mm，昨天 HH:mm，其他 MM/dd）
- 自己的消息靠右（蓝色气泡），别人的靠左（灰色气泡）
- 列表自动滚动到底部（最新消息）
- 消息数据从 ViewModel 的 StateFlow 获取
```

**评分结果**：

| 模型 | 编译通过 | 最佳实践 | 异常覆盖 | 总分 | 首次可用率 | 亮点/问题 |
|------|:---:|:---:|:---:|:---:|:---:|------|
| GPT-4o | 9 | 8 | 6 | 23 | 85% | 布局正确，但时间戳格式化逻辑有边界 bug（当年份跨年处理） |
| Claude Opus 4 | 10 | 10 | 8 | 28 | 95% | 唯一正确处理 `LazyColumn` 自动滚动的模型（`scrollToItem` + `LaunchedEffect`）；区分了不同类型的消息气泡 composable |
| Claude Sonnet 4 | 10 | 9 | 7 | 26 | 95% | 与 Opus 4 质量接近，消息对齐逻辑处理优秀 |
| Gemini 2.5 Pro | 9 | 8 | 7 | 24 | 80% | Compose 代码正确，但 `ContentScale` 和图片加载细节不够 |
| DeepSeek V3 | 8 | 8 | 5 | 21 | 70% | 基础布局正确，但 `key` 参数遗漏，重组性能差 |
| GLM-4 | 7 | 6 | 5 | 18 | 60% | `LazyColumn` 中未使用 `items` 中的 `key`；中文文案处理流畅 |
| Qwen3-235B | 8 | 7 | 5 | 20 | 70% | 中文注释生成完整，但时间戳处理有逻辑 bug |

**典型输出对比**（消息气泡代码）：

```kotlin
// Claude Opus 4 的输出 —— 正确区分左右、使用 key
@Composable
fun ChatMessageItem(
    message: ChatMessage,
    isMine: Boolean,
    modifier: Modifier = Modifier,
) {
    val arrangement = if (isMine) Arrangement.End else Arrangement.Start
    Row(
        modifier = modifier.fillMaxWidth(),

        verticalAlignment = Alignment.Bottom,

// DeepSeek V3 的输出 —— 未使用 key、头像位置未区分
@Composable
fun ChatMessageItem(message: ChatMessage) {
    Row { // 缺少 isMine 判断
        AsyncImage(model = message.avatar, contentDescription = null)
        Text(message.content)
    }
}
```

### 2.2.9 踩坑记录

| 问题 | 现象 | 解决方案 |

| 不同模型 API 返回格式不一致 | 同一段 Prompt，GPT-4o 只返回代码，DeepSeek 先解释再给代码 | 在 Prompt 中明确输出格式要求："只输出 Kotlin 代码，不要解释" 或 "先分析，再给代码" |

### 2.3.2 多模型组合策略

大多数实际场景中，你不需要只用一个模型。推荐以下**分层组合策略**：

```
┌─────────────────────────────────────┐
│  Layer 1: 日常开发（80% 工作量）      │

│  场景：写 ViewModel、Composable、     │
│         Repository 实现代码           │
│  成本：低-中                          │
├─────────────────────────────────────┤
│  Layer 2: 复杂任务（15% 工作量）      │
│  模型：Claude Opus 4                 │
│  场景：架构设计、复杂重构、          │
│         Gradle 多模块冲突             │
│  成本：高（按需使用）                │
├─────────────────────────────────────┤
│  Layer 3: 批量/廉价任务（5% 工作量）  │
│  模型：DeepSeek V3 / Qwen3           │
│  场景：自动生成测试、代码格式化、     │
│        文档生成、批量注释             │
│  成本：极低                          │
└─────────────────────────────────────┘
```

### 2.3.6 踩坑记录

| 问题 | 现象 | 解决方案 |
|------|------|----------|
| 只绑一个模型 | Git 提交代码风格三天一换，团队吐槽"AI 精神分裂" | 编码类任务统一入口模型，用 Rules 文件约束风格一致性 |
| 切模型不切 Prompt | 同一段 Prompt 在 GPT-4o 和 Claude 上的输出质量差异巨大 | 针对不同模型微调 Prompt（如 Claude 更擅长结构化回答，GPT-4o 需要更明确的步骤指令） |
| 追新不求稳 | 模型刚发布新版本就全面切换，结果遭遇稳定性问题 | 新模型至少有 1-2 周的社区反馈窗口，先在非关键任务上验证 |
| 忽视免费额度 | 每月付 API 账单，但没留意现有工具（GitHub Copilot、Cursor Pro）自带的模型额度 | 盘点已有工具订阅中包含的模型额度，避免重复付费 |
| 所有项目用同一配置 | A 项目（Kotlin + Compose）的选型方案直接套用到 B 项目（Java + XML），效果很差 | 不同技术栈用不同模型：Java 项目 GPT-4o 更好，Kotlin + Compose 项目 Claude 更好 |
| 只看排行榜不看实测 | 盲目选排行榜第一，结果和你的实际场景不匹配 | 拿 2.2.8 的动手实践任务在项目中跑一次，用数据决定 |

## 本章小结

1. **没有最好的模型，只有最适合的模型。** Claude Opus 4 综合最强但贵，DeepSeek V3 够用但细节不足。选型是取舍。

2. **Claude Sonnet 4 是 2026 年 Android 开发的"甜点模型"。** 在 Compose、Kotlin、测试三个核心场景上接近 Opus 4 的水平，但成本只有 1/5。

3. **Gemini 2.5 Pro 是 Gradle 和构建系统场景的王者。** 如果你需要频繁处理多模块构建、依赖冲突，Gemini 是必备工具。

4. **中文项目和成本敏感团队，DeepSeek V3 / Qwen3 是降本增效的最佳选择。** 100+ 倍的成本差距意味着你可以用它批量生成测试、文档、注释。

5. **多模型组合比分立好。** 把不同任务路由到不同模型，既能享受顶级质量，又能控制预算。

6. **模型正在快速迭代——保持关注，定期重新评估。** 你今天的选型方案，三个月后可能就不再是最优解。建议每季度跑一次 2.2 节的实测再看排名。
